# Sisal v0.5.0 roadmap ‚Äî driver parity & total test coverage

Where [v0.4.0](v0.4.0-roadmap.md) closed _feature_ gaps surfaced by the
`neon-hot-feed` example, v0.5.0 closes _parity_ gaps **between the adapters**.

**Mission.** Reach full feature parity across `@sisal/pg`, `@sisal/neon`,
`@sisal/sqlite`, and `@sisal/libsql` **wherever a dialect makes it possible**,
and cover **every** feature with an integration test on **every** adapter that
supports it. Where parity is _not_ possible (a feature is genuinely
Postgres-only), make the limitation explicit and graceful instead of silent.

**Outcome.** A single, generated **cross-driver feature matrix** in the docs
that replaces the four divergently-shaped per-engine matrices ‚Äî one row per
feature, one column per adapter, every ‚úÖ backed by a passing integration test.

## Where this came from

A parity audit (June 2026) compared the four compatibility matrices against the
_actual_ integration suites and the core render code, then ran real databases.
The asymmetry between offerings turned out to be **almost entirely missing test
coverage**, not missing capability:

- **SQLite** suite: 30/30 ‚úÖ &nbsp; **libSQL** suite: 27/27 ‚úÖ (both run
  first-hand).
- A throwaway probe ran the features the libSQL/Neon matrices _omit_ ‚Äî column
  naming, keyset pagination, prepared statements, and `db.batch` (atomic commit
  **and** rollback) ‚Äî against a real libSQL database: **5/5 passed**. These
  features share the exact render + driver paths that pg/sqlite already prove
  (`LIBSQL_DIALECT = "sqlite"`; Neon is `createPgOrmDriver` +
  `POSTGRES_DIALECT`).
- The same probe confirmed `.distinctOn()` is a **hard** limit on libSQL (the
  engine rejects the SQL), so the Postgres-only rows are correctly absent.

The three differences between offerings, only the first of which is a real
limit:

1. **Genuine dialect limits** ‚Äî `distinctOn`, row locking (`.for(...)`), array
   operators (`@>`/`<@`/`&&`), the typed function caller (`db.call`), and
   `ilike` (degrades to `LIKE`). Postgres-family only; SQLite-family engines
   have no equivalent.
2. **Round-trip differences** ‚Äî `json`/array as `TEXT`, `boolean` as `0/1`,
   `bytea` as `ArrayBuffer` on libSQL, `numeric`/`bigint` as numbers vs strings.
   The feature works; the value shape differs (marked ‚öÝÔ∏è today).
3. **Coverage gaps masquerading as imbalance** ‚Äî column naming, keyset,
   prepared, `sql`-in-`SET`/`VALUES`, and `db.batch` work on every driver but
   are not tested on every driver. **This is the bulk of v0.5.0.**

## 1. libSQL integration coverage parity ‚Äî P1

**Problem.** `integration/libsql_features_test.ts` tests 26 features; the SQLite
suite tests three more that libSQL also supports but never exercises: **column
naming** (snake_case default / `.named()` / `preserve`), **keyset pagination**
(both predicate forms), and **prepared statements**. The
[libsql-compatibility](libsql-compatibility.md) matrix therefore _looks_
narrower than SQLite for no real reason. libSQL renders identical SQL to SQLite
(`LIBSQL_DIALECT = "sqlite"`), so these pass unchanged ‚Äî verified directly by
the audit probe (5/5, including `db.batch` rollback).

**Proposed work.** Port the three SQLite tests into the libSQL suite (the
assertions are dialect-identical; only `connect`/teardown differ). They are a
near-mechanical copy of the existing `sqlite: column naming ‚Ä¶`,
`sqlite: keyset pagination ‚Ä¶`, and `sqlite: prepared statement ‚Ä¶` tests.

**Affected.** `integration/libsql_features_test.ts`,
`docs/libsql-compatibility.md` (matrix rows + count `27 ‚Üí 30`),
`docs/index.html` badge (`27/27 ‚Üí 30/30`).

**Acceptance criteria.**

- `SISAL_LIBSQL_IT=1 deno test integration/libsql_features_test.ts` covers
  column naming, keyset (expanded + row-value), and prepared statements, all
  green on a local file and against Turso.
- The libSQL matrix gains the three rows and reads `30 / 30`.
- No new ‚öÝÔ∏è behavior notes are needed (these features round-trip identically).

## 2. Neon integration coverage parity ‚Äî P1

**Problem.** `integration/neon_features_test.ts` covers 23 features; the pg
suite covers **six more** that Neon also supports ‚Äî `db.batch`, `sql` in
`SET`/`VALUES`/`onConflict`, keyset pagination, prepared statements, column
naming, and the typed function caller ‚Äî but the Neon suite tests none of them.
Neon is PostgreSQL reached through `createPgOrmDriver` + `POSTGRES_DIALECT`, so
the entire `@sisal/pg` surface is available; the gap is purely that the Neon
suite is slower to run (it needs a live endpoint or the Docker `neon-proxy`).

**Proposed work.** Add the six missing tests to the Neon suite (mirroring the pg
suite). Run them through the bundled `neon-proxy` so CI can exercise them
without a real Neon account, exactly as the existing Neon suite does.

**Affected.** `integration/neon_features_test.ts`, `docs/neon-compatibility.md`
(six matrix rows + run count), `docs/index.html` Neon badge.

**Acceptance criteria.**

- The Neon suite exercises batch, `sql`-in-`SET`/`VALUES`, keyset, prepared,
  column naming, and the typed function caller, green via `neon-proxy`.
- The Neon matrix reaches feature-row parity with the pg matrix (every
  non-Postgres-divergent row present).
- Alternatively, if a feature is deliberately left to inherit from `@sisal/pg`
  without re-verification, the matrix says so explicitly with a one-line reason
  (no silent omissions).

## 4. Typed dialect guards for Postgres-only builders ‚Äî P2

**Problem.** `.distinctOn(...)` (builders.ts) and `.for(...)` row locking render
**unconditionally** ‚Äî there is no dialect guard. Calling them on a SQLite-family
database produces SQL the engine rejects, so misuse surfaces as a raw database
**syntax error** rather than a typed `OrmError` that names the real cause. The
audit confirmed this against live libSQL.

**Proposed work.** When a Postgres-only construct (`distinctOn`, `.for(...)`
locking, the array operators) is rendered against a non-Postgres dialect, throw
a clear `OrmError` (e.g.
`"distinctOn is PostgreSQL-only; the sqlite dialect has no
equivalent"`) at
render time, before execution. This is the "parity where it is _not_ possible"
half of the mission: the limit stays, but it becomes graceful and
self-explaining.

**Affected.** `@sisal/orm` (`builders.ts` render path, `errors.ts`), parity
tests in `packages/{sqlite}/drizzle_parity_test.ts`.

**Acceptance criteria.**

- Rendering `distinctOn` / `.for(...)` / array operators on a SQLite-family
  dialect throws a typed `OrmError` naming the construct and the dialect.
- Postgres/Neon behavior is unchanged.
- A unit test pins the thrown error for each construct per dialect.

**Affected.** `docs/feature-matrix.md` (or a shared notes doc), the four

**Acceptance criteria.**

- Every `‚ùå` and `‚öÝÔ∏è` cell in the unified matrix links to a one-paragraph

## 7. Richer DDL: functions & triggers ‚Äî P2 (deferred from v0.4.0 item 5)

**Context.** [v0.4.0 item 5](v0.4.0-roadmap.md) shipped richer **index** DDL
(per-column `ASC`/`DESC`, partial `WHERE`, and `` sql`...` `` expression keys)
and already emitted `CHECK` constraints inline. The remaining sub-item ‚Äî 5.3,
explicitly the "stretch" ‚Äî is **functions and triggers**, which is a larger
design than an index tweak and so moves here.

**Problem.** The DDL generators emit only schema objects derived from
`defineTable` (tables, columns, constraints, indexes). They cannot express a
stored `CREATE FUNCTION` / `CREATE TRIGGER` body, so an app like
`examples/neon-hot-feed` keeps its `app.vote_post` function and hot-score
function as hand-written `.sql` migrations and treats `src/schema.ts` as a typed
mirror rather than the generator's source of truth.

**Proposed work.** A way to attach raw, dialect-specific DDL fragments
(functions, triggers, extensions) to a schema snapshot so generated migrations
include them verbatim, ordered **after** table/column/constraint creation. This
needs design for: ordering relative to tables, `down`/drop generation,
drift/checksum handling for opaque SQL bodies, and per-dialect gating (Postgres
functions vs. SQLite triggers differ).

**Affected.** `@sisal/orm` (`schema.ts` snapshot ‚Äî a new
`functions`/`schemaObjects` array; `table.ts` or a new `defineSchemaObject`
helper), each adapter DDL generator, and `@sisal/migrate` diff classification.

**Acceptance criteria.**

- A snapshot can carry ordered raw DDL fragments emitted after table creation,
  with matching drop statements for `down`.
- `examples/neon-hot-feed` can regenerate its full `0001_init.sql` (functions
  included) from `src/schema.ts`.
- Drift detection accounts for changed function/trigger bodies.
- Move the corresponding [drizzle-parity](drizzle-parity.md) rows.

**Status (üü° in progress).** The core landed: `createSchemaSnapshot` accepts a
`schemaObjects` array of raw, dialect-gated DDL fragments
(`SisalSchemaObjectSnapshot` ‚Äî function/trigger/extension/view/raw, with `up`,
optional `down`, optional `dialect`). The three additive DDL generators emit the
selected objects **after** their index loop, in declared order; an optional
`dialect` gates each object to one engine (a Postgres-only function is skipped
by the SQLite-family generator), and dialect-agnostic objects emit everywhere.
New `@sisal/orm` exports: `defineSchemaObject`, `selectSchemaObjects` (the
dialect+change filter the generators call), and `schemaObjectDropStatements`
(reverse-order `down`). Verified by `packages/orm/schema_test.ts`, pg/sqlite
render tests, and a `schema objects` integration test in all four suites (now 35
each) running a real trigger + view on PostgreSQL 18, Neon, SQLite, and libSQL.
Down emission and drift now land too:
`schemaObjectDropStatements(snapshot,
dialect)` gives the matching reverse-order
drops (a pg integration test applies the up then the down and asserts the
function/view are gone), and because `schemaObjects` are part of the normalized
snapshot, `equalSchemaSnapshots` ‚Äî and therefore `checkDrift` ‚Äî already flag a
changed function/trigger **body** (pinned by a `@sisal/migrate` test).
Regenerating `examples/neon-hot-feed` surfaced one more gap ‚Äî the builder's
`.default()` took only a literal or a JS client-side function, so a column
couldn't express a **SQL-expression server default** (`gen_random_uuid()`,
`now()`), even though the snapshot (`SisalColumnDefault.kind === "expression"`)
and the generators already emit one. That landed as a
`.default(sql\`‚Ä¶\`)`overload (stored as`ColumnDefinition.sqlDefault`; the column is omitted on
insert so the database fills it ‚Äî no insert-builder change needed). With it,`neon-hot-feed`now **generates** its full init (tables, DESC indexes, the
CHECK, and both functions moved into`schemaObjects`) from`src/schema.ts`‚Äî no
hand-written`.sql`files ‚Äî verified end-to-end against real Neon. The
[drizzle-parity](drizzle-parity.md) matrix had no schema-object row to move
(Drizzle has no structured function/trigger concept), so Sisal's`schemaObjects`
is ahead rather than at parity. **Item 7 is complete.**

## 8. Portable atomic "transaction script" ‚Äî P2 (procedure parity)

**Context.** The [`neon-rising-feed`](../examples/neon-rising-feed/README.md) ‚Üî
[`libsql-rising-feed`](../examples/libsql-rising-feed/README.md) pair builds the
identical `/rising` feed on both engines. The activity recorder ‚Äî dedupe the
actor, bump a counter, bump `unique_actors` only on the actor's first touch,
recompute the bucket score ‚Äî is the same logic, but it reads **completely
differently** per engine because SQLite has no stored procedures:

- **Neon:** one `app.record_post_activity(...)` function, called as a single
  atomic statement through the typed `db.call` surface ‚Äî Deno-Deploy friendly.
- **libSQL:** the same steps reimplemented in TypeScript inside an interactive
  `db.transaction(...)` (`insert().onConflictDoNothing().returning()` to detect
  a new actor, an `onConflictDoUpdate` upsert with raw-`sql` increments, a
  `select()` readback).

Both are clean, but the application code is duplicated and shaped by the engine,
not the domain.

**Problem.** There is no Sisal-level way to express "run these steps atomically"
that compiles to a **database function call where one exists** and an
**interactive transaction otherwise**. `db.batch` is close but forbids a
statement depending on a previous one's result, which this read-modify-write
needs.

**Proposed work (design).** A "transaction script" / atomic-operation helper:
author the steps once against the builder; on Postgres-family adapters it can be
backed by (or generated into) a `CREATE FUNCTION` and invoked as one statement,
while SQLite-family adapters run it as an interactive transaction. Needs design
for: how dependent results flow between steps, how it interacts with item 7
(function DDL generation), and the serverless single-round-trip guarantee.

**Affected.** `@sisal/orm` (new API + render/execute paths), every adapter, and
item 7's DDL generation. **Acceptance:** the rising-feed recorder reads
identically in both examples; it is one round trip on Neon HTTP and one
interactive transaction on libSQL; covered on every adapter.

**Status (üü° in progress).** The uniform path landed: `defineAtomicOperation`
(`@sisal/orm`) authors dependent read-modify-write steps once and runs them as
one **interactive transaction on every adapter** ‚Äî covered by
`packages/orm/atomic_test.ts` (network-free) + an `atomic operation` integration
test in all four suites and a unified-matrix row. **Deferred:** the
**single-round-trip Postgres path** (dispatch to a generated function or a
data-modifying CTE ‚Äî items 7/12) and the rising-feed example unification (the
examples still showcase their engine-optimal patterns until the function path
exists). The authored body is forward-compatible with that dispatch.

## 9. `FILTER` aggregates + interval/date math + `dateTrunc` ‚Äî P2

**Problem.** The moving-window score is a conditional aggregate over time
windows:

```sql
sum(activity_score) FILTER (WHERE bucket_start >= p_now - interval '15 minutes')
```

Sisal's builder has `sum`/`count`/`avg`/‚Ä¶ but **no `FILTER (WHERE ‚Ä¶)` clause**,
**no interval/date arithmetic**, and **no `dateTrunc`/time-bucket helper**. So
in `neon-rising-feed` the whole score lives in a raw SQL function, and in
`libsql-rising-feed` it is computed in TypeScript after reading the buckets ‚Äî
and the 5-minute bucket flooring is hand-written per dialect (`app.bucket_5m` in
SQL, `bucket5mIso` in TS).

**Proposed work.** (a) A `filter(condition)` modifier on aggregate expressions
that renders `agg(...) FILTER (WHERE ...)` (Postgres) and degrades to
`agg(CASE WHEN ... THEN ... END)` on SQLite. (b) An interval/`now`-relative date
helper for `where`/expressions. (c) A portable `dateTrunc('5 minutes', col)` /
bucket helper that renders per dialect (`date_trunc` /
`to_timestamp(floor(...))` on PG, `unixepoch`/`strftime` math on SQLite).

**Affected.** `@sisal/orm` (`operators.ts`, `sql.ts`, render paths), adapter
dialect rendering. **Acceptance:** the rising-score window query is
builder-native on both engines; render tests pin the per-dialect SQL; the
examples drop their raw-SQL/TS-only window math.

**Status (üü° in progress).** All three parts (a/b/c) landed as builder helpers.
`filter(agg, condition)` appends a `FILTER (WHERE ‚Ä¶)` clause ‚Äî and because
modern SQLite/libSQL support `FILTER` natively (the planned `CASE WHEN`
degradation proved unnecessary), it renders identically on every adapter. The
date math: `dateTrunc(field, source)` (calendar-field truncation), `now()`,
`dateAdd`/`dateSub(source, duration)` (interval arithmetic ‚Äî

(floor to an arbitrary-width bucket, e.g. 5 minutes). Each renders its own
per-dialect SQL
(`date_trunc`/`now()`/`¬± interval`/`to_timestamp(floor(epoch/N)*N)` on Postgres;
`strftime`/`datetime('now')`/chained `datetime` modifiers/

`dialectSql` primitive (throws `ORM_DIALECT_UNSUPPORTED` when no
variant/fallback matches). New `@sisal/orm` exports: `filter`, `dateTrunc`,
`DateTruncField`, `now`, `dateAdd`, `dateSub`, `dateBin`, `DateDuration`,
`dialectSql`. Verified by `packages/orm/aggregates_test.ts` and
`filter aggregate + dateTrunc` / `date math window` integration tests on all
four engines ‚Äî the latter runs the **actual rising-score moving window**
(`filter` + `dateSub` + `now`) and `dateBin` bucketing ‚Äî so "the window query is
builder-native on both engines" is met (three matrix rows added). The
`neon-rising-feed-ctes` example gained an **additive** builder-native
`selectRisingScore` (`src/queries.ts`) that computes its moving-window score
with `filter` + `dateSub`, verified against real Neon to equal the raw-SQL
recompute CTE and the TypeScript model ‚Äî resolving that example's item-9
pressure point. **Remaining (so item 9 stays in progress):** the full
rising-feed example _rewrites_ that would drop their raw-SQL / TS-only window
math are left as-is by design ‚Äî `libsql-rising-feed` deliberately teaches the
no-stored-proc TypeScript approach, `neon-rising-feed` the stored-function
approach, and the CTE example's atomic _write_ still needs a data-modifying CTE;
the new helpers make a builder-native aggregate possible without rewriting those
lessons.

## 10. `.optional()` should not widen the SELECT row type ‚Äî P3

**Problem.** `.optional()` is documented as an **insert-only** axis that "does
not change column nullability." But marking a nullable column `.optional()` adds
`undefined` to its inferred **SELECT** type (`Date | null | undefined`), so a
plain `InferSelect` row no longer matches a hand-written `T | null` interface.
Surfaced in `neon-rising-feed`: `rising_score_updated_at` had to stay
required-on-insert (passing an explicit `null`) to keep the feed row type clean.

**Proposed work.** Make `.optional()` affect only `InferInsert` (the insert
optionality), leaving `InferSelect` as `T | null`. **Affected.** `@sisal/orm`
column/type inference. **Acceptance:** an `.optional()` nullable column infers
`T | null` on select and `T | null | undefined`-omittable on insert; a type test
pins both; the example can use `.optional()` without widening the feed row.

## 11. Coerce numeric reads to `number` (float8 returns as string) ‚Äî P2

**Context.** Surfaced while validating the
[`neon-rising-feed`](../examples/neon-rising-feed/README.md) example against
real databases (June 2026). **`@sisal/pg` is the only adapter that returns
`double precision` as a string** ‚Äî a driver-parity gap, which is exactly this
roadmap's theme. Verified first-hand, all reading the same
`306.0::double
precision`:

| Adapter         | `double precision` read | Notes                                        |
| --------------- | ----------------------- | -------------------------------------------- |
| `@sisal/pg`     | `"306"` (**string**)    | `jsr:@db/postgres` decodes `float8` ‚Üí string |
| `@sisal/neon`   | `306` (number)          | `@neon/serverless` decodes `float8` ‚Üí number |
| `@sisal/sqlite` | `306` (number)          | SQLite `REAL` ‚Üí number                       |
| `@sisal/libsql` | `306` (number)          | SQLite `REAL` ‚Üí number                       |

Integers (`int2`/`int4`) come back as numbers on `@sisal/pg`, and `timestamptz`
maps to `Temporal`/`Date`, so only the float types are inconsistent.

**Problem.** A column declared `columns.doublePrecision()` infers `number`, but
on `@sisal/pg` the runtime value is a `string`. This is a silent type lie with
real consequences:

- client-side numeric work breaks ‚Äî `score.toFixed(2)` throws (`"306".toFixed`
  is not a function), and `Array.sort`/comparisons run lexicographically
  (`"180" < "72"` is `true`);
- a keyset/ORDER BY whose tiebreaker is a float column still _orders correctly
  in the database_ (the DB casts the bound string param), so the bug hides until
  a consumer touches the value in JS.

**Where it surfaced.** Running the `neon-rising-feed` TypeScript path against a
local PostgreSQL 18 via `@sisal/pg`: `rising_score` came back as a string, so
the feed's `non-increasing` ordering check failed lexicographically and
`rising_score.toFixed(...)` would throw. The example itself uses
**`@sisal/neon`** (which returns a number), so it is _not_ affected and was left
uncoerced; but `@sisal/pg` (and `neon-hot-feed`'s `hot_score`, if run through
`@sisal/pg`) is.

**Proposed work.** Decode `float4`/`float8` to `number` in the `@sisal/pg`
result path so a `columns.doublePrecision` column reads back as the `number` it
infers, matching the other three adapters. `numeric`/`bigint` stay strings by
necessity (precision) and belong in the round-trip reference (item 5); this item
is specifically about the float types, where `number` is the correct, lossless
mapping.

**Affected.** `@sisal/pg` result decoding (`jsr:@db/postgres` type parsers) only
‚Äî `@sisal/neon` already returns a number;
[round-trip reference](#5-document-principled-divergences--round-trip-behavior-p2)
(item 5); the unified matrix (item 3) gains a tested "float round-trip" row.

**Acceptance criteria.**

- A `double precision` column read through `@sisal/pg` yields a JS `number`,
  matching `@sisal/neon` / `@sisal/sqlite` / `@sisal/libsql`.
- Integration coverage on pg asserts the runtime `typeof` is `number`.
- The float round-trip is added to the round-trip reference (item 5) and the
  unified matrix (item 3).

## 12. Data-modifying CTE builder (`WITH ‚Ä¶ INSERT/UPDATE/DELETE ‚Ä¶ RETURNING`) ‚Äî P2

**Context.** Surfaced by
[`neon-rising-feed-ctes`](../examples/neon-rising-feed-ctes/README.md) (June
2026) ‚Äî a deliberately function-free variant of the rising feed that models
every multi-step mutation as **one data-modifying CTE statement** (Neon
HTTP-friendly: one parameterized statement, no interactive transaction
callback). The activity recorder is a
`WITH input_data ‚Üí validated_input ‚Üí
bucket_data ‚Üí actor_insert(INSERT‚Ä¶ON CONFLICT‚Ä¶RETURNING) ‚Üí actor_flag ‚Üí
bucket_upsert(INSERT‚Ä¶ON CONFLICT DO UPDATE‚Ä¶RETURNING)`
statement; the recompute paths are
`WITH score_windows ‚Üí computed_score ‚Üí updated_post(UPDATE‚Ä¶FROM‚Ä¶
RETURNING)`.

**Problem.** Sisal's CTE builder is **SELECT-only**: `db.$with(name).as(query)`
takes a `SetOperand` (a `SELECT`/compound) and `db.with(...ctes)` terminates in
`.select(...)`. There is no way to:

- use an `INSERT`/`UPDATE`/`DELETE ‚Ä¶ RETURNING` as a CTE body,
- chain such stages (one CTE consuming another's `RETURNING`),
- or return rows from a `WITH`-wrapped data-modifying statement.

So the entire recorder and both recompute statements are hand-written raw `sql`
template strings ‚Äî by far the largest raw-SQL surface in any example. The
builder's `db.batch` (independent statements) and interactive `transaction`
(multiple round trips / open session) do **not** cover the "several dependent
steps as one atomic statement" shape, which is exactly what data-modifying CTEs
give and what the serverless story wants.

**Where it surfaced.** `examples/neon-rising-feed-ctes/src/activity.ts`
(`recordPostActivity`) and `src/rising.ts` (`recomputePostRisingScore`,
`recomputeAllRisingScores`) ‚Äî all raw `sql`.

**Proposed work (design).** A composable data-modifying CTE builder, e.g.
`db.$with("actor_insert").as(db.insert(actors).values(...).onConflictDoNothing().returning())`,
allowing CTE bodies to be insert/update/delete builders, downstream CTEs to
reference upstream `RETURNING` columns, and the chain to terminate in a final
`SELECT`/`RETURNING` whose row type is inferred. This is a large design (CTE
dependency typing, `EXCLUDED`/`ON CONFLICT` interplay, dialect support ‚Äî
Postgres yes, SQLite supports data-modifying-CTE less fully) and overlaps with
item 8 (portable atomic operations).

**Affected.** `@sisal/orm` (`builders.ts` CTE types + `database.ts`
`$with`/`with` to accept mutation operands and a mutating terminal), each
adapter's render path.

**Acceptance criteria.**

- A `WITH` chain whose stages are `INSERT/UPDATE/DELETE ‚Ä¶ RETURNING` builders
  can be assembled and executed, returning a typed row.
- The `neon-rising-feed-ctes` recorder + recompute can drop to the builder (or
  the example documents precisely which sub-feature still needs raw SQL).
- Render tests pin the generated `WITH` SQL and parameter order.

**Status (üü° in progress).** The core landed: a CTE body can be an
`INSERT`/`UPDATE`/`DELETE ‚Ä¶ RETURNING` builder
(`db.$with("x").as(db.insert(t).values(...).returning())`, new `CteOperand`
type) and the `with(...)` chain terminates in a `SELECT` that reads its
`RETURNING`. **PostgreSQL-only** ‚Äî the SQLite family rejects data-modifying
CTEs, so rendering one for a SQLite-family dialect throws a typed `OrmError`
(item-4 guard). Render tests pin the SQL + the guards; executed on PostgreSQL 18
and Neon, with a unified-matrix row (pg/neon ‚úÖ, sqlite/libsql ‚ùå). The core
**builder gaps are now closed** (the three mutation builders were first
refactored to the `#state` + `#with(patch)` shape to make this safe):

1. **`UPDATE ‚Ä¶ FROM`** ‚Äî `update(t).from(source)` (the rising-feed recompute's
   keystone). CTE columns referenced in `SET`/`WHERE` render as `"cte"."col"`.
2. **`DELETE ‚Ä¶ USING`** ‚Äî `delete(t).using(source)`, **PostgreSQL-only** (a
   typed guard throws on the SQLite family).
3. **`INSERT ‚Ä¶ SELECT`** ‚Äî `insert(t).select(query)`, so an insert reads another
   CTE's `RETURNING`
   (`with moved as (delete ‚Ä¶ returning) insert ‚Ä¶ select ‚Ä¶ from
   moved`).
4. **Mutating terminal** ‚Äî `with(...).update/insert/delete(t)`.

Chained `SELECT` CTEs already worked. `UPDATE ‚Ä¶ FROM` / `INSERT ‚Ä¶ SELECT` run on
**every adapter** (modern SQLite/libSQL included); covered by
`packages/orm/mutation_cte_test.ts` and a `mutation joins` integration test on
all four engines. **Remaining:** only **item 12c** ‚Äî dropping the
`neon-rising-feed-ctes` recorder/recompute to the builder (Phase 5) ‚Äî so item 12
stays in progress.

## 13. Typed raw-query result mapping ‚Äî P3

**Context.** Also from
[`neon-rising-feed-ctes`](../examples/neon-rising-feed-ctes/README.md): because
the mutations are raw `sql` (item 12), their result rows come back through
`db.query<T>(sql\`‚Ä¶\`)`, which **trusts the caller's`<T>`generic** and does no
column-metadata-driven decoding. Each CTE's shape is hand-written
(`RecordedBucket`,`RecomputedPost`)
and trusted to match the SQL.

**Problem.** Raw-query rows are driver-shaped with no `defineTable`-driven
mapping: there is no way to say "decode this raw result as
`post_activity_buckets` rows" and get column naming, date/number coercion
(relates to item 11), and a checked row type. The hand-typed generic can
silently drift from the SQL, and on `@sisal/pg` the `double precision` columns
even come back as strings (item 11) with no typed layer to fix it.

**Proposed work.** A typed raw-result mapper ‚Äî e.g.
`db.query(sql\`‚Ä¶\`).as(post_activity_buckets)`or`sql<typeof bucketRowShape>\`‚Ä¶\``‚Äî that applies a column descriptor map to a raw
result: physical‚ÜíJS key naming, per-column decoding (float8‚Üínumber, timestamptz‚Üí
Date/Temporal), and a derived row type. Pairs naturally with item 11 (float
decoding) and item 12 (typing a data-modifying CTE's`RETURNING`).

**Affected.** `@sisal/orm` (`database.ts` query surface + a result-metadata
mapper reusing the column descriptors `defineFunction`/select already carry).

**Acceptance criteria.**

- A raw query result can be decoded against a column-descriptor map, yielding a
  typed row with consistent naming and value coercion across adapters.
- The CTE example's hand-written `RecordedBucket` / `RecomputedPost` shapes can
  be derived from the table models instead of restated.

**Status (üü° in progress).** Both input forms landed. A raw `db.query(...)`
returns a `MappableQueryResult` ‚Äî still an awaitable `OrmQueryResult`, but with
an `.as(...)` method that decodes the raw driver rows (physical‚ÜíJS column naming

- the same opt-in Temporal decoding the builder applies). Pass a `defineTable`
  model for typed `InferSelect<table>` rows, or a free-form `ColumnMap` for a
  result that doesn't match one table (a join/aggregate/CTE projection). Float
  coercion is already handled upstream (item 11, in the `@sisal/pg` executor).
  Unknown columns pass through; a plain `await db.query(...)` is unchanged. New
  `@sisal/orm` exports: `MappableQueryResult`, `ColumnMap`, `ColumnMapping` (the
  four adapter facades widen their `query` return to match). Verified by
  `packages/orm/raw_mapping_test.ts` and a `typed raw-query mapping` integration
  test on all four engines exercising both `.as(table)` and `.as(map)`; a
  unified-matrix row added. **Remaining (so item 13 stays in progress):** only
  the `neon-rising-feed-ctes` example refactor that would derive its
  `RecordedBucket` / `RecomputedPost` shapes from the table models.
