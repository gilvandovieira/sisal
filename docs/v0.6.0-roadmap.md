# Sisal v0.6.0 roadmap — Foundations & Readiness

Where [v0.5.0](v0.5.0-roadmap.md) makes the OLTP builder boringly correct across
the four dialects, v0.6.0 is the **readiness** milestone: it earns the right to
build later packages (and reach a fifth dialect) by proving the foundation can
carry them. It is deliberately **two parallel investigation workstreams plus one
readiness report**, all of which _prepare the ground and ship no new public
feature package_:

- **Workstream A — ETL Readiness Investigation.** The minimum ORM/core changes
  so a future `@sisal/etl` (v0.10) can be built without hacks.
- **Workstream C — MySQL Support Investigation.** Map what a `@sisal/mysql`
  adapter ([v0.7](v0.7.0-roadmap.md) builds it) needs, given that the renderer
  already carries a latent `"mysql"` dialect. Investigation only — no adapter.
- **npm-release readiness report (was "Workstream B").** Node.js + npm is **no
  longer a build workstream**. It sits on **no roadmap gate**, so it is
  investigated — not built — here: v0.6 produces the
  [npm-release readiness report](npm-release-readiness.md) (what must change to
  publish on npm) and ships **no npm package**. The build + dual publish is
  **deferred to [v0.13+](v0.13.0-roadmap.md), on demand**, enabled by the v0.8
  `@sisal/core` extraction and **blocked on a chosen npm name** (the `@sisal`
  scope is taken on npm). Publishing under an undecided name is the foot-gun the
  [sequencing audit](roadmap-sequencing-audit.md) warns about.

**This release builds nothing it investigates.** v0.6 answers "can Sisal express
ETL-shaped SQL cleanly?" and "what does MySQL support require?" — it ships
neither the ETL package nor the MySQL adapter, and no npm package. See
[release discipline](roadmap.md#release-discipline-non-negotiable).

## Finding (shipped in v0.5.1) — the `@sisal/pg` driver latency stall + postgres.js fix

Out-of-band during v0.6, benchmarking a real app on `@sisal/pg` surfaced a
severe per-query latency stall that is **entirely in the driver, not Sisal** —
and a validated fix. It shipped as [v0.5.1](../CHANGELOG.md#051---2026-06-30);
it is recorded here because it establishes the driver/adapter axis this
milestone builds on.

- **Symptom.** ~40 ms per _parameterized_ query on `@sisal/pg` — dead last in a
  feed-server benchmark (120 rps @ 90 ms p50 vs Drizzle's 5,550 @ 2 ms).
- **Root cause (source-verified).** `jsr:@db/postgres` never sets `TCP_NODELAY`
  and writes the extended-protocol Parse/Bind/Describe/Execute/Sync as five
  un-coalesced TCP segments → the classic Nagle × delayed-ACK deadlock (~40 ms
  timer). Sisal's builder (~0.02 ms) and executor (~0.01 ms) add nothing —
  isolated across five paths in [`perf/`](../perf/README.md).
- **Fix (validated, 40/40 × PostgreSQL 16/17/18).** A postgres.js-backed
  `PgPool` (`createPostgresJsPool`, or `connect({ driver: "postgres-js" })`);
  postgres.js sets `TCP_NODELAY` and pipelines. **No `@sisal/orm`/executor
  change.** The feed server went **120 → 6,774 rps (~56×), 90 → 1.6 ms**, moving
  Sisal from last to first (level with Drizzle, as both then ride postgres.js).
  Type parity with `@db/postgres` preserved (int8 → `BigInt`; naive
  `date`/`timestamp` via `new Date`). The default driver stays pure-JSR
  `@db/postgres`; `npm:postgres` is a lazily-imported opt-in, consistent with
  `@sisal/libsql`.
- **Implications carried forward.**
  - Driver identity now has a real axis (`db-postgres` | `postgres-js`), feeding
    the `(engine, version)` dialect-key / capability-descriptor thread the
    [sequencing audit](roadmap-sequencing-audit.md) tracks.
  - Open follow-ups: decide whether postgres.js becomes the **default** (a
    semver/deprecation call), and/or file the one-line `setNoDelay(true)` fix
    upstream in deno-postgres to rescue the pure-JSR path.
  - Guarded going forward by `deno task perf:pg` / `perf:pg:guard`; full report:
    [`perf/PG_ADAPTER_PERF_REPORT.md`](../perf/PG_ADAPTER_PERF_REPORT.md).

## Cross-adapter parity (v0.6.0)

The v0.5.1 driver raised a portability question: do the PostgreSQL-family
adapters return identical row shapes for the same query? Measured and pinned as
a gated test —
[`integration/cross_adapter_parity_test.ts`](../integration/cross_adapter_parity_test.ts)
(runs the same schema-free rich row through each adapter and diffs the decoded
values):

- **The two `@sisal/pg` drivers (`@db/postgres` ↔ postgres.js) decode
  byte-identically** across every scalar/array/temporal type — drop-in
  interchangeable, which is the correctness half of the 0.5.1 driver work.
- **`@sisal/neon` aligns with `@sisal/pg` on every type except `bigint`.** Neon
  (via `@neon/serverless`'s node-postgres type parsers) returns `bigint` as a
  **string**; both `@sisal/pg` drivers return **`BigInt`** (values are equal,
  only the JS type differs). `int`, `text`, `bool`, `numeric`, `float8`,
  `text[]`, `jsonb`, `time`, `timestamp`, `timestamptz`, and `null` all match. A
  raw `date` column is a `Date` at neon's _local_ vs pg's _UTC_ midnight — a
  process-TZ-dependent representation difference that resolves to the same
  `Temporal.PlainDate` under `temporal: { parse: true }`, so it is documented,
  not failed.

The test pins these facts the way [`docs/drizzle-parity.md`](drizzle-parity.md)
does: a driver behavior change (e.g. neon aligning `bigint`) fails it and forces
this doc to move in step.

**Open decision — align `bigint` across the pg family + neon.** Either neon →
`BigInt` (align to the pg drivers' current behavior), or all → `string` (matches
the "preserve precision" intent noted in `@sisal/pg`'s executor, and what neon
already does). Until decided, `bigint` is the one documented cross-adapter
portability caveat.

**Pain point (hit first-hand, 2026-07-01).** This divergence is not theoretical.
Making `postgres-family-activity-vectors` run over the whole family surfaced a
silent portability bug from exactly it: `getSimilarPosts` matched a source post
by its `bigint` `post_id` with a strict `s.post_id === postId` against a string
id. That quietly works on `@sisal/neon` (bigint → string) and throws
`no stats for post 2` on `@sisal/pg` (bigint → `BigInt`, so `2n === "2"` is
`false`). It type-checks, passes on the adapter it was written against, and
breaks on the sibling **with no warning** — the worst failure mode. Worked
around in the example by comparing `String(post_id)`, but that is a footgun
every consumer must remember, and concrete motivation to pick one representation
and align it. The
[cross-adapter parity test](../integration/cross_adapter_parity_test.ts) pins
the divergence so it can't regress silently once decided.

# Workstream A — ETL Readiness Investigation

## Goal

Investigate and implement **only** the minimum ORM/core changes needed so a
future `@sisal/etl` package can express extract→transform→load as pushed-down
SQL without dropping to raw strings for the core shapes. Produce a **dialect
capability matrix** and a **proof-of-concept rollup** in docs/pseudo-code.

## Questions this release must answer

- Can Sisal express ETL-shaped SQL (grouped rollup → idempotent upsert) cleanly
  through the typed builder?
- Does the current internal query representation support this, or does it need a
  stronger SQL IR/AST? (See the [v0.8](v0.8.0-roadmap.md) IR work it feeds.)
- Can each dialect (pg/neon/sqlite/libsql) support the required SQL patterns?
- Which features belong in **core** and which belong in **dialect adapters**?
- What must be postponed to v0.8/v0.10?

## Scope — investigate & document support for

Set-based movement and aggregation (the rollup spine):

- `INSERT INTO … SELECT …` (insert-from-select) — **shipped in
  [v0.5 item 12](v0.5.0-roadmap.md)** (`insert(t).select(query)`, tested on all
  four engines). The original "single biggest ETL gap" is now a **verify + pin**
  task, not new work (see the PoC findings below).
- CTEs (`WITH`) — SELECT **and** data-modifying (Postgres-family) shipped in
  [v0.5 item 12](v0.5.0-roadmap.md); SQLite-family CTEs stay SELECT-only (typed
  guard).
- Aggregate expressions; `GROUP BY`; `HAVING` — **supported**.
- Conditional aggregation: `agg(…) FILTER (WHERE …)` — **shipped as `filter()`
  in [v0.5 item 9](v0.5.0-roadmap.md)**, and **native on the SQLite family**
  (the planned `CASE WHEN` fallback proved unnecessary).
- Upsert from select (`INSERT … SELECT … ON CONFLICT DO UPDATE`) — the pieces
  shipped (`insert().select()` + `onConflictDoUpdate`, v0.5); **verify they
  compose** into one statement per dialect.
- Typed raw-SQL escape hatches — **present** (`sql` tag binds params safely);
  the fallback for anything unbuilt.
- Computed expressions and expression **aliases** in projections — partial;
  needs a first-class typed alias surface for rollup metric columns.
- Date/time **bucket** abstraction — **shipped as `dateTrunc`/`dateBin` in
  [v0.5 item 9](v0.5.0-roadmap.md)** (per-dialect: `date_trunc`/`to_timestamp`
  on Postgres, `strftime`/`unixepoch` on the SQLite family).
- **Window functions** (`OVER`, moving averages, `lag`/`lead`, `rank`) —
  **absent, no builder at all**; the PoC's hard wall, owned by
  [v0.7](v0.7.0-roadmap.md).
- **Array projection** (`ARRAY[col, …]`) and **set-returning**
  (`unnest …
  WITH ORDINALITY`) — **absent**; raw SQL today (no array type on
  the SQLite family).

Execution & correctness primitives the runner will need:

- Batch execution; transaction boundaries — **today: supported** (`db.batch`,
  `db.transaction`). Note `db.batch` forbids cross-statement data dependencies.
- **Advisory locks or an equivalent locking strategy** — Postgres has
  `pg_advisory_lock` (already used internally by the migration history store — a
  reusable pattern); SQLite/libSQL need an equivalent (a lock row /
  `BEGIN
  IMMEDIATE`). **A2 documents the per-dialect strategy** in
  [08-job-queue-locking](../examples/advanced-sql-contracts/08-job-queue-locking.md);
  v0.9 must make it real and test-backed before v0.10 consumes it.
- **Checkpoint table design** — a watermark table (`job`, `window_end`,
  `updated_at`) so a run resumes from the last committed window. **A3 chooses an
  `@sisal/etl`-managed table** named `sisal_etl_checkpoints`; v0.9 may overturn
  that only with a recorded package-boundary reason.
- **Idempotent load strategy** — re-running a window must not double-count
  (upsert/replace semantics keyed by the rollup grain). **A4 fixes the
  invariant:** replace/upsert metrics, never additive increments.
- **Backfill/replay requirements** — re-run an arbitrary historical window range
  deterministically. **A4 defines `run`, `replay`, and `backfill` semantics** in
  [09-idempotent-backfill](../examples/advanced-sql-contracts/09-idempotent-backfill.md).

## Deliverable — proof-of-concept rollup (now a runnable example)

A `post_events → post_hourly_stats` rollup demonstrating the desired shape (the
repo's existing `*-rising-feed` examples already bucket activity — this
generalizes that into an ETL rollup):

- **Extract** from a raw `post_events` table over a half-open time window
  `[from, until)`.
- **Group** by `post_id` and an hourly bucket.
- **Compute** `views`, `votes`, `comments`, `engagement_score`.
- **Load** into `post_hourly_stats` with idempotent upsert/replace.
- **Checkpoint** the window end (kept as a _future concern_ — design, don't
  ship).

The key SQL shape v0.6 must establish a path to express:

```sql
INSERT INTO post_hourly_stats (post_id, bucket, views, votes, comments, engagement_score)
SELECT post_id,
       date_trunc('hour', occurred_at) AS bucket,
       count(*) FILTER (WHERE kind = 'view')    AS views,
       count(*) FILTER (WHERE kind = 'vote')    AS votes,
       count(*) FILTER (WHERE kind = 'comment') AS comments,
       /* weighted */                            AS engagement_score
FROM post_events
WHERE occurred_at >= $from AND occurred_at < $until
GROUP BY post_id, date_trunc('hour', occurred_at)
ON CONFLICT (post_id, bucket)
DO UPDATE SET views = excluded.views, votes = excluded.votes,
              comments = excluded.comments, engagement_score = excluded.engagement_score;
```

This single statement exercises every audited gap: insert-from-select, a date
bucket, filtered aggregates (FILTER is native on the SQLite family since v0.5 —
no `CASE WHEN` fallback needed), and upsert-from-select. v0.6 decides which of
these become typed builders in v0.8.

## Findings — the `postgres-family-activity-vectors` PoC (built first-hand)

[`examples/postgres-family-activity-vectors`](../examples/postgres-family-activity-vectors/README.md)
is a **runnable** realization of this PoC on the PostgreSQL family (`@sisal/pg`
on `@db/postgres`, `@sisal/pg` on postgres.js, or `@sisal/neon`), verified
against real Postgres. It runs the full chain — raw `post_events` → hourly
buckets (a `fold` insert-from-select) → consolidated stats with **moving
averages** → an `ARRAY[…]` activity-vector projection → cosine similarity →
daily/monthly **rollups** + event **pruning**. What it found about how far the
v0.5 builder actually reaches (this is the substance of the readiness
investigation):

- **The rollup spine is largely builder-native already (post-v0.5).**
  `insert(t).select(query)` and `onConflictDoUpdate(...)` both ship (v0.5 item
  12, tested on all four engines), as do `filter(agg, cond)` and
  `dateTrunc`/`dateBin` (v0.5 item 9, native FILTER on the SQLite family). So
  `fold_events_to_buckets` / `rollup_daily` / `rollup_monthly`
  (`INSERT … SELECT … GROUP BY … FILTER … ON CONFLICT DO UPDATE`) are **probably
  expressible through the builder today**. The v0.6 question narrows from "can
  we express the rollup?" to **"verify the full statement composes + renders per
  dialect, and pin it"** — the example currently uses raw SQL functions for
  framing, so converting them is the concrete verification task (and the honest
  way to find the limit instead of hiding it).
- **Window functions are the one hard wall — no builder of any kind.** The
  moving averages
  (`avg(...) OVER (PARTITION BY … ORDER BY … ROWS BETWEEN 5
  PRECEDING AND CURRENT ROW)`)
  cannot be expressed at all: no `OVER`, `PARTITION BY`, `ROWS`/`RANGE`,
  `lag`/`lead`, `rank`, `row_number`. This is the single biggest analytics gap
  and is **owned by [v0.7](v0.7.0-roadmap.md)**; v0.6 records it as the blocking
  discovery.
- **Array construction and set-returning have no builder.** Storing a JS array
  into an array column is builder-native, but **projecting named columns into a
  SQL `ARRAY[…]`** (the feature-vector projection) and consuming one with
  `unnest … WITH ORDINALITY` are raw. On the SQLite family there is **no array
  type at all** (`json_array`/`json_each` instead). That sibling remains a
  future contract, not a v0.6 runnable example, because the shape is genuinely
  different rather than a small adapter switch.
- **`CREATE FUNCTION` has no builder.** The whole `app.*` computation engine is
  raw `.sql`. (v0.5 B7 can _carry_ such DDL in a snapshot; it does not _emit_ a
  typed function.)
- **Clean / no gap:** the batch event insert, the stats/posts reads, and
  similarity candidate loading are all clean builder `select`; bigint ids
  round-trip as strings (documented); and the typed function caller
  (`defineFunction`/`db.call`) exists and could replace the raw `sql` calls.

**Net for v0.6:** the ETL rollup spine is closer to done than the original audit
assumed (v0.5 delivered the pieces) — so a primary v0.6 deliverable is to
**prove the builder expresses the rollup end-to-end and pin per-dialect render
tests**, rather than design new primitives. The genuinely-unbuilt analytics
shapes the PoC surfaced (window functions, array/`unnest` projection) are
recorded here and handed to [v0.7](v0.7.0-roadmap.md)/[v0.8](v0.8.0-roadmap.md).

  `json_each` alternative — are written up as future contracts in
  [`examples/advanced-sql-contracts/`](../examples/advanced-sql-contracts/README.md)
  and routed to [v0.7](v0.7.0-roadmap.md)/[v0.8](v0.8.0-roadmap.md). The direct
  handoff docs are
  [02-window-analytics](../examples/advanced-sql-contracts/02-window-analytics.md),
  [03-sessionization](../examples/advanced-sql-contracts/03-sessionization.md),
  [04-top-n-per-group](../examples/advanced-sql-contracts/04-top-n-per-group.md),
  and
  [10-json-table-extraction](../examples/advanced-sql-contracts/10-json-table-extraction.md).
- **A2 — Locking strategy. ✅ Done (2026-07-01).** The v0.6 decision is a coarse
  ETL run lock keyed by job, documented in
  [08-job-queue-locking](../examples/advanced-sql-contracts/08-job-queue-locking.md).
  PostgreSQL/Neon use a session-scoped `pg_try_advisory_lock` /
  `pg_advisory_unlock` on a stable `sisal:etl:<job>` key hashed to 64-bit (the
  same class of pattern already used by the migration history store).
  SQLite/libSQL use `BEGIN IMMEDIATE` around lock/load/checkpoint for supported
  local or interactive runs, and otherwise capability-gate rather than silently
  degrade. Future MySQL/MariaDB use `GET_LOCK(name, 0)` / `RELEASE_LOCK(name)`.
  v0.9 owns the implementation and per-engine tests before v0.10 consumes it.
- **A3 — Checkpoint design. ✅ Done (2026-07-01).** The v0.6 default is an
  `@sisal/etl`-managed checkpoint table named `sisal_etl_checkpoints` with
  logical columns `job` (primary key), `window_end`, and `updated_at`;
  adapter-specific physical timestamp types are allowed. This keeps the future
  `etl` package off `@sisal/migrate` unless v0.9 reverses the decision with a
  recorded package-boundary reason.
- **A4 — Idempotent load + backfill/replay. ✅ Done (2026-07-01).** A run
  processes half-open windows `[from, until)`, writes through upsert/replace
  semantics keyed by the rollup grain, and advances the checkpoint atomically
  with the load. `run()` acquires the job lock, reads the checkpoint, loads one
  next window, and advances `window_end`; `replay(window)` re-runs one explicit
  window without advancing the checkpoint by default; `backfill(range)` iterates
  explicit windows and does not disturb the live checkpoint by default.
  Idempotency forbids additive metric increments in the load path.
  **Replay-vs-retention invariant (added 2026-07-01):** replace-semantics replay
  over a window whose raw source rows were already pruned would silently
  overwrite good rollups with zeros — so the checkpoint contract carries a
  per-job `pruned_before` replay horizon, advanced atomically with the prune
  (never lagging the delete), and `replay`/`backfill` refuse windows behind it
  with a typed error (explicit unsafe override, mirroring
  `.unsafeAllowAllRows()`). Recorded in
  [09-idempotent-backfill](../examples/advanced-sql-contracts/09-idempotent-backfill.md);
  v0.9 implements and tests it with the rest of the substrate.

# Workstream C — MySQL Support Investigation

## Goal

Determine exactly what a future `@sisal/mysql` adapter requires, so
[v0.7](v0.7.0-roadmap.md) can implement it without surprises — and decide which
work is **core** (shared dialect plumbing) vs **adapter** (driver, executor,
DDL, type mapping). Investigation and a capability report only; **no adapter
ships in v0.6.**

## What grounds this — MySQL is latent, not absent

A June 2026 audit found the renderer already half-knows MySQL, but no adapter
exists:

- **Already present (core render layer):** `SqlDialect` includes `"mysql"`
  (`core/sql.ts`); the serializable snapshot dialect type includes `"mysql"`
  (`schema.ts`); `quoteIdentifier` emits **backtick** quoting for `mysql`
  (`core/sql.ts`); placeholders fall through to `?` (only `postgres` uses `$N`);
  `ilike` degrades to case-insensitive `LIKE` (MySQL has no `ILIKE`); and the
  builder comments already treat `FOR UPDATE`/`FOR SHARE` and row-value tuples
  as Postgres/**MySQL**-supported.
- **Absent (everything an adapter needs):** there is **no `@sisal/mysql`
  package** — no `MYSQL_DIALECT` constant, no executor, no driver/pool, no
  migrate wiring, **no MySQL DDL generator** (only
  `generate{Postgres,Sqlite,Libsql}UpStatements` exist), no column-type → MySQL
  type mapping, no integration suite, no `docs/mysql-compatibility.md`, no
  feature-matrix column, and no chosen MySQL driver.

## Scope — investigate & document

- **Driver choice & runtime** — survey MySQL drivers usable from Deno (and Node,
  per the [npm-release readiness report](npm-release-readiness.md) — pick one
  that works on both so npm stays cheap later): `jsr`/`npm` options,
  connection/pool API, prepared-statement and parameter-binding model, TLS, and
  any serverless variants (e.g. PlanetScale-style HTTP). Note the
  injectable-executor seam every adapter uses.
- **Dialect completeness** — verify/extend the latent `"mysql"` render path:
  identifier quoting (backtick + escaping), `?` placeholders, `LIMIT/OFFSET`,
  `ON DUPLICATE KEY UPDATE` vs `ON CONFLICT` (MySQL upsert differs
  **materially** from Postgres/SQLite), `RETURNING` (MySQL lacks it pre-8.0.21 /
  MariaDB differences), `ILIKE`→`LIKE`, `FOR UPDATE`/`FOR SHARE`, and whether
  the `dialectGuard` set is correct for MySQL (row locking is supported;
  `distinctOn` and array operators are not — **corrected by C3's guard sweep**).
- **Type system & round-trips** — MySQL type mapping (`BOOLEAN`=`TINYINT(1)`,
  `JSON` type, `DATETIME`/`TIMESTAMP`/timezone handling, `DECIMAL`/`BIGINT`
  string-vs-number like the pg `numeric` case), and `bytea`/`BLOB` shapes.
  **Answered by C4** — probe-verified table in
  [`mysql-ddl-mapping.md`](mysql-ddl-mapping.md).
- **DDL generation** — what a `generateMysqlUpStatements` must emit (engine,
  charset/collation, `AUTO_INCREMENT`, index/`CHECK` support quirks). **Answered
  by C4** — generation rules + version floor in
  [`mysql-ddl-mapping.md`](mysql-ddl-mapping.md).
- **MySQL vs MariaDB** — decide whether "MySQL" means MySQL 8+, MariaDB, or
  both, and where they diverge (`RETURNING`, sequences, JSON functions).
  **Answered by C5** — both, one adapter, MySQL-8-first; probe-verified
  divergence matrix in [`mysql-readiness.md`](mysql-readiness.md).
- **Upsert semantics** — was the biggest builder gap: MySQL's
  `INSERT … ON DUPLICATE KEY UPDATE` (with `VALUES()`/`new.col`) does not map
  1:1 onto the existing `onConflict*` builder. **Abstraction decided +
  implemented in C2** (dialect-mapped `onConflict*` + `excluded()`; see the task
  detail).

A **MySQL-readiness report** + a **draft capability matrix row** (a fifth column

## Task detail (Workstream C) — the v0.7 `@sisal/mysql` enabler

(Priority/effort/status are in the top tracker; this is the per-task body.)
Status reflects a first-hand probe (June 2026) that rendered real builder
queries under the `"mysql"` dialect — so the **render-ready** vs **gap** calls
are verified, not assumed.

- **C1 — Pin the `"mysql"` render path. ✅ Done (2026-07-01)** —
  `packages/orm/mysql_dialect_test.ts`. **Pinned render-ready:** backtick
  quoting (`` `posts`.`id` ``), `?` placeholders, `ilike`→`like`, plain
  `SELECT`/`INSERT`, `FOR UPDATE` row locking,
  `onConflictDoUpdate`/`onConflictDoNothing` rendering
  `ON DUPLICATE KEY UPDATE`, and `excluded()` rendering MySQL's portable
  `values(col)` proposed-row reference. **Pinned caveats**, drizzle-parity style
  — fixing any fails the pin and must move this doc: MySQL upsert conflict
  targets validate but do not render, conflict `where` throws
  `ORM_DIALECT_UNSUPPORTED`, the C3 `returning *` gap (since resolved — C3
  guards it), **`distinctOn` unguarded** (a new finding the probe missed; since
  corrected by C3's guard sweep), and the portable date helpers
  (`dateTrunc`/`now`/`dateBin`) throwing `ORM_DIALECT_UNSUPPORTED` (only
  `postgres`+`sqlite` variants exist — the one remaining pinned gap, v0.7
  adapter work).
- **C2 — Upsert divergence. ✅ Done (2026-07-01) — dialect-mapped, semantics
  recorded.** The decision (answering the open question below): **one
  dialect-mapped `onConflict*` surface, not a separate Drizzle-style
  `onDuplicateKeyUpdate`** — the ETL load step (v0.10) needs the same builder
  statement to render on every dialect, and forking the callsite per dialect is
  the leak the risk section warns about. Under `mysql`,
  `onConflictDoUpdate`/`onConflictDoNothing` render `ON DUPLICATE KEY UPDATE` in
  the core render path, pinned by `packages/orm/mysql_dialect_test.ts`, with
  these recorded semantics:
  - **Conflict target: validated, not rendered.** ODKU fires on ANY unique-key
    violation; the API still requires and validates the target, and the
    semantics coincide with `ON CONFLICT` exactly when the target is the table's
    only unique constraint — the normal upsert grain. This is the one documented
    semantic widening.
  - **Conflict `where`: no MySQL equivalent.** Rendering one under `mysql`
    throws a typed `ORM_DIALECT_UNSUPPORTED` (pinned).
  - **`excluded(column)` → `values(col)`** — the one proposed-row spelling valid
    across MySQL 5.7→9.x (deprecated since 8.0.20 but present through the
    current manuals) **and** MariaDB, which never adopted the alias form.
    MySQL's 8.0.19+ `AS new` row alias is MySQL-only, so switching to it is
    version/variant-aware work owned by the `(engine, version)` dialect-key
    thread — not the version-less v0.6 dialect. The typed helper also fixes a
    latent naming-strategy footgun: a raw `` sql`excluded.hotScore` `` misses
    the `hot_score` physical mapping; `excluded()` resolves and quotes it
    (pinned in `set_values_expr_test.ts`).
  - **`onConflictDoNothing` → a no-op self-assignment** (`col = col`; the first
    resolvable target column, else the first primary-key column, else the first
    declared column). `INSERT IGNORE` was rejected — it swallows unrelated
    errors (data truncation, FK violations), which is not "do nothing on
    conflict".

  v0.7's adapter consumes this as-is; the row-alias upgrade and the MariaDB
  variant split are C5 / dialect-identity inputs.
- **C3 — `RETURNING` divergence. ✅ Done (2026-07-01) — typed guard +
  dialect-guard sweep.** The decision: under the version-less `"mysql"` dialect,
  **any `returning()` on insert/update/delete throws a typed
  `ORM_DIALECT_UNSUPPORTED` at render time** instead of emitting SQL the engine
  rejects. Why a guard and not MariaDB emission: MySQL 8/9 has **no**
  `RETURNING` at all, and MariaDB's support is per-statement **and** per-version
  (`DELETE` 10.0.5+, `INSERT`/`REPLACE` 10.5+, `UPDATE` only 13.0+ and
  single-table) — so even the MariaDB variant needs the `(engine, version)`
  dialect key; it cannot be expressed on a version-less dialect. The
  fetch-by-key fallback is an adapter/executor concern (a second round trip, not
  a render rewrite) — a v0.7 `@sisal/mysql` design input. Postgres/SQLite
  rendering is unchanged (both support `RETURNING`; the guard's misleading
  hardcoded "PostgreSQL-only" error suffix was dropped). **Bundled dialect-guard
  sweep** — the same change corrected every guard C1 pinned as wrong for MySQL,
  so the dialect now throws rather than rendering invalid SQL: `distinctOn`
  (+`mysql`), array operators (+`mysql`), data-modifying CTEs (+`mysql` —
  MySQL/MariaDB `WITH` is `SELECT`-only), `DELETE … USING` (+`mysql` — MySQL's
  multi-table form needs the target repeated in `USING`; mapping it is v0.7
  work), and a **new** guard on `UPDATE … FROM` (MySQL's equivalent is the
  multi-table `UPDATE t, s SET …` shape). All pinned in
  `packages/orm/mysql_dialect_test.ts`; row locking (`FOR UPDATE`/`FOR SHARE`)
  correctly stays unguarded.
- **C6 — Driver survey (Deno + Node). ✅ Done (2026-07-01) — benchmarked, driver
  chosen.** Full report with numbers:
  [`perf/MYSQL_DRIVER_SURVEY.md`](../perf/MYSQL_DRIVER_SURVEY.md) (probe:
  `perf/mysql_driver_survey.ts`, `deno task perf:mysql`; run against real MySQL
  8.4.10 **and** MariaDB 11.8.8 from Deno 2.9, dual-runtime-verified on Node
  26). **Chosen default: `npm:mysql2`** (MIT, de-facto standard, Deno+Node
  verified, prepared statements + pool + TLS, no `@db/postgres`-class stall —
  parameterized p50 ≈ 0.08 ms) **with `supportBigNumbers` + `bigNumberStrings`
  mandatory in the executor** — mysql2's default `BIGINT` decode is silently
  lossy past 2⁵³ (measured: …93 → …92); with the options it decodes to a
  precision-safe string, matching `@sisal/neon`'s convention. **Opt-in
  candidate: `npm:mariadb`** — fastest on every path (~1.5–2× lower sequential
  p50, ~24.5k vs ~20k pooled qps), `BigInt`-correct by default, but LGPL-2.1 →
  lazily-imported opt-in via the injectable-executor seam (the postgres.js
  pattern). **Watch: `jsr:@db/mysql`** — works (MySQL 8 `caching_sha2` auth,
  `BigInt`-correct, competitive latency) but rc-only (`3.0.0-rc.1`,
  `latest: null`) and Deno-only, breaking the npm door-open constraint.
  **Serverless: `npm:@planetscale/database`** = the future `@sisal/neon`-style
  adapter variant (HTTP, service-bound; survey-only). Key negative result: **no
  candidate has the Nagle-class parameterized-query stall** the pg survey found.
- **C4 — Type/DDL mapping. ✅ Done (2026-07-01) — probe-verified on both
  engines.** Full report: [`mysql-ddl-mapping.md`](mysql-ddl-mapping.md); probe:
  `perf/mysql_ddl_probe.ts` (`deno task perf:mysql:ddl`), run against MySQL
  8.4.10 **and** MariaDB 11.8.8 with the C6-chosen driver. Every column kind is
  mapped (probe-applied + round-tripped): `boolean` → `BOOLEAN` (= `TINYINT(1)`,
  `0`/`1` round-trip like the SQLite family), serial/bigserial →
  `INT`/`BIGINT AUTO_INCREMENT` (placement validated at generation time — must
  be a key, one per table), `json`/`jsonb`/`.array()` → `JSON` (parsed on MySQL,
  **text on MariaDB** — its `JSON` is a `LONGTEXT` alias), `timestamp` →
  `DATETIME(6)`, `timestamptz` → `TIMESTAMP(6) NULL` (2038 cliff on MySQL,
  probe-confirmed; MariaDB extended), `numeric`/`bigint` → string round-trips,
  `bytea` → `LONGBLOB`, `uuid` → `CHAR(36)` (+ `DEFAULT (uuid())` works on
  both). DDL rules with probe findings behind each: **table-level FKs only**
  (inline `REFERENCES` is _silently ignored_ by MySQL — the pg generator's
  after-CREATE ordering transfers as-is), paren-wrapped expression defaults (the
  only form both engines accept on TEXT/JSON columns), no
  `CREATE INDEX IF NOT EXISTS` (MySQL rejects), DESC ✓ / functional MySQL-8-only
  / partial unsupported → typed generation errors, `CHECK` enforced on both,
  **no `ENGINE`/`CHARSET` clause** (the collation-name trap), `GET_LOCK` as the
  migrator's advisory-lock analogue. Version floor: MySQL ≥ 8.0.16, MariaDB ≥

  RETURNING`
  MariaDB-only, `UPDATE … RETURNING` neither, ODKU `VALUES()` both / row alias

# Shared sections

## Risks

- **A stronger IR may be required** — if insert-from-select / filtered
  aggregates can't be expressed cleanly as fragments, v0.8 IR work grows. v0.6's
  job is to surface this early, not absorb it.
- **Dialect divergence on data-modifying CTEs** — Postgres supports them; SQLite
  "less fully". Risk of a builder that only works on one family.
- **Scope creep into ETL** — the temptation to "just build the runner." Held off
  by the explicit non-goals.
- **Locking portability** — `pg_advisory_lock` has no SQLite equivalent; a wrong
  abstraction here leaks into v0.10.
- **Version/variant-blind dialect identity** — `SqlDialect`/`SisalDialectName`
  are version-less 4-value unions (they even collapse neon→postgres,
  libsql→sqlite); MySQL-vs-MariaDB and 5.7-vs-8 need a finer key. Decide the
  `(engine, version)` shape **now**, before v0.7 ships `@sisal/mysql` and v0.8
  freezes the IR — a late axis breaks the exported `dialectGuard`/IR surface
  (the snapshot is versioned via `SCHEMA_SNAPSHOT_VERSION` and can migrate; the
  guard signature can't). See the
  [sequencing audit](roadmap-sequencing-audit.md).
- **(npm) deferred, not dropped** — the build + dual publish moves to v0.13+;
  the only risk is forgetting the cheap door-open constraints (Node-capable
  driver, runtime-agnostic code), captured in the
  [npm-release readiness report](npm-release-readiness.md).
- **(C) MySQL upsert is not Postgres upsert** — `ON DUPLICATE KEY UPDATE`
  differs materially from `ON CONFLICT`; a forced-fit abstraction would leak
  into v0.7 and the ETL load step. **Retired by C2 + C3 (2026-07-01):** the
  `"mysql"` render path is pinned by tests
  (`packages/orm/mysql_dialect_test.ts`) and no known construct renders wrong
  SQL — the upsert is dialect-mapped and everything MySQL can't express
  (`RETURNING`, `UPDATE … FROM`, `DELETE … USING`, data-modifying CTEs,
  `distinctOn`, array operators, the date helpers) throws a typed
  `ORM_DIALECT_UNSUPPORTED` at render time.

## Dialect compatibility notes

PostgreSQL is the reference ETL target (full insert-from-select, FILTER,
advisory locks, materialized views). Neon = Postgres surface but **prefer
single-statement / generated SQL** (WebSocket Pool supports interactive
transactions, but serverless favors one round trip). SQLite/libSQL: insert-from-
select and ON CONFLICT exist at the engine level (verify in-suite), `FILTER`
degrades to `CASE WHEN`, no materialized views, no advisory locks (use a lock
row / `BEGIN IMMEDIATE`); libSQL adds remote/Turso caveats. **MySQL (prospective
fifth dialect, Workstream C):** backtick quoting, `?` placeholders, and the
dialect-mapped `ON DUPLICATE KEY UPDATE` upsert (+ `excluded()`) are
render-ready; everything MySQL cannot express — `RETURNING`, `UPDATE … FROM`,
`DELETE … USING`, data-modifying CTEs, `distinctOn`, array operators, the
portable date helpers — throws a typed guard (C3). What remains for v0.7 is the
adapter itself: driver, type mapping (`BOOLEAN`=`TINYINT(1)`), DDL,
MySQL-vs-MariaDB variants, and mysql variants for the guarded shapes.
