# Sisal v0.6.0 roadmap — Foundations & Readiness

Where [v0.5.0](v0.5.0-roadmap.md) makes the OLTP builder boringly correct across
the four dialects, v0.6.0 is the **readiness** milestone: it earns the right to
build later packages (and reach a fifth dialect) by proving the foundation can
carry them. It is deliberately **three parallel workstreams**, all of which
_prepare the ground and ship no new public feature package_:

- **Workstream A — ETL Readiness Investigation.** The minimum ORM/core changes
  so a future `@sisal/etl` (v0.10) can be built without hacks.
- **Workstream B — Node.js + npm Runtime Readiness.** Make the packages
  installable/runnable on Node 24+ and publishable to npm alongside JSR, without
  giving up the Deno workflow. (This is the runtime/packaging track folded into
  v0.6; see [roadmap overview](roadmap.md).)
- **Workstream C — MySQL Support Investigation.** Map what a `@sisal/mysql`
  adapter ([v0.7](v0.7.0-roadmap.md) builds it) needs, given that the renderer
  already carries a latent `"mysql"` dialect. Investigation only — no adapter.

**This release builds nothing it investigates.** v0.6 answers "can Sisal express
ETL-shaped SQL cleanly?" and "what does MySQL support require?" — it ships
neither the ETL package nor the MySQL adapter. See
[release discipline](roadmap.md#release-discipline-non-negotiable).

# Workstream A — ETL Readiness Investigation

## Goal

Investigate and implement **only** the minimum ORM/core changes needed so a
future `@sisal/etl` package can express extract→transform→load as pushed-down
SQL without dropping to raw strings for the core shapes. Produce a **dialect
capability matrix** and a **proof-of-concept rollup** in docs/pseudo-code.

## Questions this release must answer

- Can Sisal express ETL-shaped SQL (grouped rollup → idempotent upsert) cleanly
  through the typed builder?
- Does the current internal query representation support this, or does it need a
  stronger SQL IR/AST? (See the [v0.8](v0.8.0-roadmap.md) IR work it feeds.)
- Can each dialect (pg/neon/sqlite/libsql) support the required SQL patterns?
- Which features belong in **core** and which belong in **dialect adapters**?
- What must be postponed to v0.8/v0.10?

## Scope — investigate & document support for

Set-based movement and aggregation (the rollup spine):

- `INSERT INTO … SELECT …` (insert-from-select) — **shipped in
  [v0.5 item 12](v0.5.0-roadmap.md)** (`insert(t).select(query)`, tested on all
  four engines). The original "single biggest ETL gap" is now a **verify + pin**
  task, not new work (see the PoC findings below).
- CTEs (`WITH`) — SELECT **and** data-modifying (Postgres-family) shipped in
  [v0.5 item 12](v0.5.0-roadmap.md); SQLite-family CTEs stay SELECT-only (typed
  guard).
- Aggregate expressions; `GROUP BY`; `HAVING` — **supported**.
- Conditional aggregation: `agg(…) FILTER (WHERE …)` — **shipped as `filter()`
  in [v0.5 item 9](v0.5.0-roadmap.md)**, and **native on the SQLite family**
  (the planned `CASE WHEN` fallback proved unnecessary).
- Upsert from select (`INSERT … SELECT … ON CONFLICT DO UPDATE`) — the pieces
  shipped (`insert().select()` + `onConflictDoUpdate`, v0.5); **verify they
  compose** into one statement per dialect.
- Typed raw-SQL escape hatches — **present** (`sql` tag binds params safely);
  the fallback for anything unbuilt.
- Computed expressions and expression **aliases** in projections — partial;
  needs a first-class typed alias surface for rollup metric columns.
- Date/time **bucket** abstraction — **shipped as `dateTrunc`/`dateBin` in
  [v0.5 item 9](v0.5.0-roadmap.md)** (per-dialect: `date_trunc`/`to_timestamp`
  on Postgres, `strftime`/`unixepoch` on the SQLite family).
- **Window functions** (`OVER`, moving averages, `lag`/`lead`, `rank`) —
  **absent, no builder at all**; the PoC's hard wall, owned by
  [v0.7](v0.7.0-roadmap.md).
- **Array projection** (`ARRAY[col, …]`) and **set-returning**
  (`unnest …
  WITH ORDINALITY`) — **absent**; raw SQL today (no array type on
  the SQLite family).

Execution & correctness primitives the runner will need:

- Batch execution; transaction boundaries — **today: supported** (`db.batch`,
  `db.transaction`). Note `db.batch` forbids cross-statement data dependencies.
- **Advisory locks or an equivalent locking strategy** — Postgres has
  `pg_advisory_lock` (already used internally by the migration history store — a
  reusable pattern); SQLite/libSQL need an equivalent (a lock row /
  `BEGIN
  IMMEDIATE`). Document the per-dialect strategy.
- **Checkpoint table design** — a watermark table (`job`, `window_end`,
  `updated_at`) so a run resumes from the last committed window.
- **Idempotent load strategy** — re-running a window must not double-count
  (upsert/replace semantics keyed by the rollup grain).
- **Backfill/replay requirements** — re-run an arbitrary historical window range
  deterministically.

## Deliverable — proof-of-concept rollup (now a runnable example)

A `post_events → post_hourly_stats` rollup demonstrating the desired shape (the
repo's existing `*-rising-feed` examples already bucket activity — this
generalizes that into an ETL rollup):

- **Extract** from a raw `post_events` table over a half-open time window
  `[from, until)`.
- **Group** by `post_id` and an hourly bucket.
- **Compute** `views`, `votes`, `comments`, `engagement_score`.
- **Load** into `post_hourly_stats` with idempotent upsert/replace.
- **Checkpoint** the window end (kept as a _future concern_ — design, don't
  ship).

The key SQL shape v0.6 must establish a path to express:

```sql
INSERT INTO post_hourly_stats (post_id, bucket, views, votes, comments, engagement_score)
SELECT post_id,
       date_trunc('hour', occurred_at) AS bucket,
       count(*) FILTER (WHERE kind = 'view')    AS views,
       count(*) FILTER (WHERE kind = 'vote')    AS votes,
       count(*) FILTER (WHERE kind = 'comment') AS comments,
       /* weighted */                            AS engagement_score
FROM post_events
WHERE occurred_at >= $from AND occurred_at < $until
GROUP BY post_id, date_trunc('hour', occurred_at)
ON CONFLICT (post_id, bucket)
DO UPDATE SET views = excluded.views, votes = excluded.votes,
              comments = excluded.comments, engagement_score = excluded.engagement_score;
```

This single statement exercises every audited gap: insert-from-select, a date
bucket, filtered aggregates (FILTER is native on the SQLite family since v0.5 —
no `CASE WHEN` fallback needed), and upsert-from-select. v0.6 decides which of
these become typed builders in v0.8.

## Findings — the `neon-activity-vectors` PoC (built first-hand)

[`examples/neon-activity-vectors`](../examples/neon-activity-vectors/README.md)
is a **runnable** realization of this PoC on Neon/Postgres, verified against
real Postgres. It runs the full chain — raw `post_events` → hourly buckets (a
`fold` insert-from-select) → consolidated stats with **moving averages** → an
`ARRAY[…]` activity-vector projection → cosine similarity → daily/monthly
**rollups** + event **pruning**. What it found about how far the v0.5 builder
actually reaches (this is the substance of the readiness investigation):

- **The rollup spine is largely builder-native already (post-v0.5).**
  `insert(t).select(query)` and `onConflictDoUpdate(...)` both ship (v0.5 item
  12, tested on all four engines), as do `filter(agg, cond)` and
  `dateTrunc`/`dateBin` (v0.5 item 9, native FILTER on the SQLite family). So
  `fold_events_to_buckets` / `rollup_daily` / `rollup_monthly`
  (`INSERT … SELECT … GROUP BY … FILTER … ON CONFLICT DO UPDATE`) are **probably
  expressible through the builder today**. The v0.6 question narrows from "can
  we express the rollup?" to **"verify the full statement composes + renders per
  dialect, and pin it"** — the example currently uses raw SQL functions for
  framing, so converting them is the concrete verification task (and the honest
  way to find the limit instead of hiding it).
- **Window functions are the one hard wall — no builder of any kind.** The
  moving averages
  (`avg(...) OVER (PARTITION BY … ORDER BY … ROWS BETWEEN 5
  PRECEDING AND CURRENT ROW)`)
  cannot be expressed at all: no `OVER`, `PARTITION BY`, `ROWS`/`RANGE`,
  `lag`/`lead`, `rank`, `row_number`. This is the single biggest analytics gap
  and is **owned by [v0.7](v0.7.0-roadmap.md)**; v0.6 records it as the blocking
  discovery.
- **Array construction and set-returning have no builder.** Storing a JS array
  into an array column is builder-native, but **projecting named columns into a
  SQL `ARRAY[…]`** (the feature-vector projection) and consuming one with
  `unnest … WITH ORDINALITY` are raw. On the SQLite family there is **no array
  type at all** (`json_array`/`json_each` instead) — a genuine per-dialect wall
  that justifies a separate `sqlite`/`libsql` example.
- **`CREATE FUNCTION` has no builder.** The whole `app.*` computation engine is
  raw `.sql`. (v0.5 B7 can _carry_ such DDL in a snapshot; it does not _emit_ a
  typed function.)
- **Clean / no gap:** the batch event insert, the stats/posts reads, and
  similarity candidate loading are all clean builder `select`; bigint ids
  round-trip as strings (documented); and the typed function caller
  (`defineFunction`/`db.call`) exists and could replace the raw `sql` calls.

**Net for v0.6:** the ETL rollup spine is closer to done than the original audit
assumed (v0.5 delivered the pieces) — so a primary v0.6 deliverable is to
**prove the builder expresses the rollup end-to-end and pin per-dialect render
tests**, rather than design new primitives. The genuinely-unbuilt analytics
shapes the PoC surfaced (window functions, array/`unnest` projection) are
recorded here and handed to [v0.7](v0.7.0-roadmap.md)/[v0.8](v0.8.0-roadmap.md).

# Workstream B — Node.js + npm Runtime Readiness

Make every package install and run on **Node.js 24+** from **npm** and
dual-publish JSR + npm, while the Deno workflow stays exactly as it is today.
The library cores already run under Node's type-stripping; the blockers are
packaging metadata, the six `jsr:`/`npm:` import sites, and the `.ts` import
extensions — not the library logic. B1–B8 below carry the full detail.

**Package naming (JSR `@sisal` vs npm `@scope`).** The `@sisal` scope is
published and owned on **JSR**, but is **unavailable on npm**. So the Deno
workspace, every source import, and the JSR packages keep the `@sisal/*` names
unchanged, while the npm artifacts publish under a **placeholder scope written
`@scope/*`** throughout this workstream. The final npm name is an open branding
decision (an alternate scope like `@sisal-db/*`, unscoped `sisal-*` packages,
your own scope, …); `@scope` is a deliberate stand-in so the work is not tied to
a name — swap it in one find-replace once chosen. Mechanically, the npm **build
remaps** the source `@sisal/*` scope to `@scope/*` (package `name`, dependency
names, and the internal import specifiers), so dev, test, and JSR publishing are
unaffected and keep using `@sisal/*`; only the published npm artifact carries
`@scope/*`.

## Where this came from

A Node/npm feasibility audit (June 2026) ran the actual codebase against real
**Node v26.4.0 / npm 11.16.0** in a throwaway git worktree, and built on the
signal already recorded in [editor-lsp.md](editor-lsp.md) (a plain TypeScript
language server reports the `@sisal/*` workspace imports as unresolved — "a
future npm packaging signal"). The audit's finding: **the repo is unusually
close to dual-runtime already.** The library cores carry zero Deno-runtime
coupling, all four adapters already _inject_ their driver, and the code reads
the `Deno` global defensively through `globalThis`. The blockers are
concentrated in **packaging metadata, module-specifier syntax, one static
import, and the dev toolchain** — not in the library logic.

Probes run under real Node:

| Probe                                                                    | Result                                                       |
| ------------------------------------------------------------------------ | ------------------------------------------------------------ |
| Load + run the pure `@sisal/orm` core (relative `.ts`, type-stripped)    | ✅ module graph instantiates and executes real logic         |
| `@sisal/pg` → `@sisal/orm` after adding minimal `package.json` `exports` | ✅ `createSchemaSnapshot` + DDL generation run cross-package |
| Load `pg/orm/pool.ts` (static `jsr:@db/postgres` import)                 | ❌ `ERR_UNSUPPORTED_ESM_URL_SCHEME: protocol 'jsr:'`         |
| Import bare `@sisal/orm` with no `package.json`                          | ❌ `ERR_MODULE_NOT_FOUND` (pure resolution, not a code bug)  |
| `import.meta.main`, `node:sqlite` (`DatabaseSync`), `node:test`          | ✅ all present                                               |

Three concrete blockers, none of which touch business logic:

1. **Bare cross-package specifiers** (`@sisal/orm`, `@sisal/migrate`, …) — Deno
   resolves these from the workspace; Node needs `package.json` `exports`.
   Proven resolvable by synthesizing minimal manifests in the worktree.
2. **`jsr:` / `npm:` URL-scheme imports** — Node's loader hard-rejects them.
   Only **six sites** (B2).
3. **`.ts` import extensions** on all 271 internal imports — Deno-mandatory, but
   they must become `.js` in published output, so a build step is required (item
   3).

The coupling inventory the audit produced:

- **Library source is essentially clean.** The only `Deno.*` in non-test source
  is `Deno.env` in `cli.ts`, already read via `globalThis.Deno?.env?.get` with a
  try/catch. The migration filesystem is abstracted behind
  `MigrationFileSystem`; `denoMigrationFileSystem()` throws cleanly when `Deno`
  is absent, leaving room for a `node:fs` sibling.
- **Drivers are injectable.** Every adapter accepts a caller-supplied
  pool/client/database; only the _default_ import is Deno-specific.
- **The dev toolchain is the largest surface** — 200 `Deno.test` calls in the
  unit suites, the custom `Deno.lint` plugin, and five `denoland/setup-deno` CI
  workflows.

## Sequencing & build tooling (Workstream B)

(Priority/effort/status are in the top tracker; B1–B8 detail sections follow.)

**Sequencing.** B1 → B2 → B4 are the de-risking spine and need no build tooling:
once manifests resolve `@sisal/*` (B1) and adapters stop importing `jsr:`/`npm:`
eagerly (B2), the dual-runtime test harness (B4) can turn `node --test` green
for the whole workspace — proving Node compatibility before any packaging
exists. B3 (build) then produces installable artifacts, B6 ships them, and
B5/B7/B8 round out the CLI, contributor tooling, and docs. A suggested milestone
order: **M1** = B1 + B4 (pure packages dual-run); **M2** = B2 (adapters
dual-run); **M3** = B3 (installable npm tarballs); **M4** = B5 + B8 (CLI +
integration + examples on Node); **M5** = B6 + B7 (CI + dual publish + lint
parity).

**Build-tool decision (open).** B3 can use Deno's official
[`dnt`](https://github.com/denoland/dnt) (Deno→Node transform — rewrites
specifiers and `.ts`→`.js`, emits dual ESM/CJS + `.d.ts` + `package.json`, keeps
Deno as source of truth) or a Node-native build (`tsdown`/`unbuild` +
`tsc rewriteRelativeImportExtensions`). The roadmap recommends **dnt** for the
first cut and revisiting only if a fully Node-native build becomes worthwhile.

## B1. Shared package manifests & workspace resolution — P1

**Problem.** Node cannot resolve the bare `@sisal/*` cross-package specifiers
(45 `@sisal/orm`, 20 `@sisal/migrate`, plus subpath imports) — Deno resolves
them from the `deno.json` workspace, but Node needs `package.json` `exports`.
The worktree audit confirmed this is _pure packaging metadata_: synthesizing
minimal manifests made `@sisal/pg` → `@sisal/orm` resolve and execute under Node
with no code change.

**Proposed work.** Add a `package.json` beside each `deno.json` (six library
packages + a root workspace manifest). Deno keeps reading `deno.json`; Node
reads `package.json`. Each package manifest declares `"type": "module"`,
`"engines": { "node": ">=24" }`, an `"exports"` map mirroring the `deno.json`
subpath exports (`.`, `./core`, `./orm`, `./migrate`, `./ddl`, `./schema`, and
migrate's `./cli`/`./workflow`) but pointing at built artifacts with
`import`/`require`/`types` conditions, cross-package deps via the `workspace:*`
protocol, and the real npm driver deps (B2). The root manifest declares
`"workspaces"` mirroring the `deno.json` workspace array. Each package's local
`"name"` stays `@sisal/*` so source imports resolve in the workspace during
dev/test; the npm **build** (B3) renames the published artifact to `@scope/*` —
the dev/JSR identity and the npm identity diverge only in published output

**Affected.** A new `package.json` per package + root; a generator under

**Acceptance criteria.**

## B2. Driver indirection & the six `jsr:`/`npm:` import sites — P1

**Problem.** Six import sites use Deno-only specifier schemes Node rejects
(`ERR_UNSUPPORTED_ESM_URL_SCHEME`). Critically, `@sisal/pg` imports its driver
**statically** (`pg/orm/pool.ts:2`, `pg/migrate/pool.ts:2`), so merely importing
the adapter fails on Node even though the executor is injectable — unlike
`@sisal/sqlite`/`@sisal/libsql`/`@sisal/neon`, which import their driver lazily.

**Proposed work.** Centralize each adapter's _default_ driver load behind a lazy
resolver that branches on `globalThis.Deno`, keeping the existing injection
seams (a caller-supplied pool/client/database always wins). Per site:

| Site                                                   | Change                                                                                                                                                                                                   |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pg/orm/pool.ts`, `pg/migrate/pool.ts`                 | static `jsr:@db/postgres` → **lazy**; default to `pg` (node-postgres) on Node, wrapped to the existing `PgPool`/`PgClient` shape (`client.query` → `queryObject`, both return `{rows, rowCount}`).       |
| `sqlite/orm/database.ts`, `sqlite/migrate/database.ts` | `jsr:@db/sqlite` → resolve to the built-in **`node:sqlite`** `DatabaseSync` on Node (no dependency) / `@db/sqlite` on Deno, wrapped to `SqliteLikeDatabase` (`prepare → { all, run: number }`, `close`). |
| `libsql/client.ts`                                     | drop the `npm:` prefix → `@libsql/client`; declare it a dependency. Works on Node unchanged.                                                                                                             |
| `neon/client.ts`                                       | already a **bare** `@neon/serverless`; map it to npm **`@neondatabase/serverless`** (identical `Pool`/`Client`/`*Config` API) — no code change beyond the dependency alias.                              |
| `cli.ts` adapter strings (`jsr:@sisal/<adapter>/…`)    | emit a Node-resolvable bare specifier when running under Node (`@sisal/pg/ddl` in source/dev, remapped to `@scope/pg/ddl` by the npm build, B3); keep `jsr:@sisal/…` under Deno.                         |

**Affected.** `@sisal/pg` (the one behavioral change — lazy default import),
`@sisal/sqlite`, `@sisal/libsql`, `@sisal/neon`, `packages/migrate/cli.ts`;
driver deps `pg` (+ `@types/pg`), `@libsql/client`, `@neondatabase/serverless`.

**Acceptance criteria.**

- Importing any adapter under Node succeeds with no driver installed until a
  real connection is opened (parity with the lazy adapters today).
- A Node consumer can open a real connection per engine: `pg` for Postgres,
  `node:sqlite` (or `better-sqlite3`) for SQLite, `@libsql/client` for libSQL,
  `@neondatabase/serverless` for Neon.
- The injected-driver path is unchanged on both runtimes; render output is
  identical.

## B3. npm build pipeline — dual ESM/CJS + types — P1

**Problem.** All 271 internal imports carry explicit `.ts` extensions (Deno
requires them), so published JavaScript and `.d.ts` must rewrite `.ts` → `.js`;
npm consumers via `tsc`/bundlers cannot import `.ts` specifiers. A build step is
therefore mandatory for npm even though Deno needs none.

**Proposed work.** A per-package build, ordered by the dependency DAG (`orm` →
`migrate` → adapters), that rewrites specifiers, rewrites `.ts` → `.js`, and
emits **ESM + CJS + `.d.ts`** into `dist/`, wired to the `package.json`
`exports` from B1. Recommended: Deno's `dnt`, configured to skip its own test
pass (B4 runs tests natively) and to omit the `Deno` global shim (runtime code
already guards `globalThis.Deno`). As part of the build, **remap the source
`@sisal/*` scope to the npm `@scope/*` scope** — the published package `name`,
its `@sisal/*` dependencies, and the internal `@sisal/*` import specifiers all
become `@scope/*` (`@sisal` is unavailable on npm; the JSR/dev identity stays
`@sisal/*` — see _Package naming_). `dnt` does this rename per package.

**Affected.** A build script per package; root build orchestration in `tools/`;
`package.json` `exports`/`files`/`dist`.

**Acceptance criteria.**

- `npm pack` of each package contains ESM + CJS entrypoints and `.d.ts` for
  every subpath export.
- A clean `npm install` of the packed tarballs, imported from both an ESM and a
  CJS Node 24 project, type-checks and runs.
- Build output for the pure packages contains no `jsr:`/`npm:` specifiers and no
  `.ts` import paths.
- The published npm output references only `@scope/*` names (no `@sisal/*`
  remains in the emitted `name`, deps, or import specifiers); JSR/dev output
  still uses `@sisal/*`.

# Workstream C — MySQL Support Investigation

## Goal

Determine exactly what a future `@sisal/mysql` adapter requires, so
[v0.7](v0.7.0-roadmap.md) can implement it without surprises — and decide which
work is **core** (shared dialect plumbing) vs **adapter** (driver, executor,
DDL, type mapping). Investigation and a capability report only; **no adapter
ships in v0.6.**

## What grounds this — MySQL is latent, not absent

A June 2026 audit found the renderer already half-knows MySQL, but no adapter
exists:

- **Already present (core render layer):** `SqlDialect` includes `"mysql"`
  (`core/sql.ts`); the serializable snapshot dialect type includes `"mysql"`
  (`schema.ts`); `quoteIdentifier` emits **backtick** quoting for `mysql`
  (`core/sql.ts`); placeholders fall through to `?` (only `postgres` uses `$N`);
  `ilike` degrades to case-insensitive `LIKE` (MySQL has no `ILIKE`); and the
  builder comments already treat `FOR UPDATE`/`FOR SHARE` and row-value tuples
  as Postgres/**MySQL**-supported.
- **Absent (everything an adapter needs):** there is **no `@sisal/mysql`
  package** — no `MYSQL_DIALECT` constant, no executor, no driver/pool, no
  migrate wiring, **no MySQL DDL generator** (only
  `generate{Postgres,Sqlite,Libsql}UpStatements` exist), no column-type → MySQL
  type mapping, no integration suite, no `docs/mysql-compatibility.md`, no
  feature-matrix column, and no chosen MySQL driver.

## Scope — investigate & document

- **Driver choice & runtime** — survey MySQL drivers usable from Deno (and Node,
  per Workstream B): `jsr`/`npm` options, connection/pool API,
  prepared-statement and parameter-binding model, TLS, and any serverless
  variants (e.g. PlanetScale-style HTTP). Note the injectable-executor seam
  every adapter uses.
- **Dialect completeness** — verify/extend the latent `"mysql"` render path:
  identifier quoting (backtick + escaping), `?` placeholders, `LIMIT/OFFSET`,
  `ON DUPLICATE KEY UPDATE` vs `ON CONFLICT` (MySQL upsert differs
  **materially** from Postgres/SQLite), `RETURNING` (MySQL lacks it pre-8.0.21 /
  MariaDB differences), `ILIKE`→`LIKE`, `FOR UPDATE`/`FOR SHARE`, and whether
  the `dialectGuard` set is correct for MySQL (row locking is supported;
  `distinctOn` and array operators are not).
- **Type system & round-trips** — MySQL type mapping (`BOOLEAN`=`TINYINT(1)`,
  `JSON` type, `DATETIME`/`TIMESTAMP`/timezone handling, `DECIMAL`/`BIGINT`
  string-vs-number like the pg `numeric` case), and `bytea`/`BLOB` shapes.
- **DDL generation** — what a `generateMysqlUpStatements` must emit (engine,
  charset/collation, `AUTO_INCREMENT`, index/`CHECK` support quirks).
- **MySQL vs MariaDB** — decide whether "MySQL" means MySQL 8+, MariaDB, or
  both, and where they diverge (`RETURNING`, sequences, JSON functions).
- **Upsert semantics** — the biggest builder gap: MySQL's
  `INSERT … ON DUPLICATE
  KEY UPDATE` (with `VALUES()`/`new.col`) does not map
  1:1 onto the existing `onConflict*` builder. Document the abstraction.

## Deliverable

A **MySQL-readiness report** + a **draft capability matrix row** (a fifth column
alongside pg/neon/sqlite/libsql) covering upsert, returning, CTEs, window
functions, filtered aggregates, row locking, date bucketing, and transaction
behavior — classifying each as **render-ready today / needs core work / needs
adapter work / MySQL-specific divergence**.

## Task detail (Workstream C) — the v0.7 `@sisal/mysql` enabler

(Priority/effort/status are in the top tracker; this is the per-task body.)
Status reflects a first-hand probe (June 2026) that rendered real builder
queries under the `"mysql"` dialect — so the **render-ready** vs **gap** calls
are verified, not assumed.

- **C1 — Pin the `"mysql"` render path.** **Probed render-ready:** backtick
  quoting ✅ (`` `posts`.`id` ``), `?` placeholders ✅, `ilike`→`like` ✅, plain
  `SELECT`/`INSERT` ✅. Capture these as render tests so they don't regress.
- **C2 — Upsert divergence (gap confirmed).** `onConflictDoUpdate` renders
  Postgres `on conflict (…) do update` under `mysql`; MySQL needs
  `ON DUPLICATE KEY UPDATE col = VALUES(col)` (or 8.0.20+ `AS new … new.col`).
  Design the abstraction in the core render path.
- **C3 — `RETURNING` divergence (gap confirmed).** The renderer emits
  `returning *` under `mysql`; MySQL 8 has **no** `RETURNING` (MariaDB 10.5+
  does). Design dialect/version handling — emit on MariaDB, else a fetch-by-key
  fallback or a typed guard.
- **C6 — Driver survey (Deno + Node).** `jsr`/`npm` MySQL drivers, pool /
  prepared-statement / TLS model, serverless (PlanetScale HTTP), and the
  injectable-executor seam every adapter uses.
- **C4 — Type/DDL mapping** for the future `generateMysqlUpStatements`:
  `BOOLEAN`=`TINYINT(1)`, serial→`AUTO_INCREMENT`, `JSON`,
  `DATETIME`/`TIMESTAMP` /tz, `DECIMAL`/`BIGINT` string-vs-number, `BLOB`,
  engine/charset.
- **C5 — MySQL-vs-MariaDB capability split** (`RETURNING`, sequences, JSON fns)
  — decide one adapter with feature flags vs documented divergences.

**Handoff to v0.7:** C1/C6 are the readiness baseline; **C2 and C3 are the two
render-path gaps that must be designed in v0.6** (they produce wrong SQL today),
so v0.7 can build `@sisal/mysql` against a correct dialect instead of patching
it. C4/C5 are adapter-build inputs v0.7 consumes.

## Workstream C non-goals

No `@sisal/mysql` package; no driver dependency added; no MySQL integration
tests; no DDL generator implementation. Those are [v0.7](v0.7.0-roadmap.md).

# Shared sections

## Risks

- **A stronger IR may be required** — if insert-from-select / filtered
  aggregates can't be expressed cleanly as fragments, v0.8 IR work grows. v0.6's
  job is to surface this early, not absorb it.
- **Dialect divergence on data-modifying CTEs** — Postgres supports them; SQLite
  "less fully". Risk of a builder that only works on one family.
- **Scope creep into ETL** — the temptation to "just build the runner." Held off
  by the explicit non-goals.
- **Locking portability** — `pg_advisory_lock` has no SQLite equivalent; a wrong
  abstraction here leaks into v0.10.
- **(B) `.ts`→`.js` + scope remap** at build is mandatory for npm; getting the
  dual JSR `@sisal/*` ↔ npm `@scope/*` identity wrong breaks consumers.
- **(C) MySQL upsert is not Postgres upsert** — `ON DUPLICATE KEY UPDATE`
  differs materially from `ON CONFLICT`; a forced-fit abstraction would leak
  into v0.7 and the ETL load step. And the latent `"mysql"` render path is
  **untested** — it may be subtly wrong.

## Open questions

- Should `INSERT … SELECT` be a first-class builder, or only reachable through a
  data-modifying CTE (v0.5 item 12)?
- Does the readiness work need an introspectable expression AST, or is the
  compose-only fragment IR enough to compile ETL SQL? (Feeds v0.8.)
- Checkpoint table: owned by `@sisal/etl`, or a `@sisal/migrate`-managed system
  table?
- Is a portable advisory-lock abstraction worth it, or is locking Postgres-first
  with documented degradation on SQLite/libSQL?
- **(C)** Does "MySQL" mean MySQL 8+, MariaDB, or both — and how is
  `RETURNING`'s uneven support handled? Should upsert get a MySQL-aware
  `onDuplicateKeyUpdate` surface, or a dialect-mapped `onConflict`?

## Acceptance criteria

- A written **ETL-readiness report** answering the five questions, with the
  capability matrix and the `post_events → post_hourly_stats` PoC, committed to
  docs.
- Each required SQL shape is classified: **buildable today / needs a v0.8
  builder / stays raw**, per dialect.
- Any **minimum** core changes identified as truly prerequisite (not the ETL
  package itself) are itemized with effort, or explicitly deferred to v0.8.
- **(B)** `npm install @scope/*` runs on Node 24+; unit suite green under both
  runtimes; dual JSR+npm publish gated to matching versions.
- **(C)** A committed **MySQL-readiness report**, a draft capability-matrix
  column, and a chosen driver, with every required item classified
  core-vs-adapter and the upsert/`RETURNING`/type-mapping divergences documented

## Dialect compatibility notes

PostgreSQL is the reference ETL target (full insert-from-select, FILTER,
advisory locks, materialized views). Neon = Postgres surface but **prefer
single-statement / generated SQL** (WebSocket Pool supports interactive
transactions, but serverless favors one round trip). SQLite/libSQL: insert-from-
select and ON CONFLICT exist at the engine level (verify in-suite), `FILTER`
degrades to `CASE WHEN`, no materialized views, no advisory locks (use a lock
row / `BEGIN IMMEDIATE`); libSQL adds remote/Turso caveats. **MySQL (prospective
fifth dialect, Workstream C):** backtick quoting + `?` placeholders are
render-ready; the divergences to map are upsert (`ON DUPLICATE KEY UPDATE`),
uneven `RETURNING`, `BOOLEAN`=`TINYINT(1)`, and MySQL-vs-MariaDB — all
implemented in v0.7, not here.
