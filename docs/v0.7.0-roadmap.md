# Sisal v0.7.0 roadmap — Analytics Readiness & MySQL Support

v0.7 carries **two parallel workstreams**:

- **Workstream A — Analytics Readiness Investigation.** The OLAP analogue of
  v0.6's ETL readiness: design the analytical IR and prototype the API, but **do
  not ship `@sisal/analytics`** (that's [v0.11](v0.11.0-roadmap.md)).
- **Workstream B — MySQL Support Implementation.** Build the `@sisal/mysql`
  adapter on the findings of
  [v0.6 Workstream C](v0.6.0-roadmap.md#workstream-c--mysql-support-investigation).
  This one **does ship code** — a fifth dialect adapter.

(Per the roadmap being a living guideline, these two are bundled by sequencing,
not by theme; either could move to its own release if scope demands.)

## Priority summary — everything to track

All v0.7 tasks, consolidated. **Stream:** A = Analytics readiness (investigate,
don't ship) · B = MySQL adapter (ships code). Each ID has a detail section in
the matching workstream below.

**What v0.6 already delivered** (so this table is smaller than the workstream

# Workstream A — Analytics Readiness Investigation

The OLAP readiness question: how should Sisal support analytical queries without
bloating the OLTP ORM? This is the weakest current area — a June 2026 audit
found **no window-function support at all** (no `OVER`, `PARTITION BY`, ranking,
`lag`/`lead`, percentiles, moving averages) and none of it on any prior roadmap.
So the analytical IR is genuinely new design, not a refactor.

## Goal

Investigate how a typed, OLAP-oriented analytics layer can be built **as a
package on top of `@sisal/core`** (see [architecture](architecture.md)), without
the OLTP builder depending on it. Produce an **analytical IR sketch** and prove
the result types can be inferred cleanly.

## Questions this release must answer

- Can analytics be built as a separate package on core primitives (yes per the
  audit — core exposes a renderable fragment IR), and where is the seam?
- What **analytical IR** is needed (source/dimensions/metrics/windows/…)?
- What belongs in **core expressions** vs. a future `@sisal/analytics`?
- Can analytical **result types** be inferred from metric/dimension definitions?
- Which features are **portable** across pg/neon/sqlite/libsql/future mysql, and
  which are **Postgres-first**?

## Scope — investigate / prototype the API concepts

- **metrics** and **dimensions** (the core of a typed analytical query)
- **time buckets** (builds on the v0.6 bucket abstraction)
- **windows**: moving averages, **ranking**, **lag**, **lead**
- **percentiles** (`percentile_cont`/`disc`) — investigation only
- **period comparisons** (this window vs previous)
- **rollup querying** (query the tables `@sisal/etl` will produce)
- **result typing** for computed metrics

## Deliverable — analytics IR sketch

A documented IR shape (design, not implementation), e.g.:

```
source        — a table / rollup / subquery
joins         — dimension joins
filters       — predicates (reuse core operators)
dimensions    — group keys (incl. time buckets)
metrics       — typed aggregate/window expressions
windows       — partition/order/frame for windowed metrics
derivedFields — computed-from-metrics expressions
order, limit
executionPreference — pushdown target hint (pg today; DuckDB later, v0.13+)
```

The sketch must show how it **compiles into the core fragment IR** and renders
per dialect, and how it infers a typed result row from `{dimensions, metrics}`.

## Examples to include (mapping to the repo's existing feeds)

- **`/new` feed stays ORM territory** — plain keyset SELECT; not analytics.
- **`/hot` feed** uses a cached/stored score (the `neon-hot-feed` example) —
  read a precomputed column, not a live analytical query.
- **`/rising` feed** benefits from **analytics over recent rollups/events** —
  the moving-window score the `*-rising-feed` examples currently hand-write in
  SQL/TS is the canonical analytics-IR target.
- **Dashboard metrics** should query **prepared rollups** (`@sisal/etl` output)
  where possible, not raw events.

## Non-goals

No polished analytics package; no renderer integration; no production dashboard
helper; no DuckDB-first implementation; no ETL runtime.

## Risks

- **Bloating the ORM** — analytics primitives leaking into `@sisal/orm`.
  Mitigated by designing for `@sisal/core` + a separate package from day one.
- **Portability illusion** — windows/percentiles render differently (or not at
  all) on SQLite/libSQL; the IR must encode capability, not assume Postgres.
- **Type-inference complexity** — inferring result rows from metric/dimension
  maps may strain TS; prototype before committing.
- **IR over-design** — risk of a grand relational algebra. Keep it a compile
  sketch tied to real feed examples.

## Status (2026-07-01)

Workstream A is complete. The committed
[analytics-readiness report](analytics-readiness.md) answers the five readiness
questions, records the analytical IR sketch, includes the `/rising` rollup
walkthrough, and classifies analytical features across PostgreSQL, Neon, SQLite,
libSQL, and future MySQL/MariaDB. The network-free
[`analytics_result_inference_test.ts`](../packages/orm/analytics_result_inference_test.ts)
prototype proves a dimensions/metrics/derived-fields map can infer an exact
readonly result row.

A3 chose the **documented future core seam**: a minimal `over()`/window
expression primitive belongs in core SQL expressions because `OVER (...)` is
generic SQL grammar, while the rich metric/dimension/window-domain API belongs
in the future `@sisal/analytics` package. v0.7 does **not** ship `over()` or any
window helper.

No `@sisal/analytics` package exists, no public API/export was added, and
`docs/feature-matrix.md` remains unchanged until analytics features are
implemented and backed by integration scenarios.

## Open questions

- Does analytics need the introspectable AST flagged in
  [v0.8](v0.8.0-roadmap.md), or can it compile straight to fragments? **Answered
  for v0.7:** compile straight to fragments for the preview IR; a transformable
  AST stays v0.8+ work only if later packages need rewriting.
- Are window functions a **core** primitive (shared with the ORM builder) or an
  **analytics-package** construct? **Answered by A3:** a minimal `over()`/window
  expression belongs in core; the rich metric/dimension API belongs in the
  analytics package.
- Percentiles: portable abstraction or Postgres-first? **Classified by A4:**
  Postgres-first until other engines have probe-backed semantics or an explicit
  approximation policy.

## Acceptance criteria

- A committed **analytics-readiness report** + **IR sketch** answering the five
  questions, with the per-feed examples.
- A typed-result-inference **prototype** (types only, no package) showing a
  metric set → row type.
- A portability classification per analytical feature across five dialects,
  including the future MySQL/MariaDB adapter.
- No `@sisal/analytics` package, renderer, or dashboard helper exists.

## Affected packages

`@sisal/orm`/core (candidate minimal window/expression primitives; IR seam),
investigation against the four shipped adapters plus the future MySQL/MariaDB
adapter. No new package created.

- Docs: the `/rising` analytics-IR walkthrough over a rollup table.
- Type-level prototype tests for metric→row inference (network-free).
- The analytics example targets are preserved as product-shaped contracts in
  [`examples/advanced-sql-contracts/`](../examples/advanced-sql-contracts/README.md):
  [02-window-analytics](../examples/advanced-sql-contracts/02-window-analytics.md),
  [03-sessionization](../examples/advanced-sql-contracts/03-sessionization.md),
  [04-top-n-per-group](../examples/advanced-sql-contracts/04-top-n-per-group.md),
  [05-cohort-retention](../examples/advanced-sql-contracts/05-cohort-retention.md),
  and
  [06-funnel-analysis](../examples/advanced-sql-contracts/06-funnel-analysis.md)
  — each becomes a runnable example once the window/analytics surface lands.

## Dialect compatibility notes

PostgreSQL/Neon: full window functions, percentiles, FILTER. SQLite/libSQL:
window functions exist in modern SQLite but PARTITION/frame coverage and
percentiles vary — classify each as portable / degraded / Postgres-first. The IR
must carry an execution-preference + capability gate so unsupported analytics
fail typed (à la the v0.5 `dialectGuard`), not as raw engine errors. **MySQL**
(landing this release, Workstream B): MySQL 8 has window functions and CTEs, so
much of the analytics surface is portable — the readiness IR should reserve a
MySQL column from the start.

# Workstream B — MySQL Support Implementation

Build the `@sisal/mysql` adapter from the
[v0.6 Workstream C](v0.6.0-roadmap.md#workstream-c--mysql-support-investigation)
findings — Sisal's fifth dialect and first new adapter since libSQL. The
`"mysql"` dialect is no longer merely latent: after v0.6 it **renders correct
SQL or throws a typed guard for every construct** (quoting, placeholders,
`ilike`→`LIKE`, locking, the dialect-mapped upsert + `excluded()`; guards on
`RETURNING`, `UPDATE … FROM`, `DELETE … USING`, data-modifying CTEs,
`distinctOn`, array operators, `FULL JOIN`, `filter()`, the date helpers). v0.7
turns that correct dialect into a real, tested adapter — and lifts the guards
with real mysql renderings (`filter()`/date helpers in B2, multi-table mutation
joins in B10).

## Goal

Ship `@sisal/mysql` at OLTP feature-parity with the other adapters **wherever
MySQL supports it**, with the divergences (upsert, `RETURNING`, types) handled
explicitly — not silently mis-rendered.

## Why this is low-friction architecturally

Sisal's adapter conventions make a new engine an additive change: **every
adapter has the same internal shape** (`<adapter>/orm/` = dialect · executor ·
driver · pool · errors; `<adapter>/migrate/` = driver · history · migrator ·
ddl), the real driver is imported **lazily** with an **injectable executor** (so
unit tests stay network-free), and the CLI's `sisal init` targets are a single
`INIT_TARGETS` registry — **adding a database target is one registry entry**. So
`@sisal/mysql` slots into existing seams rather than reworking core.

## Scope — build

- **`@sisal/mysql` package** mirroring `@sisal/pg`/`@sisal/sqlite`:
  `MYSQL_DIALECT = "mysql"`, executor, lazy driver + pool, errors; and
  `migrate/` (driver, history, migrator, `ddl`).
- **Driver integration (B4)** — wire the drivers **chosen and benchmarked in
  v0.6 C6**: `npm:mysql2` as the lazily-imported default with
  `supportBigNumbers` + `bigNumberStrings` **mandatory** (its default `BIGINT`
  decode is silently lossy past 2⁵³), `npm:mariadb` as the lazy opt-in (the
  postgres.js pattern), behind the injectable-executor seam. Executor value
  normalization per C4: JSON parse-on-read (MariaDB returns text — its `JSON` is
  a `LONGTEXT` alias), the `TINYINT(1)` → boolean decision, `Buffer` →
  `Uint8Array`.
- ~~**Correct/extend the latent `"mysql"` render path**~~ — **done in v0.6**: C1
  pinned quoting/placeholders/`ilike`/locking, C2 mapped the upsert, and the
  C3/C5 guard sweep corrected every `dialectGuard` set — **no construct renders
  wrong SQL under `mysql`**. ~~What remains is **lifting** two typed throws with
  real renderings (B2).~~ **Done — B2 (2026-07-01)**: `filter()` renders the
  `CASE WHEN` fallback and the date helpers render their mysql variants; see the
  task detail.
- ~~**Upsert**~~ — **decided and implemented in v0.6 C2**: `onConflict*`
  dialect-maps to `ON DUPLICATE KEY UPDATE` in core, with `excluded()` →
  `values(col)` (target validated-not-rendered; conflict `where` throws; DO
  NOTHING → no-op self-assignment). The adapter **consumes** this; the 8.0.19+
  `AS new` row alias is variant/version work on the B1 axis.
- **`RETURNING` (B7)** — the render path already throws a typed guard (v0.6 C3;
  MySQL 8/9 has **no** `RETURNING`, and MariaDB's is per-statement and
  per-version). The adapter work is the execution strategy: a fetch-by-key
  fallback in the executor, and/or lighting MariaDB
  `INSERT`/`DELETE …
  RETURNING` per-statement through the B1 capability
  descriptor.
- **`generateMysqlUpStatements` (B5)** — additive-only DDL implemented **to the
  probe-verified C4 spec** ([mysql-ddl-mapping.md](mysql-ddl-mapping.md)):
  table-level FKs only (inline `REFERENCES` is silently ignored by MySQL),
  paren-wrapped expression defaults, **no** `ENGINE`/`CHARSET` clause, no
  `CREATE INDEX IF NOT EXISTS`, and the three generation-time validations
  (`AUTO_INCREMENT` placement, no bare `TEXT` keys, partial/functional-index
  rejection); destructive diffs withheld like the other generators. Version
  floor: MySQL ≥ 8.0.16, MariaDB ≥ 10.10.
- ~~**Type mapping & round-trips**~~ — **designed and probe-verified in v0.6
  C4** (the full kind→type table incl. `BOOLEAN`=`TINYINT(1)`, `timestamp` →
  `DATETIME(6)`, `timestamptz` → `TIMESTAMP(6)` with the 2038 caveat, `.array()`
  → `JSON`, `bytea` → `LONGBLOB`); v0.7 implements it in B4/B5.
- **Advisory-lock equivalent (B6)** — `GET_LOCK`/`RELEASE_LOCK` (named locks)
  for the migration history store (parity with pg's `pg_advisory_lock`), as
  recorded in C4 — also the lock the v0.6 A2 ETL contract expects.
- **CLI (B9)** — one `INIT_TARGETS` entry (id/aliases/dialect/connection hints).
- **Tests, docs, matrix (B8)** — fill the reserved `"mysql"` slot in the
  consolidated integration structure: an `IntegrationTarget` descriptor
  (capabilities/value shapes fall straight out of the C4/C5 probes) + a
  `mysql_family_scenarios.ts` list, gated `SISAL_MYSQL_IT=1` (Docker
  `mysql:8.4` + `mariadb:11`, the C-series probe servers),
  `docs/mysql-compatibility.md`, and a **fifth column in the feature matrix**
  (test-backed, per the existing guard).

## Task detail (Workstream B)

- **B1 — the `(engine, variant, version)` dialect axis. ✅ Done (2026-07-01).**
  The C5 dialect-identity decision is now core code, landed **before the v0.8 IR
  freeze** as required, pinned by `packages/orm/dialect_identity_test.ts`:
  - **`DialectIdentity`** (`{ dialect, variant?, version? }`) — `SqlDialect`
    stays the render key; a bare dialect means "base engine, version unknown".
    Accepted everywhere a dialect was
    (`renderSql`/`renderToPlan`/`normalizeSqlInput`, non-breaking), carried by
    the `Database` facade as `dialectIdentity`
    (`createDatabase({ dialect, variant, version })`, inherited by transaction
    facades) — the adapter (B4) fills it from the server handshake.
  - **`dialectGuard` generalized, declaratively** — guard chunks stay
    serializable data (the IR-freeze constraint): targets may narrow to one
    `variant`, and `unless` refinements (`{ variant?, minVersion? }`) lift a
    guard per identity. **Fail closed:** an unknown version never satisfies
    `minVersion` — capabilities light up only when the adapter has identified
    the server. `compareServerVersions` (dotted-numeric prefix, suffix-blind:
    `"11.8.8-MariaDB-ubu2404"`) is exported for adapters.
  - **First real consumer:** the `RETURNING` guard split per statement kind with
    the probe-verified MariaDB floors — `INSERT` 10.5+, `DELETE` 10.0.5+,
    `UPDATE` 13.0+ — so a `mariadb` identity renders
    `INSERT`/`DELETE …
    RETURNING` while `UPDATE … RETURNING` stays guarded
    until 13.0, and plain `mysql` keeps the C3 behavior unchanged (dormant until
    B4 wires the identity).
  - **Snapshot half:** `SisalSchemaSnapshot` gains optional
    `dialectVariant`/`dialectVersion` (additive — older snapshots unchanged,
    `SCHEMA_SNAPSHOT_VERSION` stays 2), accepted by `createSchemaSnapshot` for
    B5's variant-aware DDL decisions.
  - Open for B4: the identity is static at facade construction, but adapters
    connect lazily — the connect path needs a resolve-then-build step (or a
    re-identify hook) to fill variant/version from `select version()`.
- **B2 — mysql renderings for `filter()` + the date helpers. ✅ Done
  (2026-07-01).** The last two typed throws are lifted with real renderings,
  **executed live on MySQL 8.4.10 and MariaDB 11.8.8 with semantic checks**
  (identical values to the pg/sqlite behavior) before pinning in
  `packages/orm/mysql_dialect_test.ts`:
  - **`filter()` → `agg(CASE WHEN cond THEN operand END)`.** The compose-only
    fragment IR cannot introspect an opaque `sum("col")`, so the exported
    aggregates (`count`/`countDistinct`/`sum`/`avg`/`min`/`max`) now stamp
    rebuild metadata into a WeakMap (the `RESULT_METADATA` precedent — frozen
    fragments and the public surface untouched); `filter()` rebuilds the mysql
    variant from it (`count(*)` counts a literal `1`; `countDistinct` keeps
    `DISTINCT`; NULLs from non-matching rows are skipped by every aggregate,
    matching `FILTER` semantics — verified with live sums/counts/ averages). A
    hand-written `` sql`…` `` aggregate has no metadata and keeps the typed
    throw — an honest bound, and a data point for the v0.8 introspectable-IR
    question.
  - **Date helpers**: `dateTrunc` → `DATE_FORMAT` (string result like the SQLite
    family; MySQL minutes are `%i`), `now()` → `NOW(6)` (microseconds, matching
    pg), `dateAdd`/`dateSub` → nested `DATE_ADD(…, INTERVAL ? unit)` with bound
    quantities, `dateBin` → `FROM_UNIXTIME(FLOOR(UNIX_TIMESTAMP(src)/N)*N)`.
    Live checks covered hour-bucket grouping, 30-minute binning, ±shifts, and
    the moving-window predicate (`gte(col, dateSub(now(), {...}))`) on both
    engines.
  - **B4 finding (recorded):** mysql2's binary protocol rejects a bound
    `LIMIT ?` on MySQL 8.4 (`Incorrect arguments to mysqld_stmt_execute`) but
    accepts it on MariaDB 11.8; identical SQL passes via the text protocol on
    both. The B4 executor must handle LIMIT/OFFSET binding (coerce, inline
    validated integers, or route through `query()`).
- **B3 — scaffold `@sisal/mysql`. ✅ Done (2026-07-01).** `packages/mysql/`
  exists with the shared adapter shape (`orm/` = dialect · errors · pool ·
  executor · driver · mod), wired into the workspace, `check`, `test`, and docs
  gates, and **smoke-tested against live MySQL 8.4.10 and MariaDB 11.8.8**
  (builder round trip through the real lazy `mysql2` pool, the C2 ODKU upsert,
  transaction rollback, and MariaDB `INSERT … RETURNING` returning real rows
  through the B1 identity):
  - **Injectable executor seam**
    (`connect({ executor | pool | client |
    url })`) — the 8 unit tests run
    network-free against a recording fake, covering result mapping (row arrays
    vs mysql2's affected-rows header), transaction isolation/rollback, Temporal
    param normalization, batch atomicity, pool ownership on close, and
    end-to-end facade rendering.
  - **`mysql2/promise` imported lazily** on first connection (the
    `@sisal/libsql` discipline); the pool always sets the C6-mandated
    `supportBigNumbers` + `bigNumberStrings`.
  - **Text protocol (`query()`) is the execution path**, sidestepping the B2
    `LIMIT ?` binary-protocol finding by construction; a prepared-statement mode
    can arrive as a B4 opt-in.
  - **Identity plumbing:** `connect({ variant, version })` fills the B1
    `dialectIdentity` (exported `MARIADB_VARIANT`); adapter-side auto-detection
    from the handshake is B4/B9 work.
  - Exports are `.` and `./orm` only; `./migrate` and `./ddl` land with B5/B6.
    JSR publish dry-run passes for `@sisal/mysql@0.7.0`.
- **B4 — drivers + executor hardening. ✅ Done (2026-07-01).** Verified live on
  both engines **× both drivers** (four combinations); pinned by 6 new
  network-free unit tests (14 total in the package):
  - **Identity auto-detection resolves B1's lazy-connect question:** `connect()`
    runs `select version()` once (through the executor, so it works for
    `url`/`pool`/`client` sources alike), parses it with the new
    `parseMysqlServerVersion` (MariaDB self-identifies in the string), and
    builds the facade with the filled identity. Defaults: **on** for real
    sources without an explicit `variant`/`version`, **off** for injected
    executors (the test seam); `detectVersion: false` restores a fully lazy
    connect (identity stays base — fail closed, everything version-gated stays
    guarded). Live proof: against the MariaDB container, `INSERT … RETURNING`
    returns rows **with zero configuration**; against MySQL 8.4 it stays a typed
    guard.
  - **`connect({ driver: "mariadb" })` opt-in** — the C6 performance candidate,
    adapted to the shared `MysqlPool` contract (`adaptMariadbPool`) and resolved
    through a **runtime-computed specifier**, so the LGPL connector stays a
    soft, run-time-only dependency (the postgres.js pattern; the publish
    dry-run's `unanalyzable-dynamic-import` warning is that pattern's
    signature). The mysql2-compatible `supportBigNumbers`+`bigNumberStrings`
    options hold on it too (live: bigint → `"9007199254740993"` on both
    drivers).
  - **Live-probe catch:** the MariaDB connector **JSON-serializes a plain
    `Uint8Array` parameter** (`{"0":7,…}` — silent BLOB corruption); the adapter
    now re-views binary params as Node `Buffer`s (no copy) before handing them
    over. Pinned in the unit tests; both drivers now round-trip `[7,8,9]`
    byte-identically.
  - **Value normalization:** BLOB reads are re-viewed as plain `Uint8Array`s
    (matching `columns.bytea()`'s inferred type across adapters). **`TINYINT(1)`
    stays `0`/`1`** — decision recorded below. JSON on MariaDB stays text (its
    `JSON` is a `LONGTEXT` alias and the wire protocol cannot distinguish it
    from `TEXT`, so a mechanical executor-level parse is impossible without
    guessing) — a documented variant value-shape for the B8 compat doc, exactly
    like the SQLite family's json-as-text row.
- **B5 — `generateMysqlUpStatements`. ✅ Done (2026-07-01).**
  `packages/mysql/migrate/ddl.ts` (the new `@sisal/mysql/ddl` export),
  implemented to the probe-verified C4 spec and pinned by 12 network-free unit
  tests; the **generated** output was applied live to MySQL 8.4.10 and MariaDB
  11.8.8 (kitchen-sink table + FK pair + incremental `ADD COLUMN`, with firing
  defaults and enforced FK/CHECK) — closing the loop the C4 probe opened with
  hand-written DDL:
  - **The full C4 type table** (`serial`/`bigserial` → `AUTO_INCREMENT`, `real`
    → `FLOAT`, `timestamp` → `DATETIME(6)`, `timestamptz` → `TIMESTAMP(6)` with
    explicit `NULL` when nullable, `uuid` → `CHAR(36)`, `bytea` → `LONGBLOB`,
    `json`/`jsonb`/`.array()` → `JSON`, length-less `varchar` → `VARCHAR(255)`,
    `dialectType` verbatim); unknown kinds map to `TEXT`, where the TEXT rules
    below make them fail closed.
  - **The portable-defaults rule live-verified:** expression defaults always
    paren-wrap (`DEFAULT (uuid())`, `DEFAULT (CURRENT_TIMESTAMP(6))` — both
    fired on both engines), literals paren-wrap only on TEXT/BLOB/JSON-mapped
    columns (which MySQL otherwise rejects), booleans render `1`/`0`.
  - **Foreign keys table-level only, after every `CREATE TABLE`** (inline
    `REFERENCES` is silently ignored by MySQL) — the probe's child table
    deliberately sorts before its parent, proving the ordering. No
    `ENGINE`/`CHARSET` clause; `CHECK` emitted as-is (enforced live on both).
  - **Three fail-closed generation-time validations** (typed
    `ORM_DIALECT_UNSUPPORTED`): an `AUTO_INCREMENT` column must **lead** a key
    (the InnoDB rule — first column of a PK/unique/index, not mere membership),
    at most one per table, and never via `ADD COLUMN` (the additive generator
    emits no key changes); no `TEXT`/`BLOB`/`JSON`-mapped key columns (MySQL
    needs a prefix length the snapshot cannot express — the error points at
    `varchar(n)`); no partial (`WHERE`, rejected by both engines) or functional
    (MySQL-only, MariaDB rejects) indexes.
  - Same module shape as pg's generator: additive/destructive split via
    `planSchemaChangesFromDiff`, `mysql`-dialect schema objects after tables,
    schema qualifiers emitted as database qualifiers (backticked).
- **B6 — migrate wiring. ✅ Done (2026-07-01).** `packages/mysql/migrate/` now
  has the full shared shape (`driver` · `history` · `migrator` · `ddl`);
  `@sisal/mysql/migrate` is exported. Pinned by 7 network-free unit tests;
  verified live on **both engines × both drivers** (four combinations): migrate
  → idempotent re-run → rollback with B5-**generated** DDL as the migration
  bodies, ledger round-trip (`appliedAt` matched true UTC against the wall clock
  — the Date-param local-serialize/local-parse pair cancels symmetrically), and
  real two-connection lock contention (`A=true B=false` → release → `B=true`).
  - **`GET_LOCK`/`RELEASE_LOCK` named locks** — the `pg_advisory_lock` analogue
    the C4 report called for. Names pass through verbatim (no hashing, unlike
    pg's FNV-hashed bigint keys), validated non-empty and ≤ 64 chars (the MySQL
    cap) before any session is acquired. `GET_LOCK` is **connection-scoped**, so
    the store holds it on a pinned executor session (`acquireSession`, the pg
    design) — the lock dies with the connection, which is the crash-safety
    property advisory locking wants. The store coerces the lock result robustly:
    the mandated bigint-as-string driver options decode `GET_LOCK`'s BIGINT as
    the _string_ `"1"` on mysql2 (number on the mariadb connector) — both
    pinned.
  - **The history ledger obeys the adapter's own B5 rules:**
    `id varchar(255) primary key` (a `TEXT` key is invalid MySQL) and
    `applied_at datetime(6)` (no timezone conversion, no 2038 cliff);
    `appliedAt` is written as a `Date` param because an ISO string's trailing
    `Z` is rejected by `DATETIME`.
  - **`useTransaction` defaults to `false`** — unlike pg: MySQL/MariaDB have no
    transactional DDL (every `CREATE`/`ALTER` implicitly commits), so a wrap
    around schema migrations is a false promise. Opt-in remains for DML-only
    migrations, where it also makes the history mark atomic.
  - The migrate executor reuses the ORM adapter's connection-source routing, so
    `driver: "mariadb"` (and its B4 Buffer fix + decode options) applies to
    migrations for free.
- **B7 — `RETURNING` execution strategy. ✅ Done (2026-07-01).** The new
  `insertReturning(db, table, values)` helper
  (`packages/mysql/orm/returning.ts`), pinned by 8 network-free unit tests and
  verified live on **both engines × both drivers**. Strategy selection is
  automatic and rows come back in input order under both:
  - **Attempt real `INSERT … RETURNING` first** — on a facade whose detected
    identity lights it (MariaDB ≥ 10.5) this is one statement with true
    semantics. The helper catches exactly the core guard's typed error
    (`ORM_DIALECT_UNSUPPORTED` + `details.construct`) and falls back — the core
    stays the **single source of truth** for variant/version floors; no
    duplicated version logic in the adapter.
  - **Fetch-by-key fallback** in a transaction: all-explicit primary keys → one
    `INSERT` + one `SELECT` back; a single-column key with missing values →
    **one `INSERT` per row capturing each statement's own `LAST_INSERT_ID`**.
    Deliberately no first-id-plus-offset arithmetic: MySQL 8.4's live-confirmed
    default `innodb_autoinc_lock_mode = 2` (interleaved; MariaDB 11.8 ships `1`)
    does not guarantee a batch's ids are consecutive, so the arithmetic shortcut
    (Drizzle's `$returningId` approach) can silently return someone else's rows.
  - **Honest corners fail typed** (`ORM_INVALID_QUERY`), never wrong: no primary
    key, a composite key not fully provided, a `Sql`-expression key value, or a
    server-generated non-`AUTO_INCREMENT` key (`DEFAULT (uuid())` is invisible
    to `LAST_INSERT_ID`) — the live probe shows the same call returning real
    rows on MariaDB (`RETURNING` sees the generated uuid) and refusing typed on
    MySQL.
  - Plumbing: `insertId` now flows executor → driver → facade result
    (`MysqlQueryResult.insertId`; 0 normalized to absent; the mariadb
    connector's `bigint` re-viewed as a precision-safe string), and key matching
    stringifies both sides — the same id can surface as number 7
    (`LAST_INSERT_ID`), string `"7"` (bigint-as-string column decode), or
    `bigint` depending on driver and column type.
- **B8 — the integration proof. ✅ Done (2026-07-01).** The reserved `"mysql"`
  slot is filled and the fifth/sixth feature-matrix columns are live and
  test-backed. **42 shared mysql-family scenarios** run green against **both**
  engines (`integration/mysql_features_test.ts` on MySQL 8.4.10,
  `integration/mariadb_features_test.ts` on MariaDB 11.8.8), and also pass on
  the opt-in mariadb driver:
  - **One scenario list, two capability profiles.** MySQL and MariaDB are one
    adapter but distinct profiles (`IntegrationAdapterId` gained `"mariadb"`);
    `makeMysqlFamilyTarget` builds both from a shared factory that differs only
    in `returning`, `json`/`array` value shape, and env gate. Scenarios branch
    on the target's declared capabilities, so each divergence is a **tested
    fact**: MySQL's `.returning()` throws the typed guard (and the B7
    `insertReturning` fetch-by-key strategy carries the insert scenario), while
    MariaDB's auto-detected identity lights `INSERT`/`DELETE … RETURNING`;
    `UPDATE … RETURNING`, `FULL JOIN`, `distinctOn`, dm-CTEs, and
    partial/expression indexes all assert their typed guards live. B10 later
    moves `UPDATE … FROM`/`DELETE … USING` from that guard set into tested MySQL
    multi-table mutation support.
  - **Matrix and docs.** `tools/feature_matrix.ts` gained `mysql`+`mariadb`
    columns (36 features × 6 adapters = 216 cells; 198 ✅/⚠️ each backed by a
    named scenario, enforced by `docs:matrix:check`); MariaDB is its own column,
    not a footnote (`RETURNING` and JSON-as-text prove it).
    `docs/mysql-compatibility.md` is the per-engine page, and the homepage
    `#compat` section gained MySQL/MariaDB badges + link.
  - **Three real bugs the live suite caught** (invisible to the network-free
    unit tests, all fixed in core/adapter and re-pinned): JSON/array params
    reached mysql2's text protocol un-serialized (an object became invalid SQL,
    an array **expanded** into N params, silently shifting later placeholders) —
    the executor now `JSON.stringify`s plain objects/arrays; `DATETIME` columns
    decoded to client-local `Date`s (timezone-shifting server values) — the pool
    now sets `dateStrings: true`; and instants serialized with a trailing `Z`
    that MySQL rejects in datetime literals — the core renderer now tags instant
    params and rewrites them to **naive UTC** under the `mysql` render dialect
    (the "executor UTC convention"), leaving the `postgres`, `sqlite`, and
    `generic` render paths untouched.
- **B9 — CLI target + basic example. ✅ Done (2026-07-01).**
  `sisal init --target mysql` now scaffolds a `dialect: "mysql"` migration
  config with `MYSQL_URL ?? DATABASE_URL` connection hints; `mariadb` is an
  alias for the same adapter target. The CLI default adapter loader resolves
  `@sisal/mysql/ddl` and `@sisal/mysql/migrate` for `generate`, `migrate`,
  `status`, and `drift`. `examples/mysql-family-basic` joins the runnable
  workspace examples: it prints MySQL DDL without a database, and runs a tiny
  connect/create/insert/count flow against `mysql2` or the MariaDB connector via
  `SISAL_ADAPTER`.
- **B10 — multi-table mutation mapping. ✅ Done (2026-07-01).** The existing
  portable builders are unchanged, but the mysql renderer now maps
  `update(t).from(source)` to `UPDATE t, source SET t.col = … WHERE …` and
  `delete(t).using(source)` to `DELETE FROM t USING t, source WHERE …`.
  PostgreSQL/SQLite `UPDATE … FROM` rendering is unchanged; SQLite
  `DELETE … USING` remains a typed guard. Multi-table mutation plus
  `.returning()` remains guarded for the MySQL family, including MariaDB,
  because the proven B7 `RETURNING` support is per-statement/single-table. The
  shared mysql-family integration scenario now exercises `UPDATE FROM`,
  `DELETE USING`, and `INSERT SELECT`, and the matrix marks the MySQL/MariaDB
  mutation-joins cells as ✅.
  - **Follow-up (2026-07-02, CI-caught):** MariaDB parses a `WITH` prefix only
    on `SELECT`, so the CTE-prefixed forms of these mutations were a raw 1064
    there. The `WITH` prefix on `INSERT`/`UPDATE`/`DELETE` is now a
    variant-narrowed typed guard (`{ dialect: "mysql", variant: "mariadb" }`),
    the scenario branches on a `mutationCte` capability (MariaDB asserts the
    guard and runs the same mutation via a derived-table source), and the
    divergence is a row in [mysql-compatibility.md](mysql-compatibility.md).
    Mutation joins themselves stay ✅ on both engines.

## Open questions

- MySQL 8 only, or MySQL + MariaDB (affects `RETURNING`, sequences, JSON funcs)?
  **Resolved by v0.6 C5:** one adapter, **both** — MySQL ≥ 8.0.16 baseline,
  MariaDB ≥ 10.10 via the variant-aware capability descriptor; the dialect key
  is `(engine, variant, version)`, so MariaDB / older-MySQL handling stays
  **additive**.
- ~~Is `onDuplicateKeyUpdate` a MySQL-only builder method, or does `onConflict*`
  dialect-map onto it?~~ **Answered by v0.6 C2:** `onConflict*` dialect-maps (no
  separate surface), with the typed `excluded()` helper.
- ~~Does the latent `"mysql"` row-value/keyset path actually work, or need
  fixes?~~ **Answered by B8 (2026-07-01): it works.** Both keyset forms —
  expanded and row-value `(a, b) < (x, y)` — paginate identically live on MySQL
  8.4 and MariaDB 11.8 (the "keyset pagination" scenario runs both forms and
  asserts the same id order).
- ~~Where does the `TINYINT(1)` → `boolean` decode decision land (executor
  default vs opt-in), given the SQLite family's `0`/`1` precedent?~~ **Decided
  (B4, 2026-07-01): no auto-decode** — `BOOLEAN` round-trips as `0`/`1`,
  matching the SQLite family's documented value shape. A `TINYINT(1)` display
  width does not guarantee boolean semantics (users store 0–127), so
  auto-decoding would corrupt legitimate tiny integers; the matrix marks it ⚠
  like sqlite/libsql. Revisit only if a reliable signal appears.

## Acceptance criteria

- `@sisal/mysql` passes a mysql-family scenario suite covering the shared OLTP
  surface (CRUD, joins incl. the `FULL JOIN` guard, tx, prepared, `SELECT` CTEs,
  upsert via the C2 mapping, `filter()`/date helpers once B2 lifts their guards,
  migrations) against `mysql:8.4` **and** `mariadb:11`, gated by
  `SISAL_MYSQL_IT=1`.
- The feature matrix gains **test-backed MySQL (and, where they diverge,
  MariaDB) columns**; divergences are `⚠️` with documented round-trip notes,
  genuine limits are `❌`. (MariaDB is a distinct capability profile, not a
  footnote on one "fifth column" — `RETURNING` alone proves it.)
- Upsert and `RETURNING` behave correctly or fail/emulate as documented.
- The published dialect identity carries a **variant/version tag** (`mysql` vs
  `mariadb`, server version); any capability that differs by variant/version
  (`RETURNING`, `WITH RECURSIVE`, `JSON_TABLE`) is gated on it and fails
  **typed**, not as a raw engine error. This is the adapter-level seed of the
  v0.8 capability descriptor.
- `@sisal/orm`/core render output for the other four adapters (pg, neon, sqlite,
  libsql) is unchanged.

## Dialect compatibility notes

MySQL 8 supports CTEs, window functions, and row locking (much of the surface is
portable — the v0.6 C5 probe verified windows/CTEs/`INTERSECT`/`EXCEPT` on both
engines). The upsert divergence is **already mapped** (C2); the remaining
principled divergences are **uneven `RETURNING`** (guard today, B7 strategy),
**`BOOLEAN`=`TINYINT(1)`**, **JSON decoding** (parsed on MySQL, text on
MariaDB), and **MySQL-vs-MariaDB** (`LATERAL`/`->>`/functional indexes
MySQL-only; `SEQUENCE`/native `UUID`/extended `TIMESTAMP` MariaDB-only — carried
by the B1 capability descriptor). `distinctOn`, the Postgres array operators,
`FULL JOIN`, and data-modifying CTEs stay `❌` (guarded). This column feeds the
v0.9 hardening and the ETL/analytics dialect matrices.
