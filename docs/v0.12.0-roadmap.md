# Sisal v0.12.0 roadmap — `@sisal/dashboard` Preview

v0.12 introduces a **generic dashboard mapping layer** that turns analytics
result sets into **semantic presentation models**. It is the last layer of the
stack and the most easily misunderstood, so the boundary is stated up front:

> **Analytics computes. Dashboard maps. Renderer renders.**

`@sisal/dashboard` **does not render charts** and **does not depend on** D3,
React, Recharts, ECharts, Vega, or any UI library. It produces renderer-agnostic
models that a presentation layer consumes.

## Goal

Map an [analytics](v0.11.0-roadmap.md) result set into a typed, semantic
presentation model (with the metadata a renderer needs: labels, units, formats,
roles, series identity, trend) — leaving the actual drawing to whatever renderer
the app chooses.

## Package boundary

`@sisal/dashboard` depends on `@sisal/core` and on `@sisal/analytics` **types**.
**`@sisal/orm` must not depend on it**, and it must not depend on any renderer
or UI framework. (See [architecture](architecture.md).)

## Scope — initial model helpers

Each takes an analytics result + a small config and returns a semantic model:

- `kpi()` — a single headline metric (+ optional comparison/trend).
- `timeSeries()` — one or more series over a time dimension.
- `bar()` — categorical comparison.
- `leaderboard()` — ranked rows (top-N).
- `table()` — tabular projection with typed columns.
- `heatmap()` — two-dimensional intensity grid.
- `funnel()` — ordered stage conversion.
- `histogram()` — bucketed distribution.

## Semantic metadata each model carries

- field **name** and human **label**
- value **type**, **unit**, and **format** (e.g. `percent`, `currency`, `count`)
- **role** (dimension / measure / time / category / series)
- **series identity** (for multi-series charts)
- **comparison / trend** metadata where relevant (delta, direction, baseline)

## The contract

```
analytics result  →  dashboard.timeSeries(...)  →  D3 / Recharts / ECharts / custom
   (computes)             (maps to a model)              (renders)
```

A renderer adapter (in app code, not in this package) reads the semantic model
and draws it. The same model can drive any renderer.

## Non-goals

No chart rendering; no DOM/canvas/SVG; no dependency on D3/React/Recharts/
ECharts/Vega or any UI layer; no styling/theming; no data fetching (it maps a
result it's handed).

## Risks

- **Renderer leakage** — accidentally encoding one library's chart spec.
  Mitigated by keeping models renderer-agnostic and proving ≥2 renderer adapters
  in docs.
- **Over-abstraction** — too many model types too early. Start with the eight
  above, driven by real analytics outputs.
- **Metadata gaps** — a model missing the unit/format a renderer needs. Pin the
  metadata contract with tests.

## Open questions

- Is there a single `Model` union, or per-helper model types? (Leaning:
  discriminated union with shared metadata.)
- How are formats expressed — a format token enum, or an Intl-style descriptor?
- Does dashboard own number/date formatting, or only _describe_ it for the
  renderer? (Leaning: describe, don't format.)

## Acceptance criteria

- Each helper maps a real `@sisal/analytics` result into a typed semantic model
  with complete metadata (label/type/unit/format/role/series/trend as
  applicable).
- The package has **zero** runtime dependency on any renderer or UI framework
  (enforced by a dependency test).
- Docs show the **same** `timeSeries()` model consumed by **two** different
  renderers (e.g. a Recharts adapter and a custom one), proving agnosticism.

## Affected packages

New `@sisal/dashboard`; depends on `@sisal/core` + `@sisal/analytics` types
only. No `@sisal/orm` dependency; no UI dependency.

## Required examples / tests

- For each helper: an analytics-result → model mapping test asserting the
  semantic metadata.
- A docs example: one analytics time series → `dashboard.timeSeries()` → two
  renderer adapters.
- A dependency-direction test (no UI/renderer imports).

## Dialect compatibility notes

Dashboard is dialect-agnostic — it maps result sets, not SQL. Any dialect
limitation is inherited from the analytics query that produced the result and is
out of scope here.
