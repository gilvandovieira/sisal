# Sisal v0.14+ roadmap — Optional Native/Rust Acceleration Investigation

The last milestone on the line, and the most conditional. v0.14+ investigates
native/Rust acceleration **only if** benchmarks prove a real bottleneck that
**neither SQL pushdown nor DuckDB** ([v0.13+](v0.13.0-roadmap.md)) can solve
cleanly. There is no default assumption that Sisal ever ships native code.

## Goal

Decide, on **benchmark evidence**, whether a specific hot path justifies native
(Rust) acceleration — and if so, scope the smallest possible native surface.

## The gate (non-negotiable)

> **Do not introduce Rust/native complexity without a benchmark-backed reason.**

Native code is considered **only after** SQL pushdown (Postgres) and DuckDB have
been measured and shown insufficient for a _specific, reproduced_ bottleneck.
The first deliverable of this milestone is the benchmark, not the binary.

## Scope — potential areas (each must be benchmark-justified)

- **Large local Parquet processing**
- **Arrow / vectorized transformations**
- **Compression / decompression hot paths**
- **Very large CSV / JSON ingestion**
- **Custom native aggregators**
- **High-throughput event ingestion**

## Non-goals

No native code without a reproduced, benchmarked bottleneck; no rewrite of
existing TS/SQL paths that already perform adequately; no native dependency that
breaks Deno/Node portability (the v0.6 runtime reach) without an explicit,
documented tradeoff; no general "rewrite it in Rust" ambition.

## Risks

- **Complexity & portability cost** — native bindings threaten the Deno-first /
  Node-reach story, cross-platform builds, and serverless deploy. This must be
  weighed explicitly per area.
- **Maintenance burden** — a Rust surface is a long-term cost; only worth it for
  a durable, proven bottleneck.
- **Premature adoption** — the central risk the gate exists to prevent.

## Open questions

- Which bottleneck (if any) survives the v0.13+ DuckDB investigation? DuckDB may
  already cover Parquet/Arrow/vectorized work, removing the reason for Rust.
- If native is justified, what's the integration surface — Deno FFI, a Wasm
  module, a sidecar process?
- How is the native path kept optional so the pure-TS install always works?

## Acceptance criteria

- A **benchmark report** for any candidate area showing SQL pushdown **and**
  DuckDB are insufficient for a reproduced workload — or a conclusion that
  native acceleration is **not** justified (an equally valid outcome).
- If pursued: a narrowly-scoped native prototype behind a flag, with the pure-TS
  path remaining the default and the portability tradeoff documented.

## Affected packages

Investigation only; potentially a future optional native helper consumed by
`@sisal/etl`/`@sisal/analytics`. No required changes to the core stack.

## Required examples / tests

- A reproducible benchmark harness for each candidate hot path.
- If pursued: parity tests proving the native path matches the TS/SQL result.

## Dialect compatibility notes

Out of scope — native acceleration concerns local data processing
(Parquet/Arrow/ ingestion), not the SQL dialects. Any native path must not
regress the portability guarantees established in v0.6 (Deno + Node) or the
established multi-adapter SQL support, including the MySQL/MariaDB capability
profiles.
