# Sisal v0.4.0 roadmap

This document captures the API gaps surfaced while building the
[`examples/neon-hot-feed`](../examples/neon-hot-feed/README.md) example — a
Reddit-style hot feed on Neon/PostgreSQL that must stay Deno Deploy / Neon-HTTP
friendly. Each gap forced the example into a raw-SQL escape hatch. None of them
are bugs; they are missing conveniences that, if added, would let real
applications stay inside Sisal's typed surface instead of dropping to strings.

Treat this as the working backlog for the `0.4.0` cycle. Every item lists the
concrete workaround the example uses today (with file references), a proposed
API, the affected packages, and acceptance criteria. Where an item maps onto the
Drizzle surface, the [drizzle-parity](drizzle-parity.md) row to move is noted so
the two documents stay in step.

## 1. Typed database-function caller — P1 — 🟢 Done

**Shipped.** `defineFunction(name, { args, returns })` declares a function's
positional argument column types and its return shape (a `RETURNS TABLE (...)`
column map → typed row, or a single column builder → typed scalar).
`db.call(fn, args)` renders one `select * from <schema.fn>($1::t1, …)` statement
with the `::type` casts taken from the argument column types and every value
bound, then runs it with `.execute()` (rows) or `.one()` (asserts exactly one).
A scalar return renders `select fn(...) as "result"` and unwraps the value. Pure
`@sisal/orm`, no adapter changes. Exports `defineFunction`,
`FunctionDefinition`, `FunctionCall`, `FunctionConfig`, `FunctionArgsConfig`,
`FunctionReturnsConfig`, `FunctionArgsInput`, `FunctionRow`. The example's
`src/vote.ts` now uses it with no `sql` string; covered by
`packages/orm/functions_test.ts` (render + param order, casts, scalar/table,
`.one()`, validation).

**Problem.** The whole point of the example is an atomic vote via the PostgreSQL
function `app.vote_post(post_id, user_id, value)`. There was no Sisal surface
for calling a stored function and mapping its `RETURNS TABLE (...)` to a typed
row, so the call was a hand-written `sql` template and the result type was
asserted by hand.

**Where it surfaced.**

- `examples/neon-hot-feed/src/vote.ts` —
  `` sql`select * from app.vote_post(${postId}::uuid, ${userId}::uuid, ${value}::smallint)` ``
  with a manually declared `VoteResult` interface and `::uuid` / `::smallint`
  casts written into the string.

**Current workaround.** Raw `sql` tag + `db.query<VoteResult>(...)`, plus a
manual "no row returned" guard. Correct and parameterized, but stringly typed
and easy to drift from the SQL signature.

**Proposed API.** A function descriptor + caller that renders a single
`SELECT * FROM fn(args)` statement and infers argument and row types:

```ts
import { columns, defineFunction } from "@sisal/orm";

const votePost = defineFunction("app.vote_post", {
  args: {
    postId: columns.uuid(),
    userId: columns.uuid(),
    value: columns.smallint(),
  },
  returns: {
    id: columns.uuid(),
    score: columns.integer(),
    upvotes: columns.integer(),
    downvotes: columns.integer(),
    hot_score: columns.doublePrecision(),
  },
});

// args are positional-by-declared-order, cast from the arg column types;
// the result is typed from `returns`. Returns rows; a `.one()` helper can
// assert exactly one.
const [row] = await db.call(votePost, {
  postId,
  userId,
  value: -1,
}).execute();
```

The caller must emit casts from the declared arg column types (so `::uuid` /
`::smallint` come from the descriptor, not the string) and bind every argument
as a parameter.

**Affected packages.** `@sisal/orm` (descriptor, renderer, `db.call`). No
adapter changes — it renders to ordinary parameterized SQL.

**Acceptance criteria.**

- `defineFunction` infers an args object type and a row type.
- `db.call(fn, args)` renders `select * from <schema.fn>($1::t1, $2::t2, …)`
  with the casts taken from the arg column types and all values bound.
- A scalar-returning variant (`returns: columns.integer()`) yields a typed
  scalar helper.
- The example's `src/vote.ts` is rewritten to use it with no `sql` string.
- Network-free render test pins the generated SQL and parameter order.

**Open questions.** Named vs positional arguments; `VOID`/`SETOF`/`RECORD`
return shapes; OUT parameters. Start with positional args + `RETURNS TABLE`.

## 2. Serverless-safe SQL migration applier + Neon CLI target — P1 — 🟢 Done

**Shipped.** `splitSqlStatements(sql)` is now exported from `@sisal/migrate` (a
dollar-quote/`$tag$`/string/quoted-identifier/comment-aware splitter, hardened
to drop empty/stray semicolons), covered by
`packages/migrate/sql_split_test.ts`; the CLI and the `neon-hot-feed` example
both use it (the example dropped its local copy). The migrator gained a
`splitStatements` apply mode (core `createMigrator`, forwarded through
`createPgMigrator` and `createNeonMigrator`): a string `up`/`down` step is split
and run one statement per `driver.execute(...)` call, still recording history —
`createNeonMigrator` defaults it to `true` (and `useTransaction` to `false`) for
the HTTP transport. The `sisal` CLI gained a **Neon target**:
`sisal init --neon` scaffolds a `dialect: "postgres"` + `provider: "neon"`
config, and `sisal migrate` then applies through `@sisal/neon` over HTTP (one
`INIT_TARGETS` entry + a `provider` discriminator on `MigrateConfig`). Covered
by `packages/migrate/{sql_split,split_apply,cli}_test.ts`.

**Problem.** A `.sql` migration file holds several statements separated by `;`,
but the Neon serverless driver sends each query through PostgreSQL's extended
protocol, which allows exactly one statement per call. Splitting on `;` also
breaks `CREATE FUNCTION ... $$ ... ; ... $$` bodies. Sisal has no
serverless-safe applier, and the `sisal` CLI's migrate path targets the
`postgres` / `sqlite` adapters, not a Neon-HTTP applier.

**Where it surfaced.**

- `examples/neon-hot-feed/src/sql_split.ts` — a hand-rolled, dollar-quote-aware
  statement splitter (handles `$$`/`$tag$`, single-quoted strings, and line /
  block comments).
- `examples/neon-hot-feed/src/migrate.ts` — reads each file, splits it, and
  executes statements one at a time via `db.execute(raw(stmt))`.

**Current workaround.** The two files above, plus idempotent DDL
(`IF NOT EXISTS` / `CREATE OR REPLACE`) so re-runs are safe. There is no
applied- migration history table in the example.

**Proposed API.**

1. A reusable splitter in `@sisal/migrate` (e.g. `splitSqlStatements(sql)`),
   dollar-quote/string/comment aware, with tests.
2. A migration **apply mode** that runs each `.sql` file statement-by-statement
   for drivers that cannot take multi-statement queries (a per-statement vs
   per-file execution flag on the migrator), preserving the history table.
3. A Neon target for the `sisal` CLI so `sisal migrate` works against
   `@sisal/neon` (HTTP) the same way it does for `@sisal/pg`. Adding it should
   be one `INIT_TARGETS` entry plus the adapter wiring, per the CLI's existing
   "one registry entry per database" design.

**Affected packages.** `@sisal/migrate` (splitter + apply mode), `@sisal/neon`
(applier wiring), `packages/migrate/cli.ts` (Neon target + `CliDialect`).

**Acceptance criteria.**

- `splitSqlStatements` is exported from `@sisal/migrate`, covers `$$`/`$tag$`,
  string literals, and comments, and is unit tested (port the example's tests).
- The migrator can apply a multi-statement `.sql` file over a single-statement
  driver without breaking function bodies, still recording history.
- `sisal init --neon` (or equivalent) scaffolds a Neon config and
  `sisal migrate` applies migrations through `@sisal/neon`.
- The example switches to the shared splitter / applier and drops its local copy
  (or keeps it only as a teaching reference, with a pointer here).

**Notes.** `@sisal/pg` over a direct TCP connection already tolerates
multi-statement files (deno-postgres simple protocol). This item is specifically
about the **serverless/HTTP** path that the example optimizes for.

## 3. Keyset / cursor pagination helper — P1 — 🟢 Done

**Shipped.** `SelectBuilder.keyset({ orderBy, after, form? })` infers the cursor
type from the `orderBy` columns, emits the keyset comparison against `after`
(omit it for the first page) plus the matching `ORDER BY`, and returns a
`KeysetSelectBuilder` whose `.limit(n).execute()` yields `{ rows, nextCursor }`
(a `nextCursor` only when a full page came back). Two predicate forms: the
default `"expanded"` nested `or`/`and` (works with mixed directions and every
dialect) and `"row-value"` (`(a, b, c) < (x, y, z)`, a uniform sort direction).
To enable this, `asc()`/`desc()` now return an `OrderTerm` (a `Sql` subtype that
also carries the column + direction; backward compatible), and a column's
`propertyName` is a literal type so the cursor keys infer. Precision: ending
`orderBy` with a unique key (e.g. `id`) makes the order total; for date/time
cursors, use DB-returned cursor values because Temporal can represent
nanoseconds, PostgreSQL stores microseconds, and JS `Date` stores milliseconds.
Exports `OrderTerm`, `KeysetOptions`, `KeysetCursor`, `KeysetKeys`,
`KeysetPage`, `KeysetSelectBuilder`. The example's `getNewFeed` (expanded) and
`getHotFeed` (row-value) are both builder-native; covered by
`packages/orm/keyset_test.ts`. Adds the divergence-by-design row (footnote ⁸) to
[drizzle-parity](drizzle-parity.md).

**Problem.** Every feed re-implemented the keyset comparison
`(a, b, c) < (x, y, z)` by hand. The `/new` feed could do it with nested
`or`/`and`, but the `/hot` feed (a three-column keyset over a computed
`double precision` column) was clearer as raw SQL — so it dropped out of the
typed builder.

**Where it surfaced.**

- `examples/neon-hot-feed/src/queries.ts` — `getNewFeed` hand-builds
  `or(lt(created_at, c), and(eq(created_at, c), lt(id, c)))`; `getHotFeed` uses
  a raw `sql` template for the three-column keyset and matching `ORDER BY`.

**Current workaround.** Two bespoke implementations and two cursor shapes
(`NewCursor`, `HotCursor`) defined by hand.

**Proposed API.** A helper that takes an ordering spec and an "after" cursor and
emits both the `WHERE` keyset predicate and the matching `ORDER BY`:

```ts
const page = await db.select().from(posts)
  .where(eq(posts.columns.status, "published"))
  .keyset({
    orderBy: [
      desc(posts.columns.hot_score),
      desc(posts.columns.created_at),
      desc(posts.columns.id),
    ],
    after: cursor, // typed from the orderBy columns
  })
  .limit(20)
  .execute();
// page exposes rows + a `nextCursor` derived from the last row.
```

It should be able to emit either the expanded nested `or`/`and` form or a SQL
row-value comparison `(a, b, c) < (x, y, z)` where the dialect supports it.

**Timestamp-precision caveat.** Temporal can represent nanoseconds, PostgreSQL
timestamps store microseconds, and JS `Date` stores milliseconds. A keyset over
a timestamp tiebreaker can skip or duplicate rows at a page boundary if the
cursor value is re-created at a different precision. Prefer DB-returned cursor
values and always include a unique final key such as `id`.

**Affected packages.** `@sisal/orm` (builder method + cursor inference).

**Acceptance criteria.**

- A `keyset({ orderBy, after })` builder method infers the cursor type from the
  `orderBy` columns and emits the predicate + ordering.
- `nextCursor` is produced from the last row of a full page.
- Render tests pin both the nested-`or` and row-value output forms.
- Documented precision behavior for timestamp tiebreakers.
- The example's `getNewFeed` and `getHotFeed` both become builder-native.

**Drizzle reference.** Drizzle has no first-class keyset helper either; this is
a chance for Sisal to lead rather than match. Add a roadmap row to
[drizzle-parity](drizzle-parity.md) noting the divergence-by-design.

## 4. Raw expressions in builder `SET` / `VALUES` / `DEFAULT` — P2 — 🟢 Done

**Shipped.** `.set({...})`, `.values({...})`, and `onConflictDoUpdate.set`
accept a `Sql` expression for any column value alongside literals:

```ts
set({ score: sql`${posts.columns.upvotes} - ${posts.columns.downvotes}` });
values({ id, created_at: sql`now()` });
```

Expressions render **inline** (a column reference renders table-qualified; raw
text passes through) while literal values still bind as parameters. The public
types `InsertValues` / `UpdateValues` widen each column value to `value | Sql`;
`InferSelect` / `InferInsert` are unchanged. Covered by
`packages/orm/set_values_expr_test.ts` (UPDATE, INSERT, and `onConflict` mixing
literals and expressions; param binding verified). Scope is **scalar**
expressions: `UPDATE ... FROM` and `INSERT ... SELECT` remain out of scope, so
the example's `recomputeAggregates` (an `UPDATE ... FROM (subquery)`) stays raw.
`DEFAULT` expressions are a DDL concern tracked under item 5. Moves the parity
row.

**Problem.** The builder could only set/insert bound values, not SQL
expressions. You cannot write
`set hot_score = app.calculate_hot_score(score, created_at)` or insert a column
whose value is a SQL function call, so any computed mutation drops to raw SQL.

**Where it surfaced.**

- `examples/neon-hot-feed/src/seed.ts` → `recomputeAggregates` — a raw
  `UPDATE ... FROM (... LEFT JOIN ...)` whose `SET` clause calls the hot-score
  function, because the builder's `.set({...})` only accepts literal values.

**Current workaround.** Raw `db.execute(raw(\`update ...\`))`.

**Proposed API.** Allow `sql` expressions as values in `.set(...)` and
`.values(...)` (and as column defaults):

```ts
await db.update(posts)
  .set({
    score: sql`${posts.columns.upvotes} - ${posts.columns.downvotes}`,
    hot_score: sql`app.calculate_hot_score(score, created_at)`,
  })
  .where(eq(posts.columns.id, id))
  .execute();
```

`UPDATE ... FROM` / `INSERT ... SELECT` are larger asks; start with scalar `sql`
expressions in `SET`/`VALUES`, which covers the common "compute from other
columns" case.

**Affected packages.** `@sisal/orm` (builders type widening + renderer).

**Acceptance criteria.**

- `.set({...})` and `.values({...})` accept `Sql` expressions alongside literal
  values, with types updated accordingly.
- Expressions render inline and parameterized; literal values still bind.
- Render tests for an UPDATE and an INSERT mixing literals and expressions.

**Drizzle reference.** Drizzle allows `sql` in `.set()` / `.values()`; this is a
parity item. Move the relevant row in [drizzle-parity](drizzle-parity.md).

## 5. Richer DDL generation — P2 — 🟢 Done (functions/triggers → v0.5.0)

**Shipped (5.1 indexes + 5.2 CHECK).** Table-level `index()` / `uniqueIndex()`
now carry per-column direction via `asc()`/`desc()` terms, raw `` sql`...` ``
**expression** keys, and a partial-index predicate via a new `.where(predicate)`
method:

```ts
defineTable("posts", cols, (t) => [
  index("hot_feed")
    .where(sql`${t.status} = 'published'`)
    .on(desc(t.hotScore), desc(t.createdAt), desc(t.id)),
  uniqueIndex().on(sql`lower(${t.email})`),
]);
```

`generate{Postgres,Sqlite,Libsql}UpStatements` emit the `col DESC`/`ASC`
ordering, parenthesized expression keys, and a trailing `WHERE`; expression and
predicate text renders portably (table prefix stripped) exactly like CHECK
constraints, which already emitted inline. The serialized index shape is now
structured (`{ value, direction?, expression? }` + optional `where`), so
`SCHEMA_SNAPSHOT_VERSION` is **2** (regenerate snapshots). Adds the public
`IndexColumnSpec` type and the `SisalIndexColumnSnapshot` descriptor. Covered by
the pg/sqlite Drizzle-parity tests and the gated **pg 16/17/18** and **SQLite**
and **libSQL** integration suites
(`rich indexes (DESC / partial / expression) apply`), and the parity index row
moved. **Deferred:** 5.3 functions/triggers (raw `CREATE FUNCTION` / `TRIGGER`
DDL attached to a snapshot) is a larger design moved to the
[v0.5.0 roadmap](v0.5.0-roadmap.md) as its own item.

**Problem.** `generatePostgres/Sqlite/LibsqlUpStatements` emit only additive
`CREATE TABLE` / `ADD COLUMN`. They cannot express DESC index ordering, partial
or expression indexes, CHECK constraints, or functions/triggers. So the example
keeps hand-written `.sql` migrations as the source of truth and treats
`src/schema.ts` as a typed _mirror_ for the builder, not the generator's output.

**Where it surfaced.**

- `examples/neon-hot-feed/migrations/0001_init.sql` — `posts_new_feed_idx` /
  `posts_hot_feed_idx` use `... desc` column ordering and there is a
  `value in (-1, 1)` CHECK; none of this is producible from the snapshot.
- `examples/neon-hot-feed/src/schema.ts` — declares `index(...).on(...)`,
  `primaryKey`, and `check(...)` purely for typing; the DDL is written by hand.

**Current workaround.** Hand-written SQL migrations; the `defineTable` `extras`
metadata is informational only.

**Proposed API / scope (incremental).**

1. **Index ordering + uniqueness/partiality**: let `index()` / `uniqueIndex()`
   carry per-column direction (`asc`/`desc`), a `where` predicate (partial), and
   expression columns, and have the generator emit them.
2. **CHECK constraints**: emit `check(name, sql\`...\`)`into`CREATE TABLE`.
3. **Functions/triggers** (stretch): a way to attach raw DDL fragments to a
   snapshot so generated migrations can include `CREATE FUNCTION` bodies
   verbatim, ordered after table creation.

**Affected packages.** `@sisal/orm` (`table.ts` constraint model +
`createSchemaSnapshot`), each adapter's DDL generator
(`packages/{pg,sqlite,
libsql}/ddl.ts`), and `@sisal/migrate` diff
classification.

**Acceptance criteria.**

- DESC/partial/expression indexes and CHECK constraints declared via
  `defineTable` appear in generated `CREATE` DDL for each dialect.
- Destructive-change detection still holds (new constraints are additive; drops
  go to the `destructive` array).
- The example can regenerate `0001_init.sql` from `src/schema.ts` (functions may
  remain hand-written until item 5.3 lands).
- Move the corresponding [drizzle-parity](drizzle-parity.md) rows.

## 6. Non-interactive (batched) transaction API — P2 — 🟢 Done

**Shipped.** `db.batch([...])` runs several pre-built statements (query
builders, `` sql`...` `` fragments, or rendered `SqlQuery`) as one atomic,
**non-interactive** unit and returns one `OrmQueryResult` per statement. It
renders every statement up front (an unbound placeholder throws a clear
`OrmError` before touching the database), then prefers a driver's native `batch`
and otherwise wraps the statements in one `transaction` (`begin; …; commit`),
rolling the whole batch back on any failure. A `batch?` hook was added to the
`OrmDriver` contract and implemented in **every adapter** (`@sisal/pg` — and
therefore `@sisal/neon`, which builds on `createPgOrmDriver` — plus
`@sisal/sqlite` and `@sisal/libsql`), each running the batch in one atomic
transaction. Exports `BatchStatement`. Covered by `packages/orm/batch_test.ts`
(native-batch path, transaction fallback, rollback, unbound-placeholder, mixed
inputs, empty) and the gated `pg`/`sqlite`/`libsql` integration suites. No
statement may depend on a previous one's result — that is what `transaction()`
and database functions are for.

**Problem.** `db.transaction(async (tx) => { ... })` is interactive: it holds a
connection open across every awaited statement in the callback. On Deno Deploy +
Neon HTTP that is the awkward shape the example is built to avoid. There is no
way to express "run these N statements as one atomic, non-interactive
transaction in a single round trip."

**Where it surfaced.** The README's
[Important Neon note](../examples/neon-hot-feed/README.md#important-neon-note)
contrasts interactive transactions, non-interactive transactions,
single-statement mutations, and database functions. The example reaches for a
database function precisely because the middle option does not exist in Sisal.

**Current workaround.** Push multi-step logic into a database function (item 1)
or a single data-modifying statement, sidestepping interactive transactions.

**Proposed API.** A batch that collects pre-built statements and submits them as
one `begin; …; commit` unit (one round trip where the driver supports it):

```ts
await db.batch([
  db.insert(posts).values(p),
  db.update(posts).set({ score }).where(eq(posts.columns.id, id)),
]); // atomic, non-interactive
```

No values may depend on a previous statement's result (that is what interactive
transactions and database functions are for); the batch is for independent
writes that must commit together.

**Affected packages.** `@sisal/orm` (`db.batch`, driver contract addition),
adapters (`@sisal/neon`, `@sisal/pg`, …) implement a batched-execute path; the
HTTP path can use the driver's array/transaction mode.

**Acceptance criteria.**

- `db.batch([...])` renders and submits multiple statements atomically, ideally
  in one round trip on Neon HTTP.
- Failure rolls the whole batch back.
- Clear error if a builder in the batch references an unbound placeholder.
- Documented in the example README's transaction table as the "non-interactive
  transaction" row, with a runnable snippet.

## 7. Column-name mapping (camelCase ↔ snake_case) — P3 — 🟢 Done

**Shipped.** `defineTable` now derives physical column names through a naming
strategy that **defaults to `snake_case`** (`hotScore` → `hot_score`, idempotent
on already-snake_case keys). Override per table with the `naming` option
(`"snake_case"` | `"camelCase"` | `"preserve"` | a `(key) => name` function),
process-wide with `setDefaultColumnNaming(strategy)`, or per column with
`.named(...)` (always wins). The builder maps key↔physical on both write
(INSERT/UPDATE/SET/ON CONFLICT) and read (`SELECT *`/`RETURNING *` expand to
`phys AS key` once a table has any renamed column), so code that stays inside
the builder is transparent to the strategy. `InferSelect`/`InferInsert` and
`db.query.<table>` already keyed by the JS property name, so they were
unaffected. Covered by `packages/orm/naming_test.ts` and the
`parity: column casing` test; parity/legacy tables that assert literal SQL opt
out with `naming: "preserve"`. See the [drizzle-parity](drizzle-parity.md)
casing row (footnote ⁷) for the divergence-by-design note.

**Problem.** Sisal used the `defineTable` property key as the physical column
name verbatim. To match the SQL (`hot_score`, `created_at`, `post_id`), the
example's models used snake_case keys, so application code read
`posts.columns.hot_score` instead of idiomatic `hotScore`.

**Where it surfaced.**

- `examples/neon-hot-feed/src/schema.ts` — every column key is snake_case to
  line up with the hand-written DDL and the `app.vote_post` return columns.

**Current workaround.** Snake_case keys throughout TS.

**Proposed API.** Optional explicit column names and/or a table-level naming
strategy, so TS can stay camelCase while SQL stays snake_case:

```ts
const posts = defineTable("posts", {
  hotScore: columns.doublePrecision("hot_score"), // explicit name, or…
  createdAt: columns.timestamp({ withTimezone: true }),
}, { naming: "snake_case" }); // …a table strategy
```

The renderer already resolves `column.definition.name ?? propertyName`, so an
explicit-name path is a small change; a naming strategy is the larger, ergonomic
win. Result-row mapping (DB `hot_score` → JS `hotScore`) must be handled on read
for the strategy to be transparent.

**Affected packages.** `@sisal/orm` (column factories accept a name;
`defineTable` naming strategy; result mapping), with knock-on effects for
inference and `db.query.<table>` relational helpers.

**Acceptance criteria.**

- A column can declare an explicit physical name distinct from its key.
- An opt-in table/global naming strategy maps camelCase keys to snake_case
  columns for both rendering and result mapping.
- Inference (`InferSelect` / `InferInsert`) reflects the JS-side key names.
- The example can switch to camelCase keys with no SQL changes.

**Drizzle reference.** Drizzle supports explicit column names
(`integer("snake_case")`) and a `casing` option. This is a parity item; add/move
the row in [drizzle-parity](drizzle-parity.md).

## 8. Temporal date/time semantics — P3 — 🟢 Done

**Shipped.** Sisal v0.4.0 treats Temporal as the preferred date/time model.
`columns.date()` infers `Temporal.PlainDate`, `columns.time()` infers
`Temporal.PlainTime`, `columns.timestamp()` infers `Temporal.PlainDateTime`, and
`columns.timestamp({ withTimezone: true })` infers `Temporal.Instant`. Explicit
`mode: "date"` and `mode: "string"` keep legacy JS `Date` or raw string
behavior. SQL parameters normalize Temporal values to ISO strings before
adapters see them, including arrays.

PostgreSQL DDL now maps `timestamp` to `timestamp` and `timestamptz` to
`timestamptz`; existing instant columns should use
`columns.timestamp({ withTimezone: true })`. SQLite/libSQL keep date-like values
as `TEXT`. Result parsing remains opt-in through
`createDatabase({ temporal: { parse: true } })` and only applies to ORM-built
queries carrying column metadata, so raw SQL rows are left untouched.

**Acceptance criteria met.**

- Semantic SQL-to-Temporal column inference and explicit legacy modes.
- Temporal parameter serialization in ORM SQL rendering and direct adapter
  executors.
