# Sisal v0.10.0 roadmap — `@sisal/etl` Preview

The investigations ([v0.6](v0.6.0-roadmap.md)) and substrate
([v0.8](v0.8.0-roadmap.md), [v0.9](v0.9.0-roadmap.md)) are done; v0.10 ships the
**first preview of Sisal ETL** as a separate package. The model is deliberately
small: a **job definition + a single-run runner** that pushes heavy work down
into the database. It is **not** a scheduler or worker platform.

**P1 spine (the acceptance-criteria critical path).** T11→T16 build the package,
job model, rollup SQL, checkpoint, window math, and the single-window runner —
the minimum that makes `defineJob` + `run` execute a pushed-down rollup. T17
(backfill), T18 (idempotent replay), T20 (dry-run verification), T21
(capability-gating), and T23 (the Postgres example + gated integration suite)
each discharge a **named acceptance criterion**. T19 (status) and T22 (scheduler
docs) are required for the release but sit off the correctness spine. Dependency
order: **T11 → T12 → T13 → T14 → T15 → T16**, then T17/T18/T19/T20 fan out from
the runner, with T21 and T23 closing the release.

- **T11 — package scaffold.** New `@sisal/etl` `deno.json` with narrow subpath
  exports; depends on `@sisal/core` **only**, executes through an injectable
  adapter executor (so unit tests stay network/FFI-free like the rest of the
  workspace). A boundary lint/test pins that `@sisal/orm` never imports
  `@sisal/etl` and `@sisal/etl` never deep-imports an adapter.
- **T12 — `defineJob()`.** The typed job declaration — source table, target
  rollup table, grain (bucket width), window/watermark column, and the transform
  (group keys + filtered aggregates). Types flow from the v0.8 IR and the schema
  snapshot so the target columns and aggregates are checked at definition.
- **T13 — `rollup()` generated SQL.** The extract→aggregate→upsert shape as one
  (or few) pushed-down statements: insert-from-select + date-bucket +
  `FILTER`/`CASE WHEN` aggregates + upsert-from-select keyed on the grain. This
  is the v0.6 PoC promoted to the v0.8 IR. Postgres-first; no row round-trips.
- **T14 — checkpoint store.** Read/advance/prune against the **already-decided**
  `sisal_etl_checkpoints` contract (`job`, `window_end`, `pruned_before`,
  `updated_at`), consuming the v0.9 T13 validation and the v0.9 T14
  prune/`assertReplayable` helpers. v0.10 only consumes ownership; it does not
  re-decide it.
- **T15 — window computation.** Compute the next half-open window
  `[from, until)` from the checkpoint watermark + grain, holding the `<= now`
  and bucket-edge invariants the risk register calls out; this is where
  off-by-one double-counting is prevented.
- **T16 — `run(job)`.** The whole single-window runtime: read job → read
  checkpoint → compute window → **acquire the v0.9 portable lock** → send the
  T13 SQL → **advance the checkpoint atomically with the load** (v0.9 T12) →
  exit. Resolves the open question of whether `run()` owns the transaction
  boundary or composes with the caller's, and the multi-statement shape
  (`db.batch` sequence vs one CTE).
- **T17 — `backfill(job, range)`.** Deterministically re-run an explicit
  historical range as successive windows — same generated SQL, same idempotent
  upsert, no dependence on wall-clock "now."
- **T18 — `replay(job, window)`.** Idempotently re-run a single window; refuse
  windows that begin before the job's `pruned_before` horizon with the v0.9
  `ORM_REPLAY_PRUNED` typed error, with the explicit unsafe override that
  acknowledges re-derivation from a restored source.
- **T19 — `status(job)`.** Report the checkpoint position, retention horizon,
  and last-run state — the read-only observability surface.
- **T20 — dry-run / explain.** Emit the exact generated SQL without executing,
  and pin it with a golden-SQL snapshot per dialect. This is also how the
  acceptance criterion "generated, pushed-down SQL" is verified.
- **T21 — capability-gating.** Postgres is first-class (full pushdown +
  `pg_advisory_lock`). SQLite/libSQL run the rollups they support with the
  lock-row strategy and `FILTER`→`CASE WHEN`; Neon runs via the WebSocket Pool
  favoring one round trip; MySQL is gated by SEC-008 until T1. Unsupported job
  shapes return a typed error — **never a silently-degraded runner.**
- **T22 — external-scheduler docs.** How an external cron / systemd-timer /
  GitHub-Actions / `pg_cron` trigger drives one run; `pg_cron` is optional and
  must never be the only path.
- **T23 — example + integration suite.** A runnable
  `post_events → post_hourly_stats` rollup (Postgres-first), plus the gated
  integration suite modeled on the existing `*-rising-feed` tests: idempotency,
  resume, backfill, and lock-contention (two runners, one wins).

**Scheduled into v0.10:**

## Goal

Let a developer define a rollup job in TypeScript/Sisal and execute **one**
safe, idempotent run that the database does the heavy lifting for —
extract→group→ aggregate→upsert as generated SQL, with a checkpoint and a lock.

## Package boundary

`@sisal/etl` depends on `@sisal/core` (and an adapter to execute).
**`@sisal/orm` must not depend on `@sisal/etl`.** (See
[architecture](architecture.md).)

## Scope — initial features

- `defineJob()` — declare a job: source, target, grain, window column,
  transform.
- `rollup()` — the grouped extract→aggregate→upsert shape (the v0.6 PoC, now
  typed via the v0.8 IR: insert-from-select + date bucket + filtered
  aggregates + upsert-from-select).
- **checkpoint table** — a watermark store (`job`, `window_end`,
  `pruned_before`, `updated_at`).
- `run(job)` — execute the next window once.
- `backfill(job, range)` — run an explicit historical range deterministically.
- `replay(job, window)` — re-run a specific window idempotently. Both refuse
  windows behind the job's `pruned_before` replay horizon with a typed error
  (the v0.6 A4 replay-vs-retention invariant, test-backed by v0.9); an explicit
  unsafe override acknowledges re-derivation from a restored source.
- `status(job)` — report checkpoint + last-run state.
- **dry-run / explain** — print the generated SQL without executing.
- **SQL-pushdown execution** — the runner sends one (or few) generated
  statements; the database scans/groups/aggregates/upserts. No row round-trips.
- **Postgres-first implementation**; other dialects expose the subset they
  support correctly (per [v0.9](v0.9.0-roadmap.md)).
- **External-scheduler documentation** — how cron/systemd-timer/GitHub-Actions/
  pg_cron triggers a run.

## The execution model (the whole v0.10 runtime)

```
external cron / scheduler wakes up
  → calls the Sisal ETL runner
  → runner reads the job definition
  → runner reads the checkpoint
  → runner computes the next time window  [from, until)
  → runner acquires a lock where supported (pg_advisory_lock; lock-row on SQLite)
  → runner sends generated SQL to the database
  → database scans / groups / aggregates / upserts (set-based, pushdown)
  → runner updates the checkpoint
  → runner exits
```

PostgreSQL may _optionally_ schedule jobs later via `pg_cron`, but the external-
scheduler model is the default — `pg_cron` must never be the only path.

## Non-goals

No full scheduler platform; no worker queue; no Temporal/Airflow clone; no
Rust/native engine; **no DuckDB dependency required** for v0.10.

## Risks

- **Scope creep into a scheduler** — the gravitational pull of "just add cron."
  Held by the non-goals: Sisal defines the job and runs one window.
- **Locking portability** — `pg_advisory_lock` has no SQLite equivalent; the
  v0.9 lock-strategy must be solid or concurrent runs corrupt rollups.
- **Idempotency bugs** — a re-run that double-counts. Upsert keyed on the
  grain + half-open windows `[from, until)` are mandatory invariants.
- **Window boundary errors** — off-by-one at bucket edges (the rising-feed
  examples already show `<= now` discipline).

## Open questions

- Checkpoint table ownership is **answered by v0.6 A3**: default to an
  `@sisal/etl`-managed `sisal_etl_checkpoints` table, unless v0.9 overturns that
  default with evidence and records the resulting dependency edge in
  [architecture](architecture.md). v0.10 consumes the settled boundary; it does
  not decide ownership for the first time.
- Does `run()` own transaction boundaries, or compose with the caller's?
- Multi-statement jobs: a typed `db.batch`-style sequence, or one CTE statement?

## Acceptance criteria

- `defineJob` + `run` execute the `post_events → post_hourly_stats` rollup on
  PostgreSQL as **generated, pushed-down SQL** (verified by `dry-run` output). —
  **T12, T13, T16, T20**
- A run is **idempotent** (replay a window → identical target state) and
  **resumable** (checkpoint advances; a second run does the next window). —
  **T14, T15, T18**
- `backfill(range)` reproduces a historical range deterministically. — **T17**
- Concurrent runs are serialized by the lock strategy — using the
  **v0.9-tested** portable lock abstraction, not a model first built here. Where
  an engine's ETL lock/checkpoint isn't supported, the API is **capability-gated
  to Postgres-first** and returns a typed "unsupported job shape," never a
  silently-degraded runner. — **T16, T21**
- Checkpoint-table ownership is **already decided** (v0.6 default, v0.9
  validation); v0.10 only consumes it. — **T14**
- `@sisal/orm` has **no** dependency on `@sisal/etl`. — **T11**
- The 0.9.0 security-audit follow-ups (**T1–T10** above) are 🟢 and their
  findings moved to **Addressed** in [docs/security.md](security.md) — or a task
  is explicitly re-triaged there with a rationale. **T1 (SEC-008)** is a hard
  prerequisite of the lock-serialization criterion on the MySQL family; **T5
  (SEC-012)** of the checkpoint/replay criteria.
- The carried-forward v0.9 Phase 6 items scheduled here (**CF1–CF3**) are 🟢 or
  carry a recorded re-triage decision. **CF1 (`postgres.js` default)** must not
  regress the pg adapter's `bigint` behavior or its parity tests; **CF2** must
  land as either a capability-gated implementation _or_ a recorded defer with
  benchmark evidence, never a silently-degraded rollup path. CF4–CF7 remain
  deferred and are **not** release-gating.

## Affected packages

New `@sisal/etl`; depends on `@sisal/core` + an adapter; `@sisal/migrate` if the
checkpoint table is a managed system table. No change to `@sisal/orm`'s surface.

- A runnable `post_events → post_hourly_stats` ETL example (Postgres-first;

- Idempotency + resume + backfill + lock-contention tests.

## Dialect compatibility notes

PostgreSQL is the first-class target (full pushdown + advisory locks). Neon
works via the WebSocket Pool but the **single-statement / generated-SQL**
preference matters most here — favor one round trip. SQLite/libSQL: rollups run
where insert-from-select + ON CONFLICT are supported, with a lock-row strategy
and `FILTER`→`CASE WHEN`; document any unsupported job shapes rather than
silently degrading.
