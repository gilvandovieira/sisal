# Sisal v0.9.0 roadmap — Adapter Hardening, Observability & Capability Matrix

With the IR stabilized in [v0.8](v0.8.0-roadmap.md) and **MySQL added in
[v0.7](v0.7.0-roadmap.md)**, v0.9 hardens the **five** adapters against the
ETL/analytics SQL shapes surfaced by
[v0.6](v0.6.0-roadmap.md)/[v0.7](v0.7.0-roadmap.md), and turns the implicit,
scattered dialect knowledge into an **explicit, tested capability surface**.
This is the last hardening step before the public `@sisal/etl` (v0.10) and
`@sisal/analytics` (v0.11) packages are built on top.

## Goal

Make each adapter a known quantity for ETL/analytics: every dialect-divergent
behavior is **declared, documented, and tested**, with explicit fallbacks where
an engine can't match Postgres.

## Scope / focus

- **PostgreSQL** — the strongest ETL/analytics target (insert-from-select,
  FILTER, windows, advisory locks, materialized views); the reference adapter.
- **Neon** — Postgres-compatible but **serverless-aware** (WebSocket Pool;
  prefer generated SQL / atomic statements). Separate generic-Postgres behavior
  from Neon-specific behavior in code and docs.
- **SQLite** — embedded/local compatibility and fallbacks (`FILTER`→`CASE WHEN`,
  no materialized views, lock-row instead of advisory locks).
- **libSQL** — SQLite semantics + **remote/Turso runtime caveats** (hrana/HTTP,
  `ArrayBuffer` BLOBs, `intMode`).
- **MySQL / MariaDB** — harden the v0.7 adapter: `ON DUPLICATE KEY UPDATE`
  upsert, uneven `RETURNING`, `BOOLEAN`=`TINYINT(1)`, `GET_LOCK` named locks,
  and the MySQL-vs-MariaDB split.
- **Transaction behavior differences**; **returning/upsert differences**;
  **timestamp/date function differences** (the v0.6 bucket abstraction per
  dialect); **lock/checkpoint strategy differences** (advisory lock vs lock
  row).
- **Operational observability** — Hibernate-style category/level logging for
  SQL, bind summaries, migration plans/steps/locks/history, and CLI verbosity,
  so adapter hardening has inspectable diagnostics without leaking secrets.
- **Test coverage for dialect-specific fallbacks** — every degraded path
  (FILTER→CASE, ilike→LIKE, bucket math) gets a per-engine integration test.

## Specific hardening items (from the audit)

- **Declarative `(engine, variant, version-range)` capability descriptor** (the
  v0.8 surface): one source of truth the feature matrix, the `dialectGuard`, and
  ETL/analytics read. It must distinguish **neon ≠ pg, libsql ≠ sqlite, and
  mysql ≠ mariadb ≠ by-version** — the 4-value render dialect cannot (it
  collapses neon→pg, libsql→sqlite), so the matrix's six capability targets plus
  a version axis are authoritative.
- **CTE per-engine execution tests** — CTEs render dialect-agnostically today
  but have **no per-engine integration test**; add them for
  pg/neon/sqlite/libsql/mysql/mariadb.
- **Recursive CTE** per-engine verification — the builder itself is **v0.8
  work** (its task list, item 11); v0.9 pins the per-engine/per-version behavior
  it renders (MySQL 8+ vs 5.7, MariaDB, SQLite depth/cycle limits), consistent
  with this release's "no new SQL constructs beyond v0.8" non-goal.
- **MySQL functional (expression) indexes — version-gate, don't fail-closed
  always** (raised by the v0.8 advanced-SQL review, contract 11). The
  `@sisal/mysql` DDL generator currently rejects **every** expression index
  unconditionally ("version-less generator fails closed"), but MySQL **≥
  8.0.13** supports functional indexes; light them through the capability
  descriptor on MySQL 8.0.13+ and keep the typed rejection only below that and
  on **MariaDB** (which has none). Partial indexes stay `❌` for the whole MySQL
  family. Pin with DDL render tests + a live per-engine check.
- **`@sisal/pg` float8 → number** (v0.5 item 11) — the only adapter returning
  `double precision` as a string; fix so numeric reads match the other three
  adapters (critical for analytics math).
- **Insert-from-select / data-modifying CTE** execution verified per engine
  (Postgres yes; SQLite "less fully" — pin the exact supported subset).
- **Advisory-lock equivalence** — implement and test the v0.6 A2 contract the
  ETL runner will use: `pg_try_advisory_lock` / `pg_advisory_unlock` on
  PostgreSQL-family sessions, `BEGIN IMMEDIATE` for supported SQLite/libSQL
  runs, and `GET_LOCK` / `RELEASE_LOCK` for MySQL-family adapters.
- **Checkpoint-table ownership validation** — consume the v0.6 A3 default that
  checkpoints are `@sisal/etl`-managed (`sisal_etl_checkpoints`) rather than an
  `@sisal/migrate` system table. If v0.9 reverses that default, record the
  reason and the resulting `etl → migrate` edge in
  [architecture](architecture.md).
- **Replay-vs-retention refusal** — implement and test the v0.6 A4 invariant:
  the per-job `pruned_before` replay horizon advances atomically with the prune
  (never lagging the delete), and `replay`/`backfill` refuse windows behind it
  with a typed error. Test both directions: a refused replay leaves the rollup
  row unchanged (no zero-overwrite), and a crash between prune and horizon
  advance leaves the horizon ahead or equal — never behind.
- **Hibernate-style logging verbosity** — add shared `SisalLoggingOptions` /
  `SisalLogSettings` over the existing `Logger` contract, with `silent` →
  `trace` thresholds, per-category overrides, optional `trace`, and safe
  redacted SQL bind summaries. Wire it through the ORM facade, adapter ORM
  factories, core migrator, adapter migrators, `defineConfig({ logging })`, and
  the `sisal` CLI (`--log-level`, `--quiet`, repeatable `-v`/`--verbose`).
  Preserve the legacy `logger` shorthand and keep raw bind values out of logs.
- **Explicit resource management (`await using`)** — implement
  `Symbol.asyncDispose` on the closeable facades as an alias for `close()`, so
  `await using db = await connect(...)` and
  `await using migrator = createMigrator(...)` release the pool/connection and
  store at scope exit (including on an early throw) with no `try/finally`.
  Additive and non-breaking; `close()` stays the source of truth. **Done:** core
  `Database` (inherited by every adapter database via `extends Database`) and
  core `Migrator`, with scope-exit/throw-path tests and a README + `llms` doc
  example; JSR publish-clean (no slow types). **Remaining:** the adapter
  migrator facades (`createPgMigrator` / `createSqliteMigrator` / … expose
  `close()` only) get the same alias. Keep the surface async-only (no sync
  `Symbol.dispose`) and note the cross-runtime type floor (`Symbol.asyncDispose`
  needs TS ≥ 5.2 / `esnext.disposable` off Deno).

## Deliverables

- A per-dialect **capability descriptor** wired into the generated feature
  matrix.
- A shared **observability surface** for ORM and migration execution:
  category-aware logging, safe bind summaries, and CLI verbosity flags.
- **`await using` lifecycle support** on the `Database` and `Migrator` facades
  (and the adapter migrator facades), aliasing `close()` through
  `Symbol.asyncDispose`.
- Integration tests for every ETL/analytics-relevant shape on every supporting
  adapter (extending the existing gated `integration/` suites).
- Updated `docs/{pg,neon,sqlite,libsql,mysql}-compatibility.md` + the unified
  matrix.

## Non-goals

No new ETL/analytics package; no new SQL constructs beyond what v0.8 stabilized;
no DuckDB.

## Risks

- **Combinatorial test growth** — every shape × six capability targets ×
  fallbacks. Lean on the existing matrix coverage-guard to keep claims honest.
- **Serverless flakiness** — Neon WebSocket tests need the `wsproxy` image; keep
  them gated and deterministic.
- **Hidden runtime type lies** (like float8) elsewhere — audit numeric/date
  round-trips broadly, not just the known case.

## Open questions

- Recursive CTEs: in scope for analytics (hierarchies) or deferred?
- **Recursive CTE cycle detection** (raised by the v0.8 review, contract 07):
  the shipped `$withRecursive` relies on the portable depth-guard pattern
  (`WHERE depth < n`). Ship the PostgreSQL 14+ `CYCLE`/`SEARCH` clauses as a
  version-gated construct, or keep the depth-guard-only surface? A new construct
  is beyond this release's non-goal, so this is a v0.10+/backlog flag, not v0.9
  build work.
- Is materialized-view DDL worth adding for Postgres (rollup acceleration),
  given SQLite has none?
- How much SQLite-family ETL is "supported" vs "Postgres-first, documented gap"?

## Acceptance criteria

- Every dialect-divergent behavior relevant to ETL/analytics is a declared
  capability with a backing per-engine test (matrix `✅`/`⚠️`/`❌` all
  test-backed). The descriptor is keyed `(engine, variant, version-range)` (the
  v0.7 `DialectIdentity` axis generalized) and distinguishes all six capability
  targets — neon/pg, libsql/sqlite, and mysql/mariadb.
- `@sisal/pg` returns `double precision` as a JS `number`.
- Insert-from-select, upsert-from-select, data-modifying CTEs, filtered
  aggregates, and date buckets each have a per-engine pass/fallback test.
- **The ETL runner's substrate is real, not just designed:** the portable
  claim/advisory-lock abstraction (`pg_try_advisory_lock` / `GET_LOCK` /
  `BEGIN IMMEDIATE` or lock-row) and the checkpoint-table contract each have a
  per-engine test on every engine where ETL is supported (and a documented gap
  where it is not), so v0.10 does not ship a runner on an unbuilt
  lock/checkpoint model. (Promoted from a "hardening item" — it is a v0.10
  prerequisite; see the [sequencing audit](roadmap-sequencing-audit.md).)
- **Checkpoint-table ownership is validated and recorded**: v0.9 consumes the
  v0.6 default (`@sisal/etl`-managed `sisal_etl_checkpoints`) or explicitly
  reverses it with the resulting dependency edge reflected in `architecture.md`,
  so v0.10 consumes a settled boundary — not an open question restated at ship
  time.
- The five compatibility docs and the six-column unified matrix reflect the
  hardened reality.
- Logging verbosity is configurable without breaking existing `logger` users:
  `debug` exposes SQL/timings, `trace` exposes only redacted bind summaries, and
  CLI/config precedence is tested (`--log-level` / `--quiet` / `-v` override
  `defineConfig({ logging })`).
- `await using` closes a `Database` (any adapter) and a `Migrator` at scope exit
  including on an early throw, verified by test; the adapter migrator facades
  expose the same `Symbol.asyncDispose` alias, and `close()` stays directly
  callable.

## Dialect compatibility notes

This release _is_ the dialect compatibility note: it makes the capability matrix
authoritative and machine-checked. PostgreSQL is the reference; Neon mirrors it
with serverless caveats; SQLite/libSQL expose only the subset they support
correctly, with every degradation tested and documented. MySQL/MariaDB share the
`@sisal/mysql` package but keep separate capability rows for named-lock
behavior, the uneven `RETURNING` surface, and MySQL-family typed gaps.

### Phase 1 — Make the capability descriptor authoritative · [v0.10/v0.11 prereq · foundation]

_The descriptor shape was frozen at the v0.8 IR freeze (registry exists,
`dialectGuard` reads it, `docs:matrix:check` guards target-space divergence).
v0.9 makes it the single source of truth — this gates T8, T11 and everything
v0.10/v0.11 consume._

- **T3 · 🟢 Per-cell matrix wiring (item 1 residual).** The pure
  supported/unsupported rows (`distinctOn`, row locking, array operators,
  data-modifying CTE) now derive their `docs/feature-matrix.md` cells from
  `capabilitySupported(...)` via a `capabilityRow` helper, so they cannot drift
  from `DIALECT_CAPABILITIES` — output is byte-identical and `docs:matrix:check`
  stays green. Value-shape ⚠️ rows (JSON/boolean/`dateTrunc`) and fallback rows
  (MySQL-family `RETURNING`, `FULL JOIN`) stay hand-authored — they encode a
  round-trip/fallback axis the support predicate doesn't cover. _Files:_
  `tools/feature_matrix.ts`, `tools/generate_feature_matrix.ts`.
- **T4 · 🟢 Reconcile the key spaces + version axis (GI-1).** The render dialect
  (`SqlDialect` + a new runtime `SQL_DIALECTS` companion), the snapshot dialect
  (`SisalDialectName`), and the six `CAPABILITY_TARGETS` are now
  **machine-checked projections of the one `(engine, variant, version)`
  descriptor**: a reconciliation test asserts the 4-way unions are identical
  (compile-time — the guard was proven to fail on divergence), pins the 6→4
  collapse (neon→pg, libsql→sqlite, mariadb→mysql), verifies every render
  dialect has a target, and rejects capability declarations naming an unknown
  dialect/variant. The matrix's `ADAPTERS ≡ CAPABILITY_TARGETS` check
  (generator) and T3's per-cell derivation cover the matrix projection. Version
  axis + typed fail-closed predicates exist (`unless`/`minVersion`). The
  **base-engine-scoped version gate** — e.g. "MySQL ≥ 8.0.13 but MariaDB never"
  — was later delivered while unblocking T8: `baseEngine: true` on an `unless`
  exception lifts only the base engine (no variant). _Files:_
  `packages/core/sql.ts`, `packages/core/dialect_reconciliation_test.ts`.

### Phase 2 — Adapter correctness & numeric truth

- **T5 · 🟢 `@sisal/pg` `float8`/`float4` → JS `number` (v0.5 item 11).**
  **Already shipped in v0.5.0** (commit `cfb3dfa`): `coerceFloatColumns` in the
  pg executor maps float4/float8 (OIDs 700/701) string values from
  `@db/postgres` to `number` on every query/transaction path, leaving
  `numeric`/`bigint` strings for precision. The DB-gated integration scenario,
  the matrix `Float …
  → number` row, and the round-trip reference were shipped
  with it. The v0.9 re-listing was a stale-doc artifact — the extraction
  inferred "not shipped" from a missing status paragraph, but the code shipped
  it. This pass added the missing **network-free unit test** (proven to catch a
  regression) so the coercion is locked into `deno task test`, not only the
  gated integration suite. _Feeds T6._ _Files:_
  `packages/pg/orm/executor_test.ts`.
- **T6 · 🟢 Broad numeric/date round-trip suite.** Added a `bigint` axis to the
  shared `IntegrationValueShape` (pg/neon/mysql/mariadb → `"string"`,
  sqlite/libsql → `"number"`) so the decode contract is declared per target, and
  wrote a `numeric + bigint` round-trip scenario in each of the three family
  scenario files. Each asserts `typeof value === valueShape.{numeric,bigint}`;
  the string families use an int8 above 2^53 (`9007199254740993`) to prove no
  lossy-float path, while the SQLite family stays under 2^53 and documents that
  larger int8s would lose precision there. Type-checks across all six suites;
  DB-gated, so the runtime assertions await a live integration run. _Files:_
  `integration/_shared/target.ts`, `integration/_targets/*.ts`,
  `integration/_shared/{postgres,sqlite,mysql}_family_scenarios.ts`.
- **T7 · 🟢 Resolve the `bigint` cross-adapter split (decided: Postgres family →
  `string`).** `bigint` decoded to `BigInt` on `@sisal/pg` but a **string** on
  `@sisal/neon` (`2n === "2"` silently false across siblings). Resolved by

### Phase 3 — ETL correctness substrate — implement + per-engine test · [v0.10 prereq · critical path]

_v0.6 A2/A3/A4 shipped as **design only**; the sequencing audit promotes them to
v0.9 acceptance because v0.10's own acceptance hard-requires them ("concurrent
runs serialized by the lock strategy"). Ship them **real and tested**, or v0.10
ships a public runner around an unbuilt model._

- **T11 · 🟢 Portable advisory-lock / queue-claim abstraction (A2 / contract
  08).** [dep: T4] Shipped `Database.tryAdvisoryLock(name, options?)` as a
  **lightweight lock-row lease** rather than the contract's per-dialect
  session-scoped native locks (`pg_advisory_lock` / `GET_LOCK` /
  `BEGIN
  IMMEDIATE`). _Rationale:_ a session-scoped lock pins a connection and
  a server-side lock for the whole run — the exact cost we want to avoid — and
  does not compose with the chosen `await using` disposal (dispose cannot signal
  commit-vs-rollback for `BEGIN IMMEDIATE`). Instead the claim is one row in
  `sisal_advisory_locks` (default; overridable via `{ table }` so Sisal never
  forces its name into a user's schema) carrying an ISO-8601 **lease** — reap
  the name if its lease lapsed, then `ON CONFLICT DO NOTHING` claim;
  `await
  using` deletes the row on scope exit; `renew()` extends the lease and
  a `false` means "lock lost". No connection or server lock is held between
  acquire and release, and it is **uniform across all six engines** (plain DML,
  no `acquireSession` driver primitive, no adapter changes) — matching the v0.6
  compat table / v0.10 roadmap's "lock-row on SQLite". Fails closed on the
  `generic` dialect (`ORM_DIALECT_UNSUPPORTED`). Trade-off: a crashed holder's
  row lingers until its lease expires (then it is stolen), so long runs must
  `renew()` and honor a failed renew. _Files:_
  `packages/orm/core/advisory_lock.ts`, `packages/orm/core/database.ts`,
  `packages/orm/advisory_lock_test.ts` (10 network-free tests),
  `tools/feature_matrix.ts`, the three family scenario files. _Acceptance:_
  per-engine contention scenario on every ETL target (DB- gated); matrix row
  "Advisory run lock (portable lock-row lease)".
- **T12 · 🟢 Atomic load+advance invariant (A4 substrate).** Shipped
  `etlCheckpoint(db, job, options?)` (standalone factory, mirrors
  `defineAtomicOperation`): `read()` returns the last committed `window_end`
  watermark (or `null`); `advance(until, load)` runs the caller's idempotent
  load and the watermark upsert as **one `db.batch`**, so they commit together
  and a crash never advances the watermark past un-written data (never ahead of
  the load). Watermarks (`window_end`/`pruned_before`) are stored as **opaque
  TEXT** (ISO-8601 by convention; caller owns the meaning) in the
  `sisal_etl_checkpoints` table — overridable via `{ table }`, created on first
  use — which keeps the checkpoint uniform across all six engines with no
  per-adapter timestamp-decode divergence (T11's lesson). The table already
  carries the nullable `pruned_before` column for the T14 retention horizon; T12
  reads/advances `window_end` only. The DDL runs **outside** the atomic batch
  (MySQL auto-commits DDL). Backed by 7 network-free unit tests (asserting the
  single atomic batch) and a per-engine contention scenario proving idempotent
  resume + crash-safety (a failing load rolls the whole batch back and the
  watermark stays behind). _Rationale (decided with the user):_ TEXT
  watermarks + `db.batch` + standalone factory. _Files:_
  `packages/orm/core/checkpoint.ts`, `packages/orm/checkpoint_test.ts`,
  `tools/feature_matrix.ts`, the three family scenario files. Underpins T13/T14.
- **T13 · 🟢 Validate the checkpoint/watermark contract (A3 / contract 09 / item
  18).** The A3 ownership decision **held and is recorded** in
  `architecture.md`: the T12 checkpoint substrate ships in **`@sisal/orm`** (it
  needs the `Database` facade), the `sisal_etl_checkpoints` table is
  created/managed by the ORM helper at runtime, and there is **no
  `etl → migrate` edge** — so no reversal, no migrate edge to add. Added
  `readState()` to the `Checkpoint` handle (the full
  `{ windowEnd, prunedBefore, updatedAt }` contract row — also what T14 reads
  for the horizon), unit-tested. The "adapter-specific timestamp types"
  validation target was **eliminated by the T12 TEXT-watermark decision** —
  there is no per-engine timestamp surface to reconcile; T13 instead validates
  **exact TEXT round-trip fidelity** (a precise ISO string reads back
  byte-identical), **multi-job independence** (`job` is the PK), **`updated_at`
  written**, and **persistence across a fresh handle** — a per-engine contention
  scenario on every ETL target. _Files:_ `packages/orm/core/checkpoint.ts`,
  `packages/orm/checkpoint_test.ts`, `docs/architecture.md`, the three family
  scenario files.
- **T14 · 🟢 Replay-vs-retention refusal (A4).** Extended the `Checkpoint`
  handle with the mirror of `advance`: **`prune(before, deletes)`** upserts the
  per-job `pruned_before` horizon **in the same `db.batch`** as the
  source-delete statements — so the horizon moves only if the delete committed
  and never lags it (a crash rolls both back together; the horizon may run
  ahead, never behind). **`assertReplayable(from, options?)`** reads the horizon
  (via `readState`) and throws a typed **`ORM_REPLAY_PRUNED`** error when
  `from < pruned_before` (the window's source rows were pruned, so a replay
  would overwrite the rollup with missing data), with an explicit
  **`unsafeAllowPrunedReplay`** override mirroring `.unsafeAllowAllRows()`. TEXT
  markers compare lexicographically, so `from >= pruned_before` is the
  replayable boundary. _Acceptance (both directions), covered by a per-engine
  scenario:_ a refused replay throws before the load runs so the rollup row is
  untouched; a failing prune rolls the delete and the horizon back together
  (horizon stays put, rows not deleted). Backed by 13 network-free unit tests
  (prune atomicity, both refusal directions, the override, the null-horizon
  allow) and the matrix row "Retention horizon + replay refusal (ETL)". _Files:_
  `packages/orm/core/checkpoint.ts`, `packages/orm/checkpoint_test.ts`,
  `tools/feature_matrix.ts`, the three family scenario files.
- **T15 · 🟢 Write-result abstraction — inserted vs conflicted/claimed (v0.8
  item 16).** Shipped `tryInsert(db, insert): Promise<WriteOutcome>` — a
  portable **inserted-vs-conflicted** signal for a conflict-guarded insert (you
  supply `.onConflictDoNothing()`). It reads the signal the reliable way **per
  dialect**: `RETURNING` on the Postgres and SQLite families (a row comes back
  iff the insert won — exact), the affected-row count on the MySQL family (which
  has no usable `RETURNING` on a no-op `ON DUPLICATE KEY UPDATE` — 1 on insert,
  0 on conflict). Fails closed on `generic`. **T11's advisory-lock claim now
  consumes it** — so claimed-vs-not is reliable across engines instead of the
  ambiguous-under-MySQL `rowCount >= 1` it used before (the exact v0.8 pain
  point). Backed by 5 network-free unit tests (both dialect paths × both
  outcomes + the generic guard), a per-engine integration scenario, and the
  matrix row "Write outcome (inserted vs conflicted/claimed)". _Rationale:_
  dialect-gated RETURNING (MySQL/MariaDB both reject RETURNING on the no-op
  upsert, so the gate is by dialect, not the `returning` capability). _Files:_
  `packages/orm/core/write_outcome.ts`, `packages/orm/write_outcome_test.ts`,
  `packages/orm/core/advisory_lock.ts` (+ its test), `tools/feature_matrix.ts`,
  the three family scenario files. **Closes the Phase 3 ETL substrate
  (T11–T15).**
- **Open question to close:** how much SQLite-family ETL is "supported" vs
  "Postgres-first, documented gap"? If the substrate slips, scope v0.10
  **Postgres-only at the API level** — SQLite/libsql return a typed "unsupported
