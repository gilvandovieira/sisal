Sisal is a Deno-first toolkit, published to JSR, for typed schemas, planned migrations, and small Postgres, Neon, SQLite, and libSQL adapters. The ORM is driverless and never imports an adapter.

schema.ts @sisal/orm 
import { columns, defineTable } from "@sisal/orm";

const users = defineTable("users", {
  id: columns.uuid().primaryKey(),
  email: columns.text().notNull().unique(),
  name: columns.text().notNull(),
  createdAt: columns.timestamp({ withTimezone: true }),
});

// Row + insert shapes are inferred, no codegen step.
type User = InferSelect<typeof users>;
Six packages, one weave

Threads that never tangle

Each package owns one boundary and depends only on what it must. Import the smallest surface your app needs.

@sisal/orm

The driverless core: schema definitions, typed SQL fragments, predicates, immutable query builders, structured errors, and serializable schema snapshots.

Adapters depend on @sisal/orm.
The ORM never depends on an adapter.

Why Sisal

Small surface, strong guarantees

Database tooling deserves clear boundaries and predictable behaviour over a sprawling framework.

Strict boundaries

The ORM imports no adapter and no legacy namespace. Adapters build on the ORM, never the other way around.

Immutable by design

Column modifiers and query builders return new instances, so a fragment is safe to share and compose.

Safety rails

update and delete without a where throw, unless you opt in with unsafeAllowAllRows().

Planned migrations

Checksums, drift checks, and advisory-locked, transactional runs. Destructive changes are withheld for explicit handling.

Dialect-aware SQL

Parameters render as $1, $2 for Postgres and ? elsewhere; identifiers are quoted per dialect.

Drizzle parity

A compatibility map against Drizzle ORM 0.45.2, with the few Sisal divergences documented on purpose.

From schema to SQL

Snapshot it, diff it, ship it

Build a normalized snapshot from your tables, then let the adapter emit the DDL. Only additive changes are generated — destructive ones come back for you to decide.

  • API reference
  • Migration notes
  • Drizzle parity
migrate.ts @sisal/pg 
import { columns, createSchemaSnapshot, defineTable } from "@sisal/orm";
import { generatePostgresUpStatements } from "@sisal/pg/ddl";

const users = defineTable("users", {
  id: columns.uuid().primaryKey(),
  email: columns.text().notNull().unique(),
});

const snapshot = createSchemaSnapshot({
  dialect: "postgres",
  tables: [users],
});

// Additive DDL only; destructive changes are returned separately.
const { statements, destructive } = generatePostgresUpStatements(snapshot);
