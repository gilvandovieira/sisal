Sisal is a Deno-first toolkit, published to JSR, for typed schemas, planned migrations, and small Postgres & SQLite adapters. The ORM is driverless and never imports an adapter.

schema.ts @sisal/orm 
import { columns, defineTable } from "@sisal/orm";

const users = defineTable("users", {
  id: columns.uuid().primaryKey(),
  email: columns.text().notNull().unique(),
  name: columns.text().notNull(),
  createdAt: columns.timestamp({ withTimezone: true }),
});

// Row + insert shapes are inferred, no codegen step.
type User = InferSelect<typeof users>;
Four packages, one weave

Threads that never tangle

Each package owns one boundary and depends only on what it must. Import the smallest surface your app needs.

Adapter-neutral

@sisal/migrate

Migration definitions, checksums, planning, drift checks, the file workflow, and a generic runner.

./core ./workflow

PostgreSQL

@sisal/pg

Postgres execution, advisory-locked history, a migrator, and additive DDL generation.

./orm ./migrate ./ddl

SQLite

@sisal/sqlite

SQLite execution with a lazily-opened database, history storage, a migrator, and DDL.

./orm ./migrate ./ddl

@sisal/orm

The driverless core: schema definitions, typed SQL fragments, predicates, immutable query builders, structured errors, and serializable schema snapshots.

Adapters depend on @sisal/orm.
The ORM never depends on an adapter.

Why Sisal

Small surface, strong guarantees

Database tooling deserves clear boundaries and predictable behaviour over a sprawling framework.

Strict boundaries

The ORM imports no adapter and no legacy namespace. Adapters build on the ORM, never the other way around.

Immutable by design

Column modifiers and query builders return new instances, so a fragment is safe to share and compose.

Safety rails

update and delete without a where throw, unless you opt in with unsafeAllowAllRows().

Planned migrations

Checksums, drift checks, and advisory-locked, transactional runs. Destructive changes are withheld for explicit handling.

Dialect-aware SQL

Parameters render as $1, $2 for Postgres and ? elsewhere; identifiers are quoted per dialect.

Drizzle parity

A compatibility map against Drizzle ORM 0.45.2, with the few Sisal divergences documented on purpose.

From schema to SQL

Snapshot it, diff it, ship it

Build a normalized snapshot from your tables, then let the adapter emit the DDL. Only additive changes are generated — destructive ones come back for you to decide.

  • API reference
  • Migration notes
  • Drizzle parity
migrate.ts @sisal/pg 
import { columns, createSchemaSnapshot, defineTable } from "@sisal/orm";
import { generatePostgresUpStatements } from "@sisal/pg/ddl";

const users = defineTable("users", {
  id: columns.uuid().primaryKey(),
  email: columns.text().notNull().unique(),
});

const snapshot = createSchemaSnapshot({
  dialect: "postgres",
  tables: [users],
});

// Additive DDL only; destructive changes are returned separately.
const { statements, destructive } = generatePostgresUpStatements(snapshot);

How fast is it? Faster than you can notice

When an app talks to a database, almost all the waiting is the network and the database itself. Sisal's job — turning your code into a database query — happens in millionths of a second, so it is never what slows you down. We measure it against a stand-in database with zero network delay, so these are pure query-writing speeds.

100,000+ queries written in a single blink before the database has even replied
1000× longer to reach a database than to write the query the network trip is the slow part, not Sisal
≈ 0% added cost to switch database brands the same query for Postgres or SQLite
Linear bulk inserts scale evenly with rows 100 rows is about 30× one query — no surprises
Instant to plan a brand-new database layout a few simple queries' worth of work
Never the part of your app that users wait on writing SQL is dwarfed by the database itself

In short: the database and the network do the waiting; Sisal's share of the time is too small to feel. Prefer the raw numbers? Open Full results.
