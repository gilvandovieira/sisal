# Sisal v0.8.0 roadmap — Advanced SQL IR & Expression Stabilization

v0.6 and v0.7 are investigations; v0.8 is where their findings become a **stable
substrate**. The goal is to stabilize Sisal's internal SQL representation and
expression system — and extract **`@sisal/core`** — so that the future
`@sisal/etl` and `@sisal/analytics` packages can **compile into Sisal safely**,
without forcing invasive ORM rewrites later.

This release ships the first shared **core** package boundary —
**`@sisal/core`** — the substrate every other package builds on (v0.7 already
added an adapter package, `@sisal/mysql`).

the guard machinery without a net. The dependency-compatible order is five
waves; items within a wave can run in parallel.

## Goal

Turn today's two-tier representation into a documented, versioned, public
**compile target**: a stable fragment IR + expression system + dialect interface
that downstream packages build on, with golden per-dialect SQL tests guarding
it.

## What grounds this (audit)

Sisal already has a clean internal split: a **dialect-agnostic fragment IR**
(`Sql`/`SqlChunk`, `Condition`, `SqlExpression`, `OrderTerm`) rendered by a
single two-phase renderer, plus a **module-private** statement-level state
machine (`SelectState` + the clause assembler). The lower tier (`errors`, `sql`,
`operators`, `columns`, `table`, `temporal`, zero-import `schema.ts`) has **no
upward edges** into the OLTP builders, and the one cross-cut
(builder-as-subquery) is already inverted behind `QUERY_BUILDER_BRAND` +
`SubquerySource`. So extraction is **file-moves + barrel-split + a new
`deno.json`**, not a rewrite.

The key limitation v0.8 must consciously address: the fragment IR is
**compose-oriented, not transform-oriented** — expressions are stored
already-lowered (`eq()` flattens immediately); there is **no introspectable,
rewritable relational AST**.

## Scope / focus

- **Extract `@sisal/core`** — move the lower tier into a `@sisal/core` package
  (schema primitives, fragment IR, expressions/operators, dialect interface +
  renderer); re-point `@sisal/orm`, `@sisal/migrate`, and the adapters at it.
- **Stable SQL IR/AST boundaries** — version the `Sql`/`SqlChunk` contract;
  decide and document what is public (compile target) vs internal.
- **Expression aliasing** — first-class typed aliases for projection/metric
  columns (needed by ETL rollup columns and analytics metrics).
- **Computed columns** and **nested selects** — typed, reusable.
- **CTE rendering** — including the path to **data-modifying CTEs**
  (`WITH … INSERT/UPDATE/DELETE … RETURNING`, v0.5 item 12) and
  **insert-from-select rendering** (the v0.6 gap).
- **Aggregate expression rendering**; **window function rendering** where
  supported (the v0.7 prerequisite).
- **Dialect capability gates** — generalize the v0.5 `dialectGuard` into the
  explicit capability surface v0.9 hardens.
- **Golden SQL tests per dialect**; **parameter-binding correctness**;
  **prepared-plan compatibility**; **type inference for computed expressions**.

## Deliverables

- `@sisal/core` package with a documented public IR/compile-target surface.
- A statement-assembly seam: decide whether to **expose** a minimal builder-
  assembly API (so ETL/analytics reuse SELECT/CTE assembly) or keep it private
  and have downstream packages compile fragments directly. This decides whether
  ETL/analytics can stay off `@sisal/orm` (the dependency-direction gate), so it
  must be settled in v0.8, before v0.10/v0.11 consume core. (See open
  questions.)
- Golden-SQL snapshot tests per dialect for every IR construct.
- Migration notes for the `@sisal/orm` → `@sisal/core` import changes
  (`@sisal/orm/core` stays as a compatibility re-export; the notes cover when to
  prefer importing `@sisal/core` directly).

## Non-goals

No ETL/analytics package (v0.10/v0.11); no DuckDB; no query optimizer/planner;
no breaking change to the public ORM builder API — including import paths:
`@sisal/orm/core` **remains as a compatibility re-export** of `@sisal/core`, so
existing imports keep working; new code should import `@sisal/core` directly.

## Risks

- **The "do we need a transformable AST?" fork** — predicate pushdown / plan
  rewriting (relevant to DuckDB pushdown, v0.13+) would be **new core work** the
  current fragment IR cannot do. Deciding this wrong over- or under-builds v0.8.
- **Public IR = compatibility commitment** — once `@sisal/core`'s IR is public,
  it's a contract; under-specifying it forces a later breaking change.
- **Extraction churn** — touching import paths across every package; mitigated
  by the clean DAG but still broad.
- **Window/data-modifying-CTE rendering divergence** across dialects.

## Findings from MySQL-family examples

Building `examples/mysql-family-showcase` and `examples/mysql-family-feed` made
these v0.8 pressure points concrete:

- **DDL rollback is not portable.** The PostgreSQL showcase can run inside a
  rollback transaction; MySQL/MariaDB DDL implicitly commits, so the MySQL
  showcase must create namespaced tables and drop them in cleanup.
- **Write-result shape matters.** MySQL proper lacks general `.returning()`, so
  flows need `insertReturning` or fetch-after-write. MariaDB lights only some
  `RETURNING` forms by variant/version, proving the capability descriptor must
  be declarative and version-aware.
- **Actor dedupe exposes a missing abstraction.** The feed detects first touch
  in a bucket by pre-checking the actor row and then reading
  `onConflictDoNothing().execute().rowCount` on the insert path; rowCount alone
  was not reliable with the portable MySQL no-op update rendering. Sisal has no
  higher-level "inserted vs conflicted" result, so application code must know
  affected-row semantics.
- **MySQL upsert assignment order leaks through.** The bucket upsert must assign
  the derived `activity_score` before incrementing the counters, because MySQL
  evaluates `ON DUPLICATE KEY UPDATE` assignments left-to-right. A future
  expression/assignment abstraction should make that hazard harder to miss.
- **Expression reuse is still awkward.** The builder-native CTE recompute
  repeats filtered aggregate fragments and derived metric expressions; it also
  uses raw `sql` for `coalesce`, `greatest`, and the final rising-score formula.
  This strengthens the case for expression aliases and computed fields in the
  public compile target.
- **Statement assembly is a real boundary.** The feed wants a stable
  statement-assembly or `@sisal/core` compile target for chained SELECT CTEs,
  multi-table update, MariaDB's CTE-update fallback, and fetch-after-write
  orchestration, but today that composition depends on ORM builders.
- **Time handling must be explicit.** MySQL `DATETIME(6)` paths need a
  documented UTC literal policy (`YYYY-MM-DD HH:mm:ss.SSS000`), and helpers such
  as `dateTrunc`/`dateBin` return string-like values on MySQL/SQLite instead of
  PostgreSQL timestamp values.

RETURNING`. | Design a portable queue-claim/advisory-lock abstraction instead of making apps branch on locking style.                       |
| 09 Idempotent backfill           | pg/neon, mysql/mariadb                            | Builder rollup plus raw checkpoint upsert; SQLite skipped.                                                                | Define checkpoint/watermark contracts and failure-direction tests before `@sisal/etl`.                                        |
| 10 JSON table extraction         | pg/neon, mysql/mariadb; sqlite/libsql when probed | Parameterized raw `jsonb_to_recordset`, `JSON_TABLE`, or `json_each`.                                                     | Add JSON-table/set-returning function IR with typed projected columns and version gates.                                      |
| 11 Generated columns and indexes | pg/neon, mysql/mariadb, sqlite/libsql when probed | Raw DDL; MySQL partial indexes intentionally omitted as impossible.                                                       | Add generated columns, expression indexes, partial indexes, and typed unsupported-combination errors to schema snapshots/DDL. |
| 12 MySQL compatibility           | mysql/mariadb                                     | ODKU renders builder-native; base MySQL `RETURNING` renders a typed guard.                                                | Move version/variant capabilities into one descriptor consumed by `dialectGuard`, matrix generation, examples, and adapters.  |

Cross-cutting pain points from the graduation:

- **Raw SQL is safe but too necessary.** The examples use parameterized `sql`
  for windows, recursive CTEs, JSON-table projection, and richer DDL. v0.8
  should reduce these to small escape hatches, not make downstream packages
  assemble whole statements by hand.
- **Capability probing is duplicated.** SQLite/libSQL optional paths need local
  probes for JSON/window/generated/`RETURNING` support. v0.8's capability
  descriptor should make examples, guards, and the feature matrix read the same
  source of truth.
- **Generated DDL is still table-column-only.** Generated columns, partial
  indexes, expression indexes, and MySQL's lack of partial indexes are outside
  schema snapshots, so examples must issue raw DDL.
- **Write-result semantics remain leaky.** MySQL `RETURNING`, ODKU affected-row
  behavior, and SQLite CAS `UPDATE ... RETURNING` all point to a future
  "inserted/claimed/conflicted" result abstraction.
- **Statement assembly is the key package-boundary decision.** The examples can
  render advanced SQL today, but ETL/analytics packages need a stable assembler
  or core compile target that does not depend on ORM internals.

## Open questions

- ~~**Compose-only fragment IR, or add an introspectable expression AST?**~~
  **Decided (wave 1, 2026-07-02): defer the AST, ship the seam** — the fragment
  IR stays the compile target, and the additive extension point is **shipped**:
  an opaque `meta` slot on `SqlChunk`
  (`SqlChunkMeta`/`withSqlChunkMeta`/`sqlChunkMeta`), renderer-ignored and
  carried by reference through composition, proven by the round-trip fixture
  `packages/orm/sql_meta_seam_test.ts` (annotations survive the `sql` tag and
  builder assembly with byte-identical render output). A transformable AST can
  populate the slot later as a non-breaking version bump — v0.13 DuckDB pushdown
  is unblocked either way (audit Fix 3 satisfied).
- ~~Expose the statement assembler publicly, or require downstream packages to
  compile fragments + a thin assembly helper?~~ **Decided (wave 1, 2026-07-02):
  expose a minimal assembly API from `@sisal/core`** — deterministic
  assemble-from-parts functions (select / insert-from-select with group-by and
  upsert), **not** the fluent OLTP builder, which stays in `@sisal/orm`.
  Rationale: fragments-only would force ETL/analytics to hand-assemble whole
  statements (the "raw SQL is too necessary" pain point), while exposing the
  fluent builder would drag the OLTP surface into the compile target. The G1
  dependency direction holds: `@sisal/etl` / `@sisal/analytics` depend on core
  only. The wave-4 core-only fixture (item 13) may use exactly this API, nothing
  more.
- ~~Does `@sisal/migrate` depend on `@sisal/core` only, or still on
  `@sisal/orm`?~~ **Decided (wave 1, 2026-07-02): core only.** Everything
  `@sisal/migrate` consumes (the snapshot contract in `schema.ts`, errors, the
  render primitives) lives in the extracted lower tier; no OLTP-builder
  dependency exists to keep. The wave-2 extraction re-points it, and the
  import-migration notes record the change. Coupled decision, same day:
  **checkpoint-table ownership stays `@sisal/etl`-managed**
  (`sisal_etl_checkpoints`, the v0.6 A3 default, ratified per audit Fix 2) — no
  `etl → migrate` edge is added, `architecture.md`'s graph is already correct,
  and v0.9 validates the contract per engine (item 18 closed).

## Acceptance criteria

- `@sisal/core` exists, is published to **JSR** (npm deferred — see the
  [npm-release readiness report](npm-release-readiness.md)), and
  `@sisal/orm`/adapters import from it; the public ORM API is unchanged for
  users (`@sisal/orm/core` stays as a compatibility re-export of `@sisal/core`).
- Every IR construct (select/CTE/insert-from-select/aggregate/window where
  supported) has golden per-dialect SQL tests and correct parameter binding.
- A downstream package can construct and render a grouped insert-from-select +
  filtered aggregate using only `@sisal/core` exports (proven by a fixture).
- The transformable-AST decision is documented (build / defer / reject) **and,
  if deferred, the additive extension point is shipped and proven** by a fixture
  that round-trips a fragment through the seam — so v0.13 cannot be blocked by a
  frozen IR.
- The capability descriptor's key is **`(engine, variant, version-range)`** —
  the v0.7 `DialectIdentity` `(engine, variant, version)` axis generalized with
  version-range predicates. It must express all six matrix/integration
  capability targets (`pg`, `neon`, `sqlite`, `libsql`, `mysql`, `mariadb` —
  neon ≠ pg and libsql ≠ sqlite as well as the MySQL/MariaDB variant split), and
  `dialectGuard` is generalized to a version-aware capability check (e.g. MySQL
  ≥ 8.0.21 for `RETURNING`, Postgres ≥ 14 for `CYCLE`, SQLite ≥ 3.31). It is the
  single source the matrix, `dialectGuard`, and ETL/analytics read.
- The **statement-assembly seam decision is closed** — exposed minimal assembly
  API or fragments-only — documented, and reflected in the core-only fixture
  (which may use exactly what the decision makes public, nothing more).
- The **`@sisal/migrate` dependency decision is closed** (core-only vs
  `@sisal/orm`) and reflected in the extraction's import graph and the migration
  notes.

## Dialect compatibility notes

Capability gates become **declarative** (an `(engine, variant, version-range)`
capability descriptor — keyed to express the six matrix targets, **not** the
4-value render dialect, which still collapses neon→postgres and libsql→sqlite)
rather than scattered render branches. It is the single source the v0.9 matrix
and the `dialectGuard` both read. Window/data-modifying-CTE/insert-from-select
rendering is Postgres-first with explicit gates for the SQLite and MySQL
families.
