# Sisal v0.8.0 roadmap — Advanced SQL IR & Expression Stabilization

v0.6 and v0.7 are investigations; v0.8 is where their findings become a **stable
substrate**. The goal is to stabilize Sisal's internal SQL representation and
expression system — and extract **`@sisal/core`** — so that the future
`@sisal/etl` and `@sisal/analytics` packages can **compile into Sisal safely**,
without forcing invasive ORM rewrites later.

This release ships the first new package boundary: **`@sisal/core`**.

## Goal

Turn today's two-tier representation into a documented, versioned, public
**compile target**: a stable fragment IR + expression system + dialect interface
that downstream packages build on, with golden per-dialect SQL tests guarding
it.

## What grounds this (audit)

Sisal already has a clean internal split: a **dialect-agnostic fragment IR**
(`Sql`/`SqlChunk`, `Condition`, `SqlExpression`, `OrderTerm`) rendered by a
single two-phase renderer, plus a **module-private** statement-level state
machine (`SelectState` + the clause assembler). The lower tier (`errors`, `sql`,
`operators`, `columns`, `table`, `temporal`, zero-import `schema.ts`) has **no
upward edges** into the OLTP builders, and the one cross-cut
(builder-as-subquery) is already inverted behind `QUERY_BUILDER_BRAND` +
`SubquerySource`. So extraction is **file-moves + barrel-split + a new
`deno.json`**, not a rewrite.

The key limitation v0.8 must consciously address: the fragment IR is
**compose-oriented, not transform-oriented** — expressions are stored
already-lowered (`eq()` flattens immediately); there is **no introspectable,
rewritable relational AST**.

## Scope / focus

- **Extract `@sisal/core`** — move the lower tier into a `@sisal/core` package
  (schema primitives, fragment IR, expressions/operators, dialect interface +
  renderer); re-point `@sisal/orm`, `@sisal/migrate`, and the adapters at it.
- **Stable SQL IR/AST boundaries** — version the `Sql`/`SqlChunk` contract;
  decide and document what is public (compile target) vs internal.
- **Expression aliasing** — first-class typed aliases for projection/metric
  columns (needed by ETL rollup columns and analytics metrics).
- **Computed columns** and **nested selects** — typed, reusable.
- **CTE rendering** — including the path to **data-modifying CTEs**
  (`WITH … INSERT/UPDATE/DELETE … RETURNING`, v0.5 item 12) and
  **insert-from-select rendering** (the v0.6 gap).
- **Aggregate expression rendering**; **window function rendering** where
  supported (the v0.7 prerequisite).
- **Dialect capability gates** — generalize the v0.5 `dialectGuard` into the
  explicit capability surface v0.9 hardens.
- **Golden SQL tests per dialect**; **parameter-binding correctness**;
  **prepared-plan compatibility**; **type inference for computed expressions**.

## Deliverables

- `@sisal/core` package with a documented public IR/compile-target surface.
- A statement-assembly seam: decide whether to **expose** a minimal builder-
  assembly API (so ETL/analytics reuse SELECT/CTE assembly) or keep it private
  and have downstream packages compile fragments directly. (See open questions.)
- Golden-SQL snapshot tests per dialect for every IR construct.
- Migration notes for the `@sisal/orm` → `@sisal/core` import changes.

## Non-goals

No ETL/analytics package (v0.10/v0.11); no DuckDB; no query optimizer/planner;
no breaking change to the public ORM builder API beyond import paths.

## Risks

- **The "do we need a transformable AST?" fork** — predicate pushdown / plan
  rewriting (relevant to DuckDB pushdown, v0.13+) would be **new core work** the
  current fragment IR cannot do. Deciding this wrong over- or under-builds v0.8.
- **Public IR = compatibility commitment** — once `@sisal/core`'s IR is public,
  it's a contract; under-specifying it forces a later breaking change.
- **Extraction churn** — touching import paths across every package; mitigated
  by the clean DAG but still broad.
- **Window/data-modifying-CTE rendering divergence** across dialects.

## Open questions

- **Compose-only fragment IR, or add an introspectable expression AST?**
  Recommendation: keep the fragment IR as the compile target; add a
  transformable AST **only** if a concrete consumer (DuckDB pushdown,
  optimization) needs it — defer the decision with a documented extension point.
- Expose the statement assembler publicly, or require downstream packages to
  compile fragments + a thin assembly helper?
- Does `@sisal/migrate` depend on `@sisal/core` only, or still on `@sisal/orm`?

## Acceptance criteria

- `@sisal/core` exists, is published (JSR + npm per v0.6 packaging), and
  `@sisal/orm`/adapters import from it; the public ORM API is unchanged for
  users.
- Every IR construct (select/CTE/insert-from-select/aggregate/window where
  supported) has golden per-dialect SQL tests and correct parameter binding.
- A downstream package can construct and render a grouped insert-from-select +
  filtered aggregate using only `@sisal/core` exports (proven by a fixture).
- The transformable-AST decision is documented (build / defer / reject) with
  rationale.

## Affected packages

New `@sisal/core`; `@sisal/orm`, `@sisal/migrate`, `@sisal/pg`, `@sisal/neon`,
`@sisal/sqlite`, `@sisal/libsql` (re-point imports). No app-visible API break.

## Required examples / tests

- Golden-SQL suites per dialect for all IR constructs.
- A `@sisal/core`-only fixture compiling the v0.6 `post_hourly_stats` rollup
  SQL.
- Prepared-plan round-trip tests.

## Dialect compatibility notes

Capability gates become **declarative** (a per-dialect capability descriptor)
rather than scattered render branches — the single source the v0.9 matrix and
the `dialectGuard` both read. Window/data-modifying-CTE/insert-from-select
rendering is Postgres-first with explicit gates for the SQLite family.
