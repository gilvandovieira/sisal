# Sisal v0.11.0 roadmap — `@sisal/analytics` Preview

With the analytical IR designed ([v0.7](v0.7.0-roadmap.md)) and the core IR
stabilized ([v0.8](v0.8.0-roadmap.md)), v0.11 ships the first **Postgres-first
analytical query layer** — a typed OLAP API that queries rollups and analytical
tables using Sisal schema/types. It complements `@sisal/etl`: ETL builds the
rollups, analytics queries them.

## Status legend

| Symbol | Meaning                                                    |
| ------ | ---------------------------------------------------------- |
| 🔴     | Not started · 🟡 In progress · 🟢 Done — move to CHANGELOG |

## Goal

A typed analytical query API (metrics × dimensions × windows) that compiles into
the `@sisal/core` IR, infers result-row types from the metric/dimension set, and
renders correct SQL — Postgres-first, with other dialects exposing only the
subset they support correctly.

## What v0.5 → v0.10 already delivered

v0.11 should **consume** the previous roadmap substrate, not redesign it:

- **v0.5** made the OLTP builder dialect-honest: typed dialect guards, the
  generated feature matrix, `filter()` aggregates, `dateTrunc()`/`dateBin()`,
  date math, data-modifying CTEs, typed raw-query mapping, and numeric fixes
  that analytics math depends on.
- **v0.6** proved the ETL rollup spine and handed off the genuine analytics
  gaps: window functions, arrays/set-returning shapes, period comparisons, and
  rollup-query ergonomics.
- **v0.7** designed analytics without shipping it: the analytical IR sketch,
  metric/dimension → row inference prototype, and the decision that core owns
  grammar-level windows while `@sisal/analytics` owns the semantic API.
- **v0.8** shipped the compile target analytics needs: `@sisal/core`,
  `SQL_IR_VERSION`, `expr()`, `assembleSelect()`, `assembleInsertFromSelect()`,
  `over()`, frames, `rank`/`rowNumber`/`lag`/`lead`, `dateDiff()`, and the
  capability registry.
- **v0.9** made dialect truth authoritative: `(engine, variant, version)`
  identities, matrix cells backed by `capabilitySupported(...)`, numeric/date
  round-trip truth, and real per-engine verification for the IR shapes analytics
  will wrap.
- **v0.10** shipped `@sisal/etl`: typed rollup jobs, generated pushdown SQL,
  checkpoints, `run`/`backfill`/`replay`, dry-run, scheduler docs, and the
  `post_events → post_hourly_stats` example. Analytics queries those rollups; it
  does not own ETL runtime behavior.

## Package boundary

`@sisal/analytics` depends on `@sisal/core` (and an adapter to execute).
**`@sisal/orm` must not depend on `@sisal/analytics`.**

The package may accept an adapter `Database` structurally at execution time, but
it must not import adapters, database drivers, `@sisal/orm`, `@sisal/migrate`,
or ETL runtime code. Core changes are allowed only when they are shared
compile-target additions (for example, a new capability descriptor entry);
window grammar already lives in core from v0.8.

## Task detail

- **T1 — Package scaffold. ✅ Done.** New `packages/analytics` with a small
  `mod.ts`, public docs, JSR metadata, root workspace wiring, and a boundary
  test proving `@sisal/core`/`@sisal/orm`/`@sisal/migrate` never import
  analytics and analytics never imports adapters, drivers, ORM, migrate, or ETL
  runtime code.
- **T2 — Descriptor model + result inference. ✅ Done.** Graduate the v0.7
  `analytics_result_inference_test.ts` prototype into the real package. The row
  type must be inferred from dimension, metric, and window maps, preserving
  aggregate nullability and offset-window nullability.
- **T3 — Query builder. ✅ Done.** `from(source)` accepts a table, rollup table,
  or subquery fragment; `where`, `dimensions`, `metrics`, `windows`, `orderBy`,
  and `limit` are immutable; duplicate keys and unknown references fail with
  typed `OrmError`s.
- **T4 — Dimensions and buckets. ✅ Done.** `bucket(width, source)` wraps core
  `dateTrunc`/`dateBin`, marks the time axis for period comparison, and
  documents the value-shape differences (`timestamp` on Postgres, text on the
  SQLite/MySQL families).
- **T5 — Aggregate metrics. ✅ Done.** Re-export core aggregate helpers for the
  common metric surface and allow any typed `SqlExpression<T>` as a metric
  value. Metric references inside windows inline the aggregate expression,
  because SQL window evaluation happens after grouping.
- **T6 — Windowed metrics. ✅ Done.** Wrap core `over()` primitives into
  analytics descriptors: trailing `movingAvg`, `rank`, `denseRank`, `rowNumber`,
  `lag`, `lead`, and `delta`. Validate required ordering and frame sizes at
  definition time.
- **T7 — Period comparison. ✅ Done.** `compareToPreviousWindow(metric)` finds
  the single `bucket()` dimension by default, partitions by non-time dimensions,
  and adds previous/delta fields. Ambiguous or missing time axes fail typed.
- **T8 — Percentiles. ✅ Done.** `percentileCont`/`percentileDisc` ship as
  experimental public helpers with a single core capability entry for
  ordered-set aggregates. They fail closed off Postgres until
  MySQL/MariaDB/SQLite variants are probed.
- **T9 — Render and execute. ✅ Done.** `toSql()` compiles to core
  `assembleSelect()`; `render(identity)` is the dry-run surface; `execute(db)`
  accepts a structural executor with `dialectIdentity` and returns typed rows
  without importing an adapter.
- **T10 — Capability gate. ✅ Done.** `supportsQuery` and `assertQuerySupported`
  must render-probe the exact query against the exact identity before execution.
  Unsupported shapes throw `ANALYTICS_UNSUPPORTED_QUERY`, not raw engine errors.
- **T11 — Unit/golden tests. ✅ Done.** Pin Postgres SQL for a query containing
  dimensions, metrics, a windowed metric, comparison fields, `orderBy`, and
  `limit`; add type-level tests for exact row inference and runtime validation
  tests for duplicate/unknown keys.
- **T12 — Dialect capability tests. ✅ Done.** Exercise the supported window
  subset on SQLite/libSQL/MySQL when renderable, and assert typed failures for
  percentiles, unsupported frame units, and any future analytics-only construct.
- **T13 — Runnable examples / integration scenarios. ✅ Done.** The five roadmap
  examples are pinned as Postgres dry-run tests in
  `packages/analytics/examples_test.ts`: post engagement over time, active users
  by day, top posts by engagement, `/rising` velocity, and presentation-ready
  chart rows from rollups. Future live integration can build on these exact
  shapes.
- **T14 — `/rising` proof. ✅ Done.** The recent-velocity query is expressed
  through analytics descriptors (`movingAvg`, `rank`, and
  `compareToPreviousWindow`) without raw SQL in the analytics package.
- **T15 — Advanced-SQL contract graduation.** Revisit contracts 02 (window
  analytics), 03 (sessionization), 04 (top-N per group), 05 (cohort retention),
  and 06 (funnel analysis). Mark each as builder-native, analytics-native, still
  raw, or explicitly deferred with the missing helper named.
- **T16 — Matrix and compatibility docs.** Add feature-matrix rows only for
  integration-backed analytics scenarios; update compatibility docs with the
  Postgres-first support story and typed unsupported behavior.
- **T17 — Rollup-first guidance. 🟡 Partial.** The package README states the
  rollup-first model and ETL handoff. Dedicated docs/examples still need to land
  with T13/T14. Materialized views from v0.10 are an acceleration option, not an
  analytics runtime dependency.
- **T18 — Release hygiene. 🟡 Partial.** `CHANGELOG.md`, package README/module
  docs, generated LLM docs, docs coverage, type checks, unit tests, and publish
  dry-runs are current for T1–T14. It stays partial until T15–T17/T16 are either
  done or explicitly deferred.

## Initial public surface

- **typed analytical `from()`** — a source (table / rollup / subquery).
- **dimensions** — typed group keys, including `bucket()` time dimensions.
- **metrics** — typed aggregate expressions: `count()`, `sum()`, `avg()`,
  `min()`, `max()`, `countDistinct()`, and typed custom expressions.
- **windowed metrics** — `movingAvg()`, `rank()`, `denseRank()`, `rowNumber()`,
  `lag()`, `lead()`, and `delta()`.
- **`compareToPreviousWindow()`** — period-over-period comparison over a time
  bucket axis.
- **percentiles** — experimental Postgres-first `percentile_cont`/`disc` helpers
  with typed capability failures elsewhere.
- **result typing** — `{ dimensions, metrics, windows }` infers a typed result
  row.

## Non-goals

Not a polished/complete analytics product; no renderer integration; no
presentation-mapping helper; no external-engine execution; no ETL runtime in
this package. (The dashboard-mapping, DuckDB, and native-acceleration milestones
that once followed this one were dropped in July 2026 — analytics is now the
last planned feature milestone. A future external database target, e.g. a
time-series DB, is a _possible_ later direction but is **not** planned.)

## Risks

- **Postgres-only creep** — windows/percentiles barely exist on SQLite. Keep the
  API honest with capability gates (v0.9) so unsupported analytics fail typed.
- **Result-type inference complexity** — inferring rows from metric maps may
  push TS limits; the v0.7 prototype must have de-risked this before v0.11.
- **Overlap with the ORM builder** — windows might tempt duplication. Minimal
  window primitives live in `@sisal/core`; the rich API lives here.
- **Performance expectations** — analytics over raw events (not rollups) can be
  slow; steer users to query prepared rollups.

## Acceptance criteria

- A typed analytical query (dimensions + metrics + a windowed metric) compiles
  to correct PostgreSQL and returns a **typed** result row inferred from its
  definition. — **T2/T3/T6/T11**
- `compareToPreviousWindow()` produces correct period-over-period results. —
  **T7/T11/T13**
- The `/rising` velocity feed is expressible through the API (or the doc states
  precisely which sub-feature still needs raw SQL). — **T14**
- Unsupported analytics on SQLite/libSQL fail with a typed capability error, not
  a raw engine error. — **T10/T12/T16**
- The five preview examples are runnable Postgres-first, gated integration
  scenarios or documented dry-run examples where no database is needed. —
  **T13**
- `@sisal/orm` has **no** dependency on `@sisal/analytics`. — **T1**

## Affected packages

New `@sisal/analytics`; depends on `@sisal/core` and accepts an adapter executor
structurally. `@sisal/core` may receive analytics-relevant capability registry
entries, but not analytics-domain builders. No `@sisal/orm` dependency or
analytics import is allowed.

## Required examples / tests

- Unit/golden tests for T2–T11, including type-level row inference.
- Per-dialect capability tests for T12.
- The five preview examples from T13 as runnable, Postgres-first, gated
  integration tests or dry-run examples with exact SQL.
- Contract triage updates for advanced-SQL contracts 02–06.
- Boundary/import tests proving dependency direction.

## Dialect compatibility notes

**Keep the first analytics release Postgres-first.** Neon mirrors Postgres
(serverless caveats from v0.9). SQLite/libSQL expose only the window/aggregate
subset they support correctly; everything else is a typed capability gap, not a
silent degradation.

Feature-matrix rows must be added only after named integration scenarios exist.
Until then, the compatibility docs should describe analytics as preview,
Postgres-first, and capability-gated.
