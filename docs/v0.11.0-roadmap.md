# Sisal v0.11.0 roadmap — `@sisal/analytics` Preview

With the analytical IR designed ([v0.7](v0.7.0-roadmap.md)) and the core IR
stabilized ([v0.8](v0.8.0-roadmap.md)), v0.11 ships the first **Postgres-first
analytical query layer** — a typed OLAP API that queries rollups and analytical
tables using Sisal schema/types. It complements `@sisal/etl`: ETL builds the
rollups, analytics queries them.

## Status legend

## Goal

A typed analytical query API (metrics × dimensions × windows) that compiles into
the `@sisal/core` IR, infers result-row types from the metric/dimension set, and
renders correct SQL — Postgres-first, with other dialects exposing only the
subset they support correctly.

## Package boundary

`@sisal/analytics` depends on `@sisal/core` (and an adapter to execute).
**`@sisal/orm` must not depend on `@sisal/analytics`.**

## Scope — initial features

- **typed analytical `from()`** — a source (table / rollup / subquery).
- **dimensions** — typed group keys (incl. `bucket()` time dimensions).
- **metrics** — typed aggregate/window expressions: `count()`, `sum()`, `avg()`,
  `min()`, `max()`, `countDistinct()`.
- **windowed metrics** — `movingAvg()`, `rank()`, `lag()`, `lead()`.
- **`bucket()`** — time bucketing (built on the v0.6/v0.8 bucket abstraction).
- **percentile investigation** — `percentile_cont`/`disc` as a
  prototype/Postgres-first feature (may stay experimental).
- **`compareToPreviousWindow()`** — period-over-period comparison.
- **result typing for computed metrics** — `{dimensions, metrics}` → typed row.

## Examples to include

- **post engagement over time** (time series from `post_hourly_stats`).
- **active users by day** (countDistinct over a day bucket).
- **top posts by engagement** (rank/order over a rollup).
- **`/rising` feed using recent velocity** — the moving-window score the
  `*-rising-feed` examples hand-write today, now expressed via the analytics
  API.
- **dashboard-ready time series** from rollup tables (feeds
  [v0.12](v0.12.0-roadmap.md)).

## Non-goals

Not a polished/complete analytics product; no renderer integration; no dashboard
helper (that's v0.12); no DuckDB-first execution (v0.13+); no ETL runtime in
this package.

## Risks

- **Postgres-only creep** — windows/percentiles barely exist on SQLite. Keep the
  API honest with capability gates (v0.9) so unsupported analytics fail typed.
- **Result-type inference complexity** — inferring rows from metric maps may
  push TS limits; the v0.7 prototype must have de-risked this before v0.11.
- **Overlap with the ORM builder** — windows might tempt duplication. Minimal
  window primitives live in `@sisal/core`; the rich API lives here.
- **Performance expectations** — analytics over raw events (not rollups) can be
  slow; steer users to query prepared rollups.

## Open questions

- ~~Are `rank`/`lag`/`lead` exposed as core builder primitives too, or
  analytics-only?~~ **Answered by the v0.8 task list (item 10):** the
  grammar-level window expressions (`over()`, frames, `rank`/`row_number`/
  `lag`/`lead`) are core primitives; analytics wraps them in the semantic layer
  (named windows, moving averages, period comparisons).
- Percentiles: ship Postgres-first, or keep experimental until DuckDB (v0.13+)?
- How much of the API can SQLite/libSQL honestly support (modern SQLite has
  window functions; PARTITION/frame/percentile coverage varies)?

## Acceptance criteria

- A typed analytical query (dimensions + metrics + a windowed metric) compiles
  to correct PostgreSQL and returns a **typed** result row inferred from its
  definition.
- `compareToPreviousWindow()` produces correct period-over-period results.
- The `/rising` velocity feed is expressible through the API (or the doc states
  precisely which sub-feature still needs raw SQL).
- Unsupported analytics on SQLite/libSQL fail with a typed capability error, not
  a raw engine error.
- `@sisal/orm` has **no** dependency on `@sisal/analytics`.

## Affected packages

New `@sisal/analytics`; depends on `@sisal/core` + an adapter. Possibly minimal
window primitives added to `@sisal/core` (shared). No `@sisal/orm` change.

## Required examples / tests

- The five examples above as runnable, Postgres-first, gated integration tests.
- Type-level tests pinning metric→row inference.
- Per-dialect capability tests (supported / typed-error) for
  windows/percentiles.

## Dialect compatibility notes

**Keep the first analytics release Postgres-first.** Neon mirrors Postgres
(serverless caveats from v0.9). SQLite/libSQL expose only the window/aggregate
subset they support correctly; everything else is a typed capability gap, not a
silent degradation.
